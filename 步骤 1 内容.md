可以在 Docker 容器里做，但要分清楚三类数据的可见性：

| 数据类型                                                | 容器内能否拿到                | 建议采集方式                                               | 备注                                                         |
| ------------------------------------------------------- | ----------------------------- | ---------------------------------------------------------- | ------------------------------------------------------------ |
| NPU 是否可见、显存占用、设备状态、进程占用              | 可以                          | `npu-smi info`、DCMI、vLLM/MindIE metrics                  | 需要把 `/dev/davinci*`、`/dev/davinci_manager`、`/dev/devmm_svm`、`/dev/hisi_hdc`、driver/lib、`npu-smi` 等挂进容器。vLLM-Ascend 官方 Docker 示例就是这么挂载的。([vLLM](https://docs.vllm.ai/projects/ascend/en/latest/installation.html)) |
| CANN / NPU kernel timeline、算子耗时、H2D/D2H copy 事件 | 可以，但要配环境              | `msprof` / CANN profiler / vLLM-Ascend logs                | CANN 文档说明可以用 `msProf` 收集并解析 profile 数据。([昇腾社区](https://www.hiascend.com/document/detail/en/canncommercial/800/opdevg/Ascendcopdevg/atlas_ascendc_10_0056.html)) |
| CPU 指令数、cache miss、调度延迟、eBPF/perf             | 部分可以                      | `perf stat/top/record`、eBPF、`pidstat`、`numastat`        | 建议宿主机侧采集最稳；容器内采集通常需要 `--pid=host`、`--privileged` 或 `CAP_PERFMON/SYS_ADMIN/SYS_PTRACE`。 |
| PCIe / UB / H2D / D2H 传输速度与时延                    | 可以测，但不是靠一个静态命令  | 自写 `aclrtMemcpy`microbench + CANN profiler copy timeline | 不要只看标称带宽；要测不同 size、同步/异步、pinned memory、stream overlap。 |
| “NPU 指令时间”                                          | 不能按 CPU 指令级方式直接理解 | 以 NPU kernel / operator duration 近似                     | 初期把“指令时间”定义为 kernel/op timeline、AI Core 利用率、copy stall 更实际。 |

你的当前环境按已记录口径处理：**Ubuntu + CANN 9.0.0 + Atlas 800T A2 + 8 卡 NPU + 每卡 64GB HBM**。但这次计划限定为**单卡场景**，所以默认只暴露 `/dev/davinci0`，并设置 `ASCEND_RT_VISIBLE_DEVICES=0`。华为云文档也明确说明容器内可以用 `npu-smi info` 查看可用 NPU，并用 `ASCEND_RT_VISIBLE_DEVICES` 指定容器内使用的 NPU 编号。([华为云支持中心](https://support.huaweicloud.com/intl/en-us/bestpractice-modelarts/modelarts_llm_infer_5906006.html))

这一阶段不做复杂 A+K 改造，目标是：

## 4. 需要采集的指标

按 NEO 和 KTransformers 的思路，你现在不是只看 `tokens/s`，而是要记录“计算、状态、传输、队列、专家”五类指标。

### 4.1 请求级指标

```text
request_id
model_name
prompt_tokens
output_tokens
context_len
concurrency
TTFT_ms
TPOT_ms
ITL_ms
throughput_tokens_per_s
P50/P95/P99
stall_reason
```

### 4.2 NPU / HBM 指标

```text
npu_id
hbm_used_mb
hbm_peak_mb
npu_util
aicore_util
aiv_util
kernel_name
kernel_start/end
operator_latency
copy_stream_latency
```

### 4.3 CPU / DRAM / NUMA 指标

```text
cpu_util_per_core
cpu_cycles
cpu_instructions
cache_misses
context_switches
scheduler_delay
numa_node
dram_bandwidth_read/write
pinned_memory_bytes
host_oom_event
```

CPU 不要抽象成一个 “FLOPs 数”。项目硬件地图里已经强调，CPU 在推理中至少有控制面、warm tier、可重叠计算、expert compute、I/O aggregation 五类角色，需要分别建模。

### 4.4 H2D / D2H / PCIe-UB 指标

```text
copy_direction: H2D / D2H
bytes
latency_us
bandwidth_GBps
async_or_sync
stream_id
pinned_or_pageable
overlap_ratio
dma_setup_latency
```

NEO 的关键启发是：CPU 参与只有在“状态留在 CPU 侧、计算也留在 CPU 侧、跨链路传输足够小、执行能与主链路重叠”时才可能赚钱；否则 CPU/PCIe/DDR 会反噬 TPOT/P99。

### 4.5 MoE expert 指标

```text
layer_id
token_id
topk_expert_ids
expert_score
expert_hit_or_miss
expert_tier: HBM / DRAM / SSD / unknown
expert_load_bytes
expert_load_latency_us
expert_compute_latency_us
expert_wait_latency_us
prefetch_lead_time_us
wrong_prefetch_bytes
```

MoE expert 管理的建模字段应包括 expert size、top-k、routing distribution、miss penalty、prefetch accuracy；这是项目材料里对 KTransformers / FineMoE / DALI / FluxMoE / MoE-APEX 的共同抽象。


### 第 4 阶段：参考 NEO 的 KV / attention warm-tier 实验，1 周

**目标：**验证 CPU/DRAM warm tier 是否有价值。

先不要做 CPU attention kernel，先做 KV offload 和传输观测。

实验：

```text
baseline: KV 全留 NPU
offload: inactive KV -> CPU DRAM
prefix: repeated prefix + external KV
multi-turn: session interval 1s / 10s / 60s
```

指标：

```text
KV offload bytes
D2H latency
H2D restore latency
HBM saved
TTFT delta
TPOT delta
P95/P99 delta
copy overlap ratio
```

这对应 NEO 的思想边界：CPU 侧参与必须减少 HBM 压力，并且传输/计算可重叠；如果 H2D/D2H 进入逐 token 热路径，就不是优化。

------

### 第 5 阶段：参考 KTransformers 的 expert trace，不先做 CPU expert 执行，1 周

**目标：**验证专家热度，而不是马上让 CPU 跑 expert。

先做三个东西：

```text
expert routing trace
expert reuse distance
expert hotness histogram
```

每层输出：

```text
layer_id
topk_expert_ids
expert frequency
expert reuse distance
expert entropy
hot expert set size
```

然后做**模拟 expert tier**：

```text
假设 HBM 只能放 Top-N experts
假设 DRAM 放 warm experts
假设 SSD 放 cold experts
根据真实 routing trace 计算 hit/miss
```

输出：

```text
Top-8 / Top-16 / Top-32 / Top-64 experts 常驻时的命中率
expert miss penalty 预算
需要的 DRAM expert tier 容量
预取窗口需要提前多少 layer/token
```

KTransformers 的精读结论是：它不是简单把专家权重放 CPU，而是 shared/hot experts 留 GPU/NPU，routed experts 放 CPU/DRAM，并依赖 CPU SIMD/NUMA/异步调度；迁移到 Kunpeng 必须重新做 SVE/SME microbenchmark。

## 9. 这一阶段不要做的事

不要一开始就做完整 P/D 分离、NPU-SSD 直通、CPU 跑主干 FFN、GLM5.2 满血低延迟。项目融合方案也明确写了：第一阶段不承诺 GLM5.2 满血低时延服务，不承诺 1M 上下文，不承诺 NPU-SSD 直通，不承诺 CPU 替代 NPU 主算，不把完整物理 P/D 分离作为默认收益路径。

你当前最应该先拿到的是这三张图：

```text
1. context_len / concurrency -> TTFT / TPOT / HBM 曲线
2. expert_id -> frequency / reuse_distance / hotset_coverage 曲线
3. H2D/D2H size -> latency / bandwidth / overlap 曲线
```

有了这三张图，后续才知道该优先做 KV offload、expert hot cache、CPU fallback，还是只做运行时调度和 trace。
