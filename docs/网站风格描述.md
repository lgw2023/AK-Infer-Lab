可以描述为：**华为式技术评审/架构论证 PPT 风格**，本质是“高信息密度的工程决策页”，不是营销型 PPT。

## 1. 这套 PPT 的视觉风格

关键词可以这样概括：

**高密度、技术评审、表格驱动、红蓝强调、工程路线图、企业内审、数据论证型。**

更细一点：

1. **版式**
   - 16:9 横版页面。
   - 白底，灰色细边框，大量矩形分区。
   - 一页内塞入 3～5 个信息模块。
   - 模块之间用细线、浅灰框、浅黄色表头分隔。
   - 页底有统一 footer，包括公司名、保密标识、页码、logo 位置。
2. **标题**
   - 顶部大号深红色标题。
   - 标题通常是判断句或结论句，不是普通名词。
   - 例如：“端边侧 1920 SoC 在算力和显存容量存在劣势，需要系统级创新优化”。
3. **颜色**
   - 主色：深红，用于标题、风险、结论。
   - 辅色：深蓝，用于关键技术判断、优化方向、重要结论。
   - 表格底色：浅黄、浅灰、浅绿。
   - 强调色：红色数字、蓝色方案、绿色资源条。
4. **内容组织**
   - 每页都有“问题 → 数据 → 对比 → 结论”的结构。
   - 大量表格、柱状图、热力图、架构图、时间线。
   - 文字非常密集，偏研发评审材料，而非演讲展示材料。
   - 常见模块标题包括：技术挑战、竞争分析、性能分析、总结、资源调度、显存需求、KV Cache、Agent Swarm 等。
5. **字体与排版**
   - 中文黑体/微软雅黑风格。
   - 标题粗体，正文偏小。
   - 重点词用红色、蓝色、加粗、下划线混合强调。
   - 表格边框明显，单元格紧凑。
   - 几乎没有大面积留白。

## 2. 如果用网页模拟这种 PPT，建议的前后端形态

比较适合做成：

**“PPT 页面生成器 + 数据驱动可视化 + 16:9 slide viewport”**

前端可以用：

- React / Next.js
- Tailwind CSS
- ECharts / AntV / Recharts
- HTML/CSS Grid 模拟 PPT 布局
- 每一页是一个固定比例容器，例如 `aspect-ratio: 16 / 9`
- 支持导出 PDF 或图片

后端可以用：

- Node.js / Python FastAPI
- 接收实验数据：CSV、JSON、Excel、Markdown、PDF 摘要
- 输出结构化 slide schema
- 前端根据 schema 渲染页面

页面不要做成普通响应式网页，而要做成**固定画布式页面**，例如 1920×1080 或 1366×768，网页只是承载 PPT 画布。

## 3. 可直接使用的提示词模板

下面这个提示词适合给 Cursor、Claude、ChatGPT、v0、Bolt 等代码生成工具使用。

```text
你是一名前端架构师和企业技术汇报材料设计师。请基于我提供的实验数据、技术文档和结论，生成一个“网页形式的 PPT 展示系统”，用于模拟高信息密度的企业研发评审 PPT。

整体风格要求：
- 风格为高密度工程技术评审 PPT，而不是营销型网页。
- 页面比例固定为 16:9，每一页模拟一张 PPT slide。
- 白色背景，细灰色边框，模块化网格布局。
- 顶部使用深红色大标题，标题必须是结论型表达。
- 页面中大量使用表格、对比图、架构框图、时间线、热力图、柱状图和总结条。
- 重点结论使用红色或蓝色加粗文字突出。
- 风格参考企业内部技术路线评审、系统架构论证、性能瓶颈分析材料。
- 避免花哨动画、渐变背景、卡片阴影和营销化视觉。
- 信息密度要高，允许小字号，但层级必须清晰。
- 不要使用真实公司 logo 或保密标识，footer 使用占位文本即可。

视觉规范：
- 主标题颜色：深红色，例如 #B00000 或 #C00000。
- 关键技术强调：深蓝色，例如 #0033CC。
- 风险/瓶颈/劣势：红色。
- 表头背景：浅黄色或浅灰色。
- 架构模块背景：浅绿色、浅灰色。
- 边框：1px solid #666 或 #999。
- 字体：Microsoft YaHei, Arial, sans-serif。
- 标题字号：28–36px。
- 模块标题字号：18–24px。
- 表格正文：12–16px。
- 页脚字号：10–12px。

页面结构要求：
每一页 slide 需要包含：
1. 顶部结论型标题栏
2. 中间 2～4 个主分析模块
3. 每个模块包含：
   - 红色或黑色小标题
   - 表格、图表或架构图
   - 蓝色/红色高亮信息
   - 1～3 条工程判断
4. 底部总结栏
   - 左侧写“总结”
   - 右侧用 2～4 条 bullet 总结本页结论
5. 页脚
   - 左侧：PROJECT / TECHNOLOGY REVIEW
   - 中间：Internal Technical Analysis
   - 右侧：Page X

数据处理要求：
- 输入可能包含 CSV、JSON、Markdown、实验日志或技术文档摘要。
- 请先抽取关键指标，包括：
  - 模型规模
  - 并发数
  - 序列长度
  - 显存占用
  - KV Cache 命中率
  - TTFT
  - TPOT
  - TPS
  - CPU/NPU/GPU 利用率
  - 带宽
  - IO 延迟
  - 成本
  - 性能收益
- 自动识别不同实验组之间的对比关系。
- 自动生成“优势、劣势、风险、瓶颈、优化方向、结论”。

页面类型要求：
请至少生成以下几类 slide：
1. 关键问题总览页
   - 使用多个横向条目描述技术挑战。
   - 每个挑战包含：问题、影响、数据依据、优化方向。

2. 竞争分析/方案对比页
   - 使用时间线或三栏对比图。
   - 对比当前方案、原型方案、下一代方案。
   - 每个方案下方写优势和劣势。

3. 性能分析页
   - 使用图表 + 表格组合。
   - 展示并发数、TPOT、TPS、TTFT、KV Cache 命中率等指标。
   - 用红色标注瓶颈区间，用蓝色标注优化收益。

4. 资源需求分析页
   - 展示显存、内存、SSD、带宽、CPU 利用率等资源消耗。
   - 使用表格和架构图说明资源约束。

5. 总结建议页
   - 使用“现状 / 问题 / 建议 / 预期收益”的结构。
   - 结论必须具体、可执行、带数据支撑。

前端实现要求：
- 使用 Next.js + React + TypeScript。
- 使用 Tailwind CSS 实现布局。
- 使用 ECharts 或 Recharts 绘制图表。
- 每一页实现为一个 Slide 组件。
- 提供 SlideDeck 组件用于切换页面。
- 支持键盘左右键翻页。
- 支持导出当前页面为 PNG 或 PDF。
- 页面容器固定 16:9，居中显示，可根据浏览器窗口缩放。
- 所有数据通过 JSON schema 驱动渲染，不要把数据写死在组件里。

后端实现要求：
- 使用 FastAPI 或 Node.js API。
- 提供接口：
  - POST /api/parse-docs：解析上传文档和实验数据。
  - POST /api/generate-slides：根据结构化数据生成 slide schema。
  - GET /api/slides/:id：返回某个 slide deck。
- 后端输出 slide schema，前端只负责渲染。

slide schema 示例：
{
  "title": "NV Spark性能分析：单机支持5-10并发，长序列场景受限于KV Cache命中率",
  "pageNumber": 10,
  "sections": [
    {
      "type": "chart-table-combo",
      "title": "并发性能对比",
      "highlight": "单机5-10并发为甜点区间",
      "data": {}
    },
    {
      "type": "comparison-table",
      "title": "长序列KV Cache影响",
      "data": {}
    }
  ],
  "summary": [
    "单机200B模型规模下，最大支持4机互联。",
    "单并发最大序列长度可支持1M，但强依赖KV Cache命中率。",
    "TPOT 50ms SLA下，5-10并发为最优使用区间。"
  ]
}

输出要求：
- 直接生成完整可运行项目代码。
- 给出项目目录结构。
- 给出安装和运行命令。
- 提供一组 mock 数据，用于展示 3 页样例 slide。
- 代码要清晰、模块化、可扩展。
```

## 5. 对模型最关键的约束句

生成效果好不好，关键看有没有这些约束：

```text
不要做成现代 SaaS 官网风格。
不要使用大留白、圆角大卡片、渐变背景、插画风。
要做成企业内部技术评审 PPT：密集、表格多、边框多、红蓝强调、结论先行、数据驱动。
```

这句话很重要，因为很多网页生成模型默认会做成漂亮但不符合 PPT 语境的 dashboard。你的目标不是 dashboard，而是**工程评审 slide canvas**。
