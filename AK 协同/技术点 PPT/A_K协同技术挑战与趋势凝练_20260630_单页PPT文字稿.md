# 关键技术一：A+K 协同

> 单页 PPT 文字稿。版式参考 `AK 协同 ppt 样式说明.jpg`：顶部三块输入信息，中部左侧放“当前问题痛点挑战+代表文献卡”，中部右侧放“技术及趋势”，底部仅放硬件诉求摘要。

## 0. 页面主判断

A+K 协同不是“让 CPU 多算一点”，而是把 Ascend NPU 主推理、Kunpeng CPU 长尾计算与工具执行、KV/Prefix/Expert 状态层、Agent 调度控制面和 Trace/仿真闭环，围绕同一条 Agent critical path 组织起来。

## 1. 顶部三栏文字

| 顶部模块 | 建议放入文字 |
|---|---|
| 目标应用场景 | AI Coding Agent、办公助手 Agent、多模态生成 |
| 目标场景的任务包络、主要瓶颈分析 | Coding 面向仓库理解、编译测试、工具循环和大 MoE 长上下文；办公面向文档/表格/PPT、RAG、结构化输出和多轮工具；多模态面向 AR+DiT、VAE、codec、后处理与临时文件流水。共性瓶颈是长上下文、状态驻留、工具等待、专家冷热和端到端 JCT。 |
| 当前硬件规格约束与分工策略 | 以 Mini/DV100 Lite 与塔式/950PR Lite 两类机器作为当前输入边界，硬件约束来自容量、跨层搬运、工具波动和多 Agent 状态并发，而不是抽象 TOPS：Mini 形态约 64GB Bailu + 64GB DDR、PCIe 64GB/s，主要验证 A+K placement、冷热权重预取、CPU 工具/编码和搬运成本，不能等同于 744B 或 400GB KV 最终容量方案；950PR Lite 采用 84GB Bailu + 384GB DDR5 + 512GB×4 SSD，NPU-Bailu 1TB/s、NPU-CPU UB 200GB/s、CPU-DDR5 228GB/s、CPU-SSD 40GB/s，是 KV/Prefix/Expert 热温冷分层、工具感知 TTL、专家预取、NPU-SSD 直通和多用户准入的主验证平台；分工上 Ascend 保持热权重、热 KV 与密集生成主路径，Kunpeng 负责工具/检索/media、状态元数据、DDR/SSD 温冷层调度和运行时回压，系统目标从单点 tokens/s 转向“放得下、恢复得回、调得动、测得准”。 |

## 2. 左侧：当前问题痛点挑战

| 问题 | 核心挑战 | 代表文献/证据卡 |
|---|---|---|
| 数据面协同计算：NPU 主路径与 CPU 长尾任务错配，端到端链路被短小碎、工具和多模态外部计算拖慢。 | NPU 擅长规则化大矩阵，但 tokenizer、短序列矩阵、稀疏索引、JSON/grammar、codec 和后处理等固定开销容易支配短请求；细粒度 offload 会被搬运、同步和调度成本反噬；MoE 冷专家、低频专家和 fallback 专家需要进入正式计算平面。 | llm.npu、NEO、LIA/APEX、KTransformers/CoX-MoE、DiffServe/CodecSight |
| 状态与内存协同：权重、KV、Prefix、Context、Expert 同时挤压近端内存，状态“可放下”不等于“恢复得回”。 | 长上下文和大 MoE 让 KV、workspace、专家、索引共同竞争近端容量；KV/Prefix/Expert 下沉后，恢复时延会直接进入 TTFT、TPOT 和 JCT；传统 prefix cache 难处理相对顺序稳定但绝对位置漂移的上下文；专家热度会随 session、阶段和输入变化。 | InfiniGen、Mooncake/LMCache、CacheSlide、KVPR、MoE-Lightning/FineMoE |
| 控制面调度管理：Agent 从单请求变成 workflow，request-level 调度无法管理工具等待、KV 生命周期和任务完成时间。 | 单次 TTFT/TPOT 不能代表任务体验；工具队列、CPU 沙箱、检索、文件系统和 NPU 队列强耦合；工具空窗决定 KV 是 pin、offload、discard 还是 recompute；小并发交互需要 continuation priority 和 prefill token budget。 | Agentix、KVFlow、TokenCake、Continuum、Helium、MARS、Policy Runtime |
| Trace 与仿真闭环：没有真实 trace、扰动实验和 cost model，就无法判断 A+K offload、预取、扩容量或加强 CPU 是否收益为正。 | tokens/s 无法解释任务慢在哪；高利用率不是因果瓶颈证据；资源需求必须通过限算力、限容量、限带宽、限 CPU、限 I/O、限功耗逐项验证；A+K 需要 link-aware、tail-aware、energy-aware 的收益边界模型。 | BurstGPT/ServeGen/Mooncake traces、Agentic AI Workload Characteristics、ProfInfer、Chakra/CCL-Bench、Frontier |

## 3. 右侧：技术及趋势

| 技术方向 | 该技术方向的分点关键技术 | 未来趋势 |
|---|---|---|
| 方向 1：数据面协同计算 | 阶段感知执行矩阵：以 prefill、decode、工具、检索和媒体流水的负载画像决定 A/K 归属，让 Ascend 只承接高利用密集主算，Kunpeng 吸收短小碎、近数据与异常路径，减少无效搬运和空转等待。<br>CPU 近数据加速面：围绕检索、语义聚合、短序列计算和离散 I/O 建设 Kunpeng 侧加速能力，把 RAG、Coding 工具和多模态预处理从 NPU 队列中剥离，降低端到端关键路径抖动。<br>MoE 冷专家执行面：用专家热度预测、分层预取、CPU fallback 和合并执行覆盖低频专家，把 expert miss 从随机拉取变成可调度流水，降低大 MoE 的尾延迟。<br>工具媒体隔离流水：将编译测试、沙箱、文档解析、视频编解码和后处理做成独立资源池，使外部工具计算可准入、可限流、可回压，不再反向拖慢 NPU serving。 | 数据面会从手工规则分工演进为 profiler 与 cost model 驱动的自动混合执行计划，CPU 成为可度量、可准入、可回压的异构推理协处理器。 |
| 方向 2：状态与内存协同 | 逻辑虚拟介质层：把 Bailu、DDR、SSD 与后续 HMM/PIM 抽象为统一状态空间，为 KV、Prefix、Context 和 Expert 提供定位、迁移、压缩、恢复的共同语义，屏蔽底层介质差异。<br>竞容比感知分层：根据容量压力、访问热度、下次使用窗口和任务关键路径动态调整热/温/冷布局，使系统同时满足“放得下”和“恢复得回”。<br>工具感知状态保温：把工具空窗、会话阶段和 resume 代价纳入 TTL 与预取决策，短工具保留状态、中长工具下沉状态，减少工具返回后的重复 prefill。<br>专家与 KV 恢复流水：通过历史路由、隐藏状态和必要 KV 预测，把冷专家、温 KV 与冷 Prefix 的加载、重算和直通 I/O 重叠起来，让恢复时延尽量隐藏在后续计算中。 | 状态管理会从引擎内部缓存演进为跨多级介质的对象化状态平面，KV、Prefix、Context 和 Expert 将成为 Agent 长任务的可编排执行状态。 |
| 方向 3：控制面调度管理 | Agent 语义控制面：用最小化 workflow/session/stage/tool/cache 元数据贯穿 Agent 框架、推理引擎和状态层，把调度对象从单个 request 升级为完整任务流。<br>运行时策略闭环：在 Agent 框架与推理引擎之间部署 observe-score-predict-act 轻量中间层，统一决定准入、优先级、状态意图、工具排队和 early stop。<br>恢复优先调度：区分冷启动 prefill、工具返回 resume 和短 decode，用 prefill budget 与 continuation priority 抑制长上下文队头阻塞，优先完成接近收敛的任务分支。<br>多资源准入回压：同时感知 NPU 队列、CPU 工具池、DDR/SSD 水位、I/O 与功耗温度，主动限流、降级或迁移，避免工具拥塞扩大成系统级尾延迟。 | 控制面会从 request-level batching 演进到 workflow-level SLO/goodput 调度，运行时通过统一 metadata 连接工具、状态、生成主路径和能效目标。 |
| 方向 4：Trace 与仿真闭环 | 任务级 Trace 证据包：把 task、turn、request、tool、状态、资源和功耗事件串成可回放时间线，并区分真实生成 token 与缓存命中 token，复原“模型没算但状态仍驻留”的阶段。<br>扰动式瓶颈归因：通过限算力、限容量、限带宽、限 CPU、限 I/O 和限功耗实验建立因果曲线，证明影响 JCT 与 P95/P99 的真实瓶颈。<br>A+K 收益边界模型：把阶段 profiler、链路成本、能耗成本和尾延迟风险合成 cost model，用来判断卸载、预取、重算和直通存储是否收益为正。<br>What-if 规格反推：用基线、缩减和目标配置做仿真对比，把 benchmark 从分数展示转成容量、带宽、CPU、I/O 与功耗参数的设计输入。 | 评测会从 tokens/s 指标表演进为可回放的证据包和设计闭环，A+K 的软硬件协同将由 trace、扰动和仿真共同收敛。 |

## 4. 页底：硬件诉求摘要

| 方向 | 页底硬件诉求短句 |
|---|---|
| 数据面 | 需要 Kunpeng 提供可隔离的工具池、短矩阵/向量能力和媒体 I/O 能力，同时提供低开销异步搬运通路，避免细粒度协同被同步成本抵消。 |
| 状态面 | 需要把 Bailu、DDR、SSD 做成可编排的热/温/冷 tier，并支持低 bounce 的冷 KV、冷专家和 checkpoint 恢复路径。 |
| 控制面 | 需要覆盖 NPU、CPU、内存、SSD、功耗和温度的计数器与隔离接口，让 runtime 能做联合准入、回压和资源水位控制。 |
| 闭环面 | 需要稳定 run card、trace counters 和 what-if 仿真参数，使容量、带宽、直通路径、编址/隔离和功耗裕量都能由目标负载反推。 |

当前规格锚点：84GB Bailu、384GB DDR5、512GB×4 SSD，NPU-Bailu 1TB/s、NPU-CPU UB 200GB/s、CPU-DDR5 228GB/s、CPU-SSD 40GB/s；本页仅作为硬件诉求摘要，详细规格不放入“技术及趋势”主体。
