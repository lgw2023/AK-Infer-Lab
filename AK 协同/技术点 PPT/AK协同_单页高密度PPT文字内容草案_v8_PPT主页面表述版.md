## 1. 专家权重容量管理：MoE 专家分层、预取与冷热驻留

### 定位

GLM5.2 单卡压榨硬件，第一矛盾是专家权重容量。全量专家无法常驻近端内存，必须依靠 CPU 中心状态底座对专家进行分层管理。NPU 只保留主干权重、共享专家和高频专家；主存承接温专家；冷存储承接低频专家和检查点。

### 问题

- **专家容量远超近端内存：**GLM5.2 的总参数主要集中在 MoE 专家上，单卡近端内存不能按全量常驻设计。
- **专家缺失会放大尾延迟：**如果解码阶段等待专家从主存或冷存储回取，单 token 生成会被搬运和同步打断。
- **专家热度随任务变化：**不同代码仓、不同工具调用、不同会话阶段会激活不同专家，固定热集和简单换出都不可靠。
- **CPU 不能默认替代主算：**Kunpeng 的主要角色应是专家索引、热度判断、预取编排和低频补算，而不是泛化承接主干计算。

### 技术及趋势

- **专家冷热驻留：**通过路由分布、会话轨迹和层级热度联合建模，优化 MoE 专家在近端内存、DRAM/DUMA 与冷存储间的驻留位置，实现热专家近端保活、温专家提前回温、冷专家低成本恢复。
- **路由感知预取：**通过专家命中率、回取时延和预取窗口的闭环观测，降低 expert miss 对解码时延和尾延迟的冲击，实现专家搬运与 NPU 热计算路径重叠。

### 未来趋势

专家管理会从“权重卸载”走向“专家生命周期管理”。热专家是否常驻、温专家何时回温、冷专家是否恢复或降级，都需要由运行时根据真实路由和硬件链路动态决定。

## 2. 上下文状态容量管理：KV、前缀与上下文分层恢复

### 定位

KV、前缀和历史上下文不是普通缓存，而是 Agent 工作流的长期状态。它们会跨轮次、跨工具调用和跨代码片段重复出现。A+K 协同需要把这些状态从 NPU 内部缓存提升为外部可管理对象，由 CPU 中心状态底座统一承载、恢复和重算决策。

### 问题

- **长上下文挤压近端容量：**权重、专家、KV、前缀、激活和工作区同时竞争近端内存，长会话会快速放大容量压力。
- **前缀复用不等于完全命中：**代码仓、工具定义和系统提示经常相对结构不变，但绝对位置变化，传统前缀命中难以稳定覆盖。
- **冷状态恢复可能反噬收益：**状态下沉到主存或冷存储后，如果布局碎、回取慢或同步多，恢复可能比重算更差。
- **状态策略必须服务体验：**状态管理不能只追求放得下，还要控制首 token 时延、单 token 时延和长尾抖动。

### 技术及趋势

- **状态对象治理：**通过将 KV、前缀、上下文和会话状态抽象为可迁移对象，优化热温冷状态在近端内存、DRAM/DUMA 与 SSD 间的分层排布，实现长上下文容量扩展和多轮重入可控。
- **恢复重算判定：**通过比较状态回取、局部重算、压缩保留和直接丢弃的代价，降低冷状态恢复对首 token 时延和 P95/P99 的扰动，实现状态管理从被动换入走向策略化恢复。

### 未来趋势

上下文管理会从“显存缓存策略”走向“推理状态数据面”。KV、前缀、历史会话和多模态中间状态都应具备位置、热度、生命周期、恢复代价和质量风险等元数据。

## 3. 阶段级协同计算：CPU 承接冷、短、稀疏与状态相关子路径

### 定位

A+K 协同计算不是让 CPU 替代 NPU 主算，而是让 CPU 在合适阶段承接 NPU 不擅长、或不值得打断主计算路径的小粒度子任务。NPU 持续执行密集主路径，CPU 负责判断、索引、预取、状态整理和少量可掩盖计算。

### 问题

- **不同阶段资源画像不同：**预填更依赖密集算力和长序列处理，解码更依赖热权重、热 KV 和低抖动调度，状态恢复更依赖内存与存储链路。
- **细粒度协同容易被搬运抵消：**如果每个 token 都发生 CPU/NPU 同步、主存访问和格式转换，CPU 侧收益会被链路开销吞掉。
- **冷路径需要被掩盖：**CPU 参与只有在可以与 NPU 主计算、状态回温或专家预取重叠时，才有可能带来净收益。
- **物理拆分不是默认收益：**单卡一体机应先做阶段逻辑分工和队列隔离，不能把大集群式拆分直接写成默认方案。

### 技术及趋势

- **阶段级执行分工：**通过区分预填、解码、专家路由、状态恢复和多模态生成阶段，优化 CPU 与 NPU 的任务边界，实现 NPU 稠密主路径连续执行。
- **冷路径协同加速：**通过将路由判断、专家索引、稀疏小算子、状态筛选和低频补算下沉 CPU，降低细粒度交互和近端容量占用，实现冷路径可批量、可重叠、可回退执行。

### 未来趋势

A+K 协同计算会从“固定算子卸载”走向“阶段级执行计划”。真正要优化的不是某个子算子是否放到 CPU，而是整条推理链路中哪些等待、搬运和冷路径可以被提前处理或并行掩盖。

## 4. 系统级性能预测：可观测、仿真与硬件规格反推

### 定位

前面三类机制能否成立，不能靠概念判断，必须靠实测、画像和仿真闭环。A+K 需要建立面向模型阶段、专家命中、状态恢复和硬件链路的证据层，用来判断协同机制是否收益为正，并反推 CPU、主存、冷存储、互联和近端内存规格。

### 问题

- **收益边界不清：**专家分层、状态分层、CPU 参与计算和冷存储回流，都可能被搬运、同步和运行时开销抵消。
- **平均吞吐不足以解释问题：**单看 tokens/s 无法判断瓶颈到底来自近端内存、主存带宽、冷存储、互联、专家缺失还是状态恢复。
- **硬件规格需要从负载反推：**CPU 核数、主存容量、主存带宽、冷存储吞吐、互联带宽和近端内存容量，都应来自模型结构和真实工作负载。
- **单卡目标必须分级验证：**低比特加载、少量 token、4K-16K smoke、32K 低速交互和 1M 风险边界，不能混成同一类承诺。

### 技术及趋势

- **运行时画像建模：**通过采集专家命中、KV 复用、上下文恢复、算子时间线和硬件链路数据，优化 A+K 协同的瓶颈归因，实现从平均吞吐到尾延迟、容量拐点的可解释评估。
- **规格扰动反推：**通过对 CPU 核数、DRAM/DUMA 容量带宽、SSD I/O、UB 链路和近端内存进行单因子扰动，识别系统敏感项和失效边界，实现单卡一体机硬件规格可预测。

### 未来趋势

性能预测会从“事后跑分”走向“设计前反推”。每一项 A+K 协同机制都应能回答三个问题：能不能省容量，能不能省等待，能不能在目标硬件上稳定收益为正。

## 5. 底部横条：落地优先级与硬件诉求

### 硬件诉求摘要

**n+1 分级内存系统：**以 Bailu/HBM 承接热权重、热专家和热状态，以 DDR/DUMA 承接温专家、温 KV 和回温缓冲，以 SSD 承接冷状态和检查点，重点验证预取准确性、跨层带宽和恢复抖动。  
**n+2 HMM/PIM 异构 SoC：**以更大近端容量和更高带宽降低分层回流压力，以存算协同支撑 KV、激活和专家访问，重点面向更长上下文和更稳定低时延。

### 优先级说明（不需要展示在ppt中）

#### P0：先打通可观测与基础分层

**目标：**证明模型能加载、能生成少量 token、能记录关键瓶颈。  
**重点：**低比特加载、专家路由记录、KV 事件记录、主存回温路径、CANN/NPU 计数器、CPU/主存/冷存储链路实测。  
**边界：**不承诺稳定低时延，不承诺 1M 长上下文，不把冷存储放进逐 token 热路径。

#### P1：围绕专家命中率压榨 TPOT

**目标：**让 GLM5.2 不依赖全专家常驻，而依赖专家热度判断、分层驻留和提前预取。  
**重点：**热专家常驻、温专家预取、冷专家恢复、专家缺失代价、预取命中率、失败回退路径。  
**边界：**CPU 只做可证明收益为正的低频补算和状态编排，不泛化承接主干计算。

#### P2：补齐上下文恢复和阶段协同

**目标：**让长会话重入、代码仓复用和多轮上下文不被状态恢复拖垮。  
**重点：**前缀复用、KV/前缀分层、恢复与重算判定、预填/解码逻辑队列、恢复队列和反压策略。  
**边界：**冷存储只做容量层，不把随机小块恢复写成性能主路径。

#### P3：沉淀下一代硬件诉求

**目标：**将单卡压榨中的瓶颈沉淀为下一代系统和 SoC 诉求。  
**重点：**更大近端容量、更高主存带宽、更稳定互联、更低开销直通路径、统一状态对象运行时、硬件计数器和仿真接口。  
**边界：**NPU 到冷存储直通、HMM/PIM 和统一内存形态应作为下一代能力，不作为当前样机的默认性能承诺。

## 6. 讲稿/备份页：代表工作映射

> 主页面不建议放论文名。备份页只用于解释“为什么这样分层、哪些边界不能误读”。

### 6.1 专家权重容量管理

- **KTransformers：**核心不是简单把专家放到 CPU，而是通过热专家保留、路由专家外置、CPU 特化计算和异步调度，说明本地大 MoE 需要专家级分层。
- **FineMoE / DALI：**核心是利用任务语义、路由历史和残差信息预测专家热度，说明专家预取不能只靠静态频率。
- **FluxMoE：**核心是把专家变成可分页、可临时驻留的对象，说明近端内存应优先留给当前热工作集。
- **MoE-APEX：**核心是根据专家重要性和带宽代价使用不同精度，说明冷专家可以尝试降级恢复，但必须做质量回归。

### 6.2 上下文状态容量管理

- **LMCache / UCM：**核心是把 KV 和前缀从引擎内部缓存变成外部可管理对象，说明 A+K 需要统一状态对象层。
- **Mooncake：**核心是以 KV 为中心组织调度和存储资源，说明长上下文场景中状态复用比单次计算更重要。
- **CacheSlide：**核心是处理位置变化下的前缀复用，说明 Agent 场景不能只依赖完全一致的前缀命中。
- **Bidaw / SolidAttention：**核心是把主存和冷存储纳入状态恢复队列，说明冷状态管理必须让调度器知道状态位置和恢复代价。
- **Tutti：**核心是揭示普通冷存储回流会被小随机读、CPU 控制路径和同步开销反噬，说明直通路径属于下一代能力，不应提前承诺。

### 6.3 阶段级协同计算

- **NEO：**核心是 CPU 只承接部分解码注意力和对应状态，NPU/GPU 仍保持主路径，说明 CPU 参与计算必须受容量和重叠条件约束。
- **FlexInfer：**核心是按预填和解码阶段选择不同执行策略，说明 A+K 分工应由阶段和代价决定，而不是固定规则。
- **llm.npu：**核心是把 NPU 不适合的异常路径和小粒度路径交给 CPU/GPU，说明“整图交给 NPU”不是最优异构策略。

### 6.4 可观测与规格反推

- **ProfInfer：**核心是用低侵入方式记录算子、图、调度和硬件计数器，说明瓶颈归因必须进入运行时。
- **ServeGen / BurstGPT：**核心是生成和回放真实负载形态，说明一体机规格不能只用单轮请求估算。
- **LLMServingSim2.0：**核心是把模型阶段、硬件链路、状态对象和功耗纳入统一仿真，说明规格反推需要离散事件模型和实测校准。

## 7. 一句话压缩版

**A+K 协同不是把大模型硬塞进单卡，而是用 CPU 中心状态底座管理专家和上下文容量，用 NPU 保持热计算连续，用阶段协同掩盖冷路径，用可观测和仿真证明哪些硬件规格真正值得增加。**
