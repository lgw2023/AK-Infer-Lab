# A+K 协同技术挑战与趋势凝练报告

**Ascend NPU + Kunpeng CPU 面向小算力一体机的系统能力规划**

**研讨版｜更新至 2026-06-30**

调研范围：本文只聚焦 A+K 协同，即 Ascend NPU、Kunpeng CPU、DDR/NVMe、近端内存和运行时之间的协同机制。层次化内存、量化、MoE、Agent Runtime、多模态流水线和仿真评测只在其影响 A+K 协同边界时纳入。

材料口径：本文基于项目内《一体机小算力研讨逻辑笔记.md》、目标场景与硬件规格材料、仿真分析问题材料，以及 `AK 协同/` 目录下已有文献调研文档综合形成。代表论文的实验结果按现有调研材料和论文报告口径引用，不构成本项目硬件性能承诺；后续需要用 A+K 原型、真实 trace 和可控资源扰动复核。

本文按参考报告的写法组织：先给出执行摘要和术语边界，再严格沿用《一体机小算力研讨逻辑笔记.md》中已经确认的四个 A+K 技术方向，将挑战、代表文献、关键技术和 3-5 年趋势挂到对应方向下，不另起并列技术分类。

# 执行摘要

- 核心判断一：A+K 协同不是“让 CPU 多跑一点 Transformer”，而是让 Kunpeng CPU 从 host 变成“辅助计算 + 状态管理 + 调度控制 + 观测预测”的系统资源。Ascend NPU 仍承担 dense 主路径，Kunpeng 更适合承接工具、检索、KV/专家元数据、短小算子、预取、恢复和沙箱。

- 核心判断二：本项目的硬件边界决定了状态分层是主线，而不是可选优化。950PR Lite/塔式方案中，84GB Bailu 近端层、384GB DDR5 温层、4 x 512GB SSD 冷层，以及 NPU-CPU UB 200GB/s、CPU-DDR5 228GB/s、CPU-SSD 40GB/s 的链路差异，要求所有 offload 都必须 link-aware、tier-aware 和 stage-aware。

- 核心判断三：三个目标场景对 A+K 的需求不同。办公助手关注 30B/70B/120B 模型、多用户记忆、索引和 adapter；AI Coding Agent 关注 744B GLM、长上下文、Prefix/KV 复用、工具链和沙箱；多模态生成关注 text/vision encode、DiT/diffusion、VAE、codec、后处理和 I/O 的流水化。

- 核心判断四：A+K 的第一性挑战是“状态对象化”。KV、Prefix、Context、Expert、Adapter、Embedding Index、Tool Result 不应只是引擎内部临时缓存，而应成为可标识、可迁移、可预取、可压缩、可恢复、可观测的对象。

- 核心判断五：代表文献已经形成清晰趋势。llm.npu、PowerInfer-2、NEO、LIA、KVPR、Mooncake、LMCache、CacheSlide、MoE-Lightning、KTransformers、MORI、ProfInfer、Frontier 等工作共同说明：未来 3-5 年的小算力竞争点会从单点算力转向“阶段分治、状态分层、异构协同、trace 闭环”。

- 工程判断：近期最值得做的是围绕四个方向建立闭环验证：数据面先做 CPU 长尾任务白名单和 Kunpeng SVE/OMP 小算子库；状态面先做 KV/Prefix/Context/Expert 对象层；控制面先做 Agent-aware admission、resume-first 和 tool-aware KV lifecycle；横向闭环先做 A+K trace、profiler 和 what-if simulator。P/D 物理分离、跨节点专家池、near-data KV/PIM 等应在真实 trace 证明必要后再推进。

# 一、项目场景与硬件边界

## 1.1 目标应用场景

| 场景 | 项目材料中的模型与负载 | 对 A+K 协同的直接要求 |
| --- | --- | --- |
| 办公助手 Agent | 30B、70B、120B LLM；多轮对话、文档处理、企业知识检索、长期记忆 | Prefix/KV 复用、记忆索引、adapter/LoRA 热点管理、多用户状态隔离、CPU 侧检索和 rerank |
| AI Coding Agent | 744B GLM / GLM 5.2，1M 长上下文，工具调用、repo 文件读写、编译测试、沙箱 | 长上下文 KV/Prefix 分层、工具 gap 感知保温/换出、CPU 文件系统和沙箱调度、MoE 专家预取、resume-first 调度 |
| 多模态生成 | 14B Helios、27B Wan；AR + DiT、视频生成、后处理 | text/vision encode、LLM planning、DiT/diffusion、VAE、codec、postprocess 分阶段流水；CPU 处理 I/O/codec/后处理，NPU 处理密集主干 |

从仿真分析材料看，系统瓶颈不是单点算力不足，而是容量、带宽、CPU/NPU 交互、Agent 并发状态、工具 I/O、KV 生命周期和资源调度共同作用。典型信号包括：CPU/NPU 交互可达 10 次以上/token，小并发下 NPU 核利用率低于 20%；Coding Agent 工具 CPU 平均利用率低于 5% 但延迟差异可达 20 倍；工具调用和自反思会让 token 和 KV 压力倍增；Prefix Cache 在长序列下对 TTFT 影响极大。

## 1.2 当前硬件规格约束

| 层级/链路 | 项目材料中的规格 | A+K 协同含义 |
| --- | --- | --- |
| L1 近端层 | 84GB Bailu，NPU-Bailu 1TB/s | 热权重、热 KV、热点专家应尽量常驻；不能把温冷状态误当热层使用 |
| NPU-CPU | UB 200GB/s，部分 Mini 方案约 100GB/s | 适合大块预取、换出和批量状态转移；不适合细碎 ping-pong 式 CPU offload |
| L2 温层 | 384GB DDR5，CPU-DDR5 228GB/s；部分 Mini 方案 64GB DDR5、75GB/s | 冷权重、温 KV、Prefix、专家 staging、索引热段、adapter pool 的主承载层 |
| L3 冷层 | 512GB x 4 SSD，CPU-SSD 40GB/s；部分 Mini 方案 1T SSD、10GB/s | 冷 KV、冷专家、长期记忆、checkpoint、工具缓存；需要 I/O 聚合、写寿命控制和恢复时延预算 |
| 下一代路线 | 200GB/400GB HMM/HHM、Fabric >800GB/s、3D DRAM PIM 8.2TB/s、PIM 侧 KV/激活值示例 30GB | 支撑更大模型和近存计算；但仍需要显式 residency 控制，不能把统一/大容量内存当免费热层 |

硬件给出的核心启发是：A+K 系统的关键不在于“有没有卸载”，而在于“每个对象在哪一层驻留、何时预取、何时换出、由谁判断、如何证明收益大于搬运成本”。

# 二、按四个技术方向组织的挑战、代表文献与趋势

本章完全沿用笔记中的四个方向：数据面协同计算、状态与内存协同、控制面调度管理、横向闭环。MoE、KV、工具、P/D/E、多模态、仿真等不再作为并列“大技术点”另起分类，而是根据它们在系统中的作用挂到四个方向下。

## 2.1 方向一：数据面协同计算：CPU 直接参与端到端计算链路

方向边界：Ascend NPU 仍承担 dense Transformer 主路径；Kunpeng CPU 直接参与 NPU 不擅长、或不值得搬到 NPU 上做的长尾计算、短小计算、稀疏计算、工具计算、检索计算和 I/O 计算。这里的 CPU 不是传统 host，而是异构推理数据面协处理器。

### 问题挑战与代表文献

| 问题挑战 | 项目中的触发点 | 代表文献/系统 | 文献说明与 A+K 启示 |
| --- | --- | --- | --- |
| 挑战 1：动态图、短序列、小矩阵、outlier 与 fallback 让 NPU 主路径被碎任务打断 | 仿真材料显示 CPU/NPU 交互可达 10 次以上/token，小并发下 NPU 利用率低于 20% | Fast On-device LLM Inference with NPUs / llm.npu、HeteroInfer | llm.npu 将 prompt、tensor、block 拆分到 CPU/GPU/NPU 异步执行，说明 A+K 需要先定义“哪些碎任务留给 Kunpeng”，而不是把所有路径都强行图化到 NPU |
| 挑战 2：KV/attention/短矩阵是否值得 CPU 参与，取决于链路与 overlap，不取决于算子名称 | 84GB Bailu 热层不足以容纳长上下文和大模型全部状态；UB/DDR/SSD 带宽差异明显 | NEO、LIA、SuperInfer、APEX、CLO | NEO 和 LIA 证明 CPU 可参与 attention/KV/矩阵计算，但 SuperInfer/APEX/CLO 提醒 offload 会被带宽、同步粒度和 CPU 管理开销反噬；A+K 必须做 link-aware placement |
| 挑战 3：稀疏块、冷专家、低频权重和小模型 draft 不应挤占近端热层 | 744B Coding、MoE 专家、长上下文和多 Agent 并发会把热层容量快速吃满 | PowerInfer-2、KTransformers、MoE-Lightning | PowerInfer-2 将高密激活簇和稀疏簇分给不同处理器，KTransformers/MoE-Lightning 强调本地 CPU/NPU/I/O 混合；A+K 可把冷专家索引、低频专家、稀疏块放到 Kunpeng+DDR/SSD 冷路径 |
| 挑战 4：模型外计算已经进入 Agent critical path | Coding Agent 的 Bash/Edit/Read/Write、编译测试、文件 I/O、向量检索、办公文档处理、多模态 codec 都会影响 JCT | Agentix/Autellix、MORI、MARS、EPD Disaggregation、vLLM-Omni、DiffServe | Agent 与多模态文献共同说明，端到端体验不只由 decode TPOT 决定；Kunpeng 必须承担工具、检索、沙箱、codec、后处理等模型外数据面 |

### 解决当前问题的关键技术

| 关键技术 | 主要挑战 | 目标 | 技术要点 |
| --- | --- | --- | --- |
| 关键技术 1：A+K 数据面 placement profiler | CPU 是否参与计算缺少边界，容易把 offload 做成静态规则 | 建立“能下 CPU、值得下 CPU、不应下 CPU”的白名单和黑名单 | 采集算子形态、输入长度、批大小、UB bytes、DDR bytes、SSD bytes、CPU core.s、NPU wait；对 KV similarity、短序列矩阵、embedding/rerank、冷专家索引、tokenizer、codec 分别建立 cost model |
| 关键技术 2：Kunpeng SVE/OMP 长尾算子库 | 小矩阵、向量检索、KV 相似度和离散内存拷贝会被通用实现拖慢 | 把 CPU 数据面能力做成可复用库，而不是每个业务自写 | 基于 SVE+OMP 优化向量检索、rerank、KV 语义聚合、短序列矩阵、index 小算子、离散 I/O 聚合；与 MindIE/vLLM-Ascend 的 runtime queue 打通 |
| 关键技术 3：工具/检索/编解码的 CPU 隔离通道 | 工具任务平均 CPU 利用低但尾延迟高，容易干扰 NPU 推理恢复 | 让工具计算可调度、可限流、可观测 | 对 Bash、Read、Edit、Write、编译测试、向量检索、数据库、codec 建立单独队列；引入 CPU quota、I/O quota、优先级、预估恢复时间和沙箱资源账本 |
| 关键技术 4：计算/搬运/工具的异步重叠流水线 | 细粒度 CPU/NPU ping-pong 会抵消协同收益 | 用大块预取、批量转移和工具 gap 隐藏搬运 | 将 expert prefetch、KV restore、CPU 小算子、工具等待和 NPU decode 的时序统一编排；只允许能被 overlap 或显著降热层容量的任务下沉 CPU |

### 3-5 年趋势

短期 0-12 个月：从经验分工转向 profile-driven 分工，先验证 Kunpeng 在 KV similarity、embedding/rerank、短矩阵、工具链、codec 上的真实吞吐和尾延迟。

中期 12-30 个月：形成 CPU 数据面任务白名单、异步 overlap runtime 和 tool plane admission control，使 CPU 从“控制 host”变成“状态和长尾数据面协处理器”。

长期 30-60 个月：在统一地址空间、HMM/PIM、NPU-visible memory 逐步成熟后，形成自动化 CPU/NPU/近存混合执行计划，但仍以 dense 主路径留在 Ascend NPU 为基本边界。

## 2.2 方向二：状态与内存协同：CPU/DDR/SLC/SSD 承接模型状态

方向边界：CPU/DDR/SLC/SSD 不只是 offload 兜底，而是 KV、Prefix、Context、Expert、Adapter、Index、Checkpoint 的状态与内存平面。该方向直接决定 84GB Bailu、384GB DDR5、4 x 512GB SSD、UB 200GB/s、CPU-DDR5 228GB/s、CPU-SSD 40GB/s 这些硬件规格如何转化为可用能力。

| 问题挑战 | 项目中的触发点 | 代表文献/系统 | 文献说明与 A+K 启示 |
| --- | --- | --- | --- |
| 挑战 1：容量墙从权重扩展到 KV、Prefix、Context、Expert、Adapter 和 Index | 120B 办公、744B Coding、1M 上下文、多 Agent 并发远超 84GB 热层 | Mooncake、LMCache、SGLang HiCache、UCM | 这些工作把 KV cache 从引擎内部临时状态变成可外置、可持久、可分层、可观测的数据对象；A+K 应先做三层状态对象层 |
| 挑战 2：长上下文恢复不能全量搬运，必须只搬必要状态 | Prefix offload 在长序列下可显著影响 TTFT；128K/1M 上下文下冷 KV 恢复是关键 | InfiniGen、KVPR、CacheSlide、AlignedServe | InfiniGen 强调 selective KV，KVPR 用局部重算遮蔽传输，CacheSlide 处理位置漂移复用；A+K 需要 KV/Prefix/Context 的页级语义和恢复策略 |

### 解决当前问题的关键技术

| 关键技术 | 主要挑战 | 目标 | 技术要点 |
| --- | --- | --- | --- |
| 关键技术 1：层次化内存构建逻辑统一的“虚拟介质” | 负载动态变化与介质静态分配矛盾，导致热层拥塞或 DDR/SSD 资源浪费 | 在 Bailu、DDR5、SSD、未来 HMM/PIM 间形成统一对象视图和动态容量编排 | 借鉴 Slide 018 的写法，建设软件定义内存和异构介质编排：热对象在 Bailu，温对象在 DDR5，冷对象在 SSD；通过容量比感知算法、统一地址/对象空间、访问路径优选和动态重组支撑不同场景 |
| 关键技术 2：KV / Prefix / Context 对象层 | KV 生命周期短、跨轮复用难、恢复成本高 | 将 KV/Prefix/Context 从临时缓存升级为可迁移、可压缩、可恢复、可审计对象 | 对象元数据包括 `object_id`、`session_id`、`prefix_id`、`version`、`ttl`、`hotness`、`resident_tier`、`quality_metadata`、`resume_hint`、`compression_state`；支持显式 pin/evict/prefetch/restore |
| 关键技术 3：Expert / Adapter / Index 生命周期管理 | 热专家、温专家、冷专家、adapter 和检索索引混用同一热层会互相挤占 | 让不同状态对象按价值、生命周期和恢复成本分层驻留 | 热专家和热点 adapter 常驻 Bailu；温专家、温 KV、热索引段驻留 DDR5；冷专家、冷 KV、长期记忆、工具缓存落 SSD；专家对象记录 hotness、layer_id、session_affinity、precision、miss_penalty |
| 关键技术 4：Direct storage、离散 I/O 聚合与写寿命控制 | CPU bounce buffer、细碎 SSD I/O 和短生命周期 KV 写入会放大延迟和磨损 | 降低冷层访问对 decode 热路径的干扰 | 借鉴 Slide 020-021 的“权重与上下文卸载、A+K 协同加速”口径：做 NPU-friendly/NPU-SSD direct 路径、读写分离调度、128KB-1MB 级 KV/Prefix 页、离散 I/O 聚合、SSD round-robin、TTL 清理和写放大账本 |

### 3-5 年趋势

短期 0-12 个月：完成 HBM/Bailu -> DDR5 -> SSD 三层 KV/Prefix MVP，优先验证 Coding 长上下文和办公重复模板的 TTFT/P95 收益。

中期 12-30 个月：把 Expert、Adapter、Index、Tool Result 纳入统一状态对象层，形成 state object API、cache connector 和跨引擎复用能力。

长期 30-60 个月：向 HMM/HHM、near-data KV、PIM 和 NPU-visible memory 演进，但评价口径不是“可寻址容量变大”，而是热路径拷贝次数、恢复时延、bytes/J 和写寿命是否改善。

## 2.3 方向三：控制面调度管理：CPU 作为异构系统的大脑

方向边界：Kunpeng CPU 看到 session、workflow、tool、memory、expert、power、SLO 等全局信息，更适合做 Agent-aware / Model-aware / Memory-aware 控制面。它决定何时预取、何时换出、何时恢复、何时限流、何时让 NPU 优先跑 decode 热路径。

### 问题挑战与代表文献

| 问题挑战 | 项目中的触发点 | 代表文献/系统 | 文献说明与 A+K 启示 |
| --- | --- | --- | --- |
| 挑战 1：Agent 不是长一点的请求，而是 workflow/program | Coding Agent 有工具调用、文件读写、编译测试、失败重试、append-prefill 和多轮恢复 | Agentix/Autellix、MORI、MARS、PPD、AMPD、SAGA | 这些工作把调度对象从 request/token 扩展到 program、session、trajectory 和 workflow SLO；A+K runtime 必须接收 Agent metadata |
| 挑战 2：Prefill、Decode、Encode、Tool、KV Restore、Expert Prefetch 的资源画像不同 | 单机硬做 P/D 物理分离可能被 UB/DDR/SSD 搬运反噬，但完全不分阶段又会互相干扰 | DistServe、Splitwise、SuperInfer、EPD Disaggregation、vLLM-Omni、DiffServe | 阶段分离的本质不是部署形态，而是资源画像不同；Mini 应先做逻辑分池，塔式再做 worker pool，小集群再考虑物理分离 |
| 挑战 3：KV、专家和工具任务相互影响，单点调度会造成尾延迟 | 工具 CPU 平均利用率低但工具延迟差异大；KV 和专家恢复与 NPU decode 抢链路 | NEO、AlignedServe、MoE-Lightning、FineMoE、Mooncake | 调度器需要同时理解 load、SLO、memory、expert、tool idle window；只优化 batch 或只优化 KV 都不够 |
| 挑战 4：多用户办公和 Agent swarm 需要前后台隔离与恢复优先 | 办公助手存在多用户、多 adapter、多知识库；Agent swarm 会把 CPU、DDR、SSD 调度压力放大 | Agentic workload 相关研究、MARS、MORI | 控制面要从吞吐优先转向 workflow completion 优先，支持 resume-first、前台 decode 保护、后台索引/清理限流 |

### 解决当前问题的关键技术

| 关键技术 | 主要挑战 | 目标 | 技术要点 |
| --- | --- | --- | --- |
| 关键技术 1：Agent-aware 控制元数据协议 | 底层不知道哪些 KV 会很快恢复、哪些工具会长时间占 CPU、哪些请求属于同一任务 | 让 runtime 能按任务关键路径做调度 | 下沉 `session_id`、`trajectory_id`、`workflow_stage`、`tool_start/tool_end`、`expected_resume_time`、`prefix_id`、`priority`、`resume_budget`、`memory_intent`、`adapter_id`、`expert_hint` |
| 关键技术 2：阶段化队列与资源预算 | Prefill/Decode/Tool/KV Restore/Expert Prefetch/Multimodal Encode 混跑会造成尾延迟 | 用逻辑分池保护热路径，同时避免过度物理拆分 | 建设 prefill queue、decode queue、tool queue、KV restore queue、expert prefetch queue、multimodal encode queue；每个队列有 CPU、NPU、UB、DDR、SSD 预算和优先级 |
| 关键技术 3：Tool-aware KV lifecycle 与 resume-first 调度 | 工具等待窗口没有被用来保温、换出和预取，恢复时又发生冷启动 | 用工具 gap 隐藏状态迁移，让即将恢复的任务优先 | 工具开始时预测恢复时间，决定 KV pin/evict/prefetch；工具结束前触发 prefix/KV restore；resume 请求优先进入 decode 热队列 |
| 关键技术 4：Memory-aware + Expert-aware 联合调度 | KV、Prefix、Expert、Adapter、Index 互相挤占热层 | 把状态对象的价值和恢复成本纳入 admission | 根据 hotness、miss penalty、resident tier、UB budget、DDR/SSD budget 决定对象驻留；对专家预取、KV 恢复、adapter 切换做统一调度，而不是各模块各自抢资源 |

### 3-5 年趋势

短期 0-12 个月：以 AI Coding Agent trace 为主，验证 tool-aware KV lifecycle、resume-first 和 CPU 工具隔离能否降低 JCT 与恢复时延。

中期 12-30 个月：扩展到办公多用户、多 adapter、多知识库，以及多模态 stage graph，形成 workflow-aware scheduler 和 memory-aware router。

长期 30-60 个月：支持 2-8 节点小集群的状态池、专家池和 Agent swarm 调度，从 request SLO 走向 workflow SLO 与 energy-aware scheduling。

## 2.4 方向四：横向闭环：性能预测、能效建模、Trace 与仿真

方向边界：这是异构协同的证据闭环与决策支撑层，不是独立业务路径。它服务于前三个方向，用 trace、profiler、cost model、resource perturbation 和 what-if simulator 判断哪些 A+K 协同值得做，哪些会被 UB/DDR/SSD 搬运成本抵消。

### 问题挑战与代表文献

| 问题挑战 | 项目中的触发点 | 代表文献/系统 | 文献说明与 A+K 启示 |

| 挑战 1：没有真实 trace，无法反推 CPU、DDR、SSD、UB 和近端容量规格 | 当前需要冻结场景、采集 trace、识别瓶颈、反推算存网参数 | BurstGPT、Chakra、Frontier、LLMServingSim 2.0 | BurstGPT 提供真实请求 trace，Chakra 强调 execution trace schema，Frontier/LLMServingSim 支持 what-if simulation；A+K 需要自己的场景级 trace |
| 挑战 2：没有细粒度 profiler，CPU/NPU 协同容易停留在经验判断 | CPU/NPU 交互、工具 I/O、KV/专家恢复和多模态阶段都需要归因 | ProfInfer、系统 profiler 相关工作 | ProfInfer 说明 eBPF/低侵入采样可以把应用阶段和系统资源关联起来；A+K 应采集阶段、对象、资源三层事件 |

| 挑战 4：能效、SSD 寿命和 P95/P99 比平均吞吐更接近产品风险 | 小算力一体机面向本地交互，多用户与 Agent swarm 会暴露尾延迟和写放大 | LMCache、Mooncake、HiCache、Agentic workload 研究 | 横向闭环要把 bytes/token、bytes/J、write amplification、state restore P95、workflow JCT 纳入指标，而不是只看 tokens/s |

### 解决当前问题的关键技术

| 关键技术 | 主要挑战 | 目标 | 技术要点 |
| --- | --- | --- | --- |
| 关键技术 1：三场景 A+K trace schema | Coding、办公、多模态的瓶颈不同，不能用单一聊天负载代表 | 形成可复现、可对比、可反推规格的统一 trace | 任务层记录 workflow/stage/tool；模型层记录 prefill/decode/encode；状态层记录 KV/Prefix/Expert/Adapter/Index；资源层记录 CPU/NPU/UB/DDR/SSD；结果层记录 TTFT、TPOT、JCT、P95/P99、成功率 |
| 关键技术 2：阶段级 profiler 与对象级账本 | 仅有端到端 latency 无法判断瓶颈来自算力、容量、带宽还是 I/O | 把每个阶段的资源消耗和每个状态对象的驻留成本对齐 | 采集 CPU core.s、NPU util、UB bytes、DDR bytes、SSD bytes、KV hit/miss、expert hit/miss、tool wait、resume latency、write amplification |
| 关键技术 3：link-aware / tier-aware cost model | 不是所有 offload、预取和分离都值得做 | 在执行前估计搬运、计算、等待和恢复成本 | 对 84GB Bailu、384GB DDR5、4 x 512GB SSD 建立容量账本；对 UB 200GB/s、CPU-DDR5 228GB/s、CPU-SSD 40GB/s 建立传输预算；输出 placement 和 admission 建议 |
| 关键技术 4：what-if simulator 与硬件参数反推 | 缺少资源扰动就无法判断下一代硬件该加哪一项 | 用真实 trace 推导 CPU 核数、DDR 带宽、SSD IOPS、近端容量、UB 带宽、HMM/PIM 需求 | 对单资源进行扰动：只扩 DDR、只增 SSD IOPS、只加 CPU 核、只增 UB、只增近端容量；比较 TTFT、TPOT、JCT、P95/P99、能耗和写寿命 |

### 3-5 年趋势

短期 0-12 个月：形成 AI Coding、办公助手、多模态生成三类 benchmark 和 trace dashboard，用于约束 A+K 原型范围。

中期 12-30 个月：建立在线 cost model 和 simulation loop，让数据面 placement、状态对象层和控制面调度共享同一套资源账本。

长期 30-60 个月：把 trace-driven co-design 变成下一代 A+K SoC、HMM/PIM、direct storage 和小集群规格输入，形成“场景 trace -> 仿真 -> 原型 -> 硬件规格”的闭环。

# 三、场景化落地建议

## 3.1 AI Coding Agent

核心矛盾：744B GLM / GLM 5.2、1M 长上下文、repo 文件系统、工具调用、编译测试、沙箱和多轮恢复共同造成容量、状态和 CPU/NPU 交互压力。

按四个技术方向落地：

| 方向 | 优先动作 | 核心指标 |
| --- | --- | --- |
| 数据面协同计算 | CPU 工具链、repo 文件扫描、编译测试、向量检索、KV similarity、短矩阵和冷专家索引白名单 | 工具 wait hidden ratio、CPU 干扰度、NPU wait、JCT |
| 状态与内存协同 | Prefix/KV/Context/Expert 三层对象化，重点降低长上下文 TTFT 和恢复时延 | prefix hit、KV restore P95、expert hit、NVMe bytes/token |

# 五、面向研讨材料的凝练表述

可以把 A+K 协同方向在汇报中凝练成以下主线：

1. **数据面协同计算：Kunpeng CPU 不替 Ascend NPU 跑 dense 主模型，而是承担 KV 相似度、短矩阵、稀疏块、冷专家索引、tokenizer、工具、检索、codec 和后处理等长尾数据面。**
2. **状态与内存协同：84GB Bailu、384GB DDR5、4 x 512GB SSD 的分级结构，要求 KV、Prefix、Context、Expert、Adapter、Index 做成可调度对象，而不是临时缓存。**
3. **控制面调度管理：CPU 作为 Agent-aware / Memory-aware / Expert-aware 控制平面，统一管理 load、SLO、session、workflow、tool、offload、prefetch 和 resume-first。**
4. **横向闭环：A+K 协同必须用 profiler、trace、仿真和 cost model 判断哪些协同值得做，哪些会被 UB/DDR/SSD 搬运成本抵消。**
5. **AI Coding Agent 的胜负手在长上下文、工具 gap 和专家/KV 恢复；办公助手的胜负手在记忆、索引、多用户和 adapter；多模态生成的胜负手在 stage graph 和媒体 I/O。**
6. **3-5 年趋势会从单点 kernel 优化转向状态中心化 runtime：stage-aware、agent-aware、memory-aware、expert-aware、energy-aware。**
7. **下一阶段不应继续无边界调研，而应冻结场景、采集 trace、建立 profiler、识别瓶颈、反推算存带宽参数，并验证 A+K 协同原型。**

# 六、建议的下一步验证问题

1. 数据面协同计算：Kunpeng 侧 SVE/多线程在 KV similarity、embedding/rerank、短序列矩阵、专家索引、codec 上的真实吞吐和能效边界是多少。
2. 数据面协同计算：哪些 CPU 任务可被工具等待、专家预取或 KV 恢复窗口隐藏，哪些会直接增加 TPOT/JCT。
3. 状态与内存协同：在 120B 办公助手、744B Coding、14B/27B 多模态三类场景中，各自的 KV/Prefix/Expert/Index footprint 是多少，生命周期多长，冷热分布如何。
4. 状态与内存协同：对当前 84GB Bailu + 384GB DDR5 + 4 x 512GB SSD 方案，哪些对象应进入 L1/L2/L3，迁移阈值如何设定。
5. 控制面调度管理：Coding Agent 工具等待窗口是否足够隐藏 KV/专家预取；不同工具类型对 CPU、内存、SSD 的干扰程度如何。
6. 控制面调度管理：多用户办公和 Agent swarm 下，resume-first、前后台隔离、adapter 切换和 memory-aware routing 对 P95/P99 的收益是多少。
7. 横向闭环：UB 200GB/s、CPU-DDR5 228GB/s、CPU-SSD 40GB/s 下，哪些 offload 策略能覆盖搬运成本，哪些会反噬 TPOT。
8. 横向闭环：下一代 HMM/PIM/near-data KV 的规格，应由哪些真实 trace 指标反推，而不是由峰值 TOPS 单独决定。
