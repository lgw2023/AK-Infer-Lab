## 0. 页面标题与主结论

### 标题
**A+K 协同：围绕 Agent 关键路径重构 CPU/NPU/内存/存储分工**

### 副标题
面向 AI Coding Agent、办公助手 Agent、多模态生成的一体机小算力增强路径

### 顶部一句话结论
**A+K 协同不是让 CPU 替代 NPU，而是让 Kunpeng CPU 承接状态、工具、短算子和调度，让 Ascend NPU 专注高密度主推理，形成“放得下、推得快、恢复得回”的本地 Agent 算力系统。**

### 场景牵引短句
- **AI Coding Agent：**744B GLM、M 级上下文和编译/测试循环，核心矛盾是长前缀、KV 驻留、冷专家回取和 CPU 工具等待。
- **办公助手 Agent：**30B/70B/120B、本地知识库和文档工具，核心矛盾是多轮上下文复用、检索/Rerank 和低尾延迟交互。
- **多模态生成：**Helios-14B、Wan-27B、AR+DiT/DiT/VAE/Codec 流水，核心矛盾是主生成、滚动缓存、视频编解码和临时 I/O 协同。
- **系统目标：**放得下、跑得快、恢复得回、功耗可控。

## 1. 数据面协同计算：CPU 直接参与端到端计算链路

### 问题
- **主推理和长尾计算错配：**NPU 适合高密度矩阵计算，但 tokenizer、小算子、短矩阵、工具执行、检索和媒体处理会拖慢端到端任务。
- **卸载不等于加速：**CPU/NPU 分工如果切得过细，UB/PCIe、DDR、同步和调度开销会抵消计算收益。
- **模型外链路进入关键路径：**Coding 编译测试、文件 I/O、浏览器、RAG、视频编解码和后处理会直接影响 JCT，而不是只影响后台资源。

### 技术及趋势
- **阶段级 A+K 分工：**按 Prefill、Decode、工具、检索和多模态阶段明确 CPU/NPU 分界，让 NPU 承担主模型和高密度矩阵，让 CPU 承接 tokenizer、工具、检索、短矩阵和媒体链路。
- **CPU 侧短算子加速：**利用 Kunpeng 向量能力加速 KV 相似度、Prefix 匹配、向量检索和离散 I/O 聚合，减少小并发交互中的等待和数据回取开销。
- **MoE 冷专家协同：**热专家留在 NPU 近端，冷/温专家由 CPU/DDR 侧完成索引、预取或应急计算，降低专家 miss 对 TPOT 和任务尾延迟的影响。
- **工具链路并行：**将 shell、编译测试、文档解析、视频编解码和后处理纳入 CPU 数据面，在工具等待和生成流水之间做重叠，避免 NPU 主推理被模型外任务拖停。

### 未来趋势
A+K 数据面会从固定 CPU/NPU 分工走向按阶段和代价动态选择的混合执行，CPU 承接不规则、短小和状态相关计算，NPU 保持高密度主路径连续运行。

## 2. 状态与内存协同：CPU/DDR/SLC/SSD 承接模型状态

### 问题
- **近端内存被共同挤压：**权重、KV、Prefix、Context、MoE 专家和 workspace 同时竞争 Bailu/HBM，百 B 模型、M 级上下文和多 Agent 会快速触顶。
- **状态放下去后还要取回来：**KV、Prefix 和专家下沉到 DDR/SSD 后，如果恢复慢、预取不准或 I/O 抖动，会直接放大 TTFT、TPOT 和 JCT 尾延迟。
- **Agent 复用不是简单前缀命中：**代码仓、工具描述、文档片段和长期记忆往往相对稳定但位置漂移，传统 exact prefix cache 难以充分复用。
- **专家热度随任务变化：**MoE 专家访问受 session、输入、阶段和层级影响，静态常驻或简单 LRU 都难以稳定命中。

### 技术及趋势
- **KV/Prefix/Context 分层管理：**将可复用前缀、长上下文 KV 和工具回填状态按热度放入 Bailu、DDR 和 SSD，降低重复 prefill 与近端容量压力。
- **权重与专家分层卸载：**将大模型权重和 MoE 专家按热度执行常驻、预取、换出和落盘，使热数据留近端、温数据可回温、冷数据可恢复。
- **跨层直通与预取：**通过 NPU-SSD 直通、CPU 侧聚合预取和读写分离，缩短冷 KV/冷专家回流路径，减少 CPU 中转和小 I/O 放大。
- **恢复优先的状态管理：**把状态命中率、恢复时延、专家 miss、重算成本和质量损失一起纳入策略，避免只追求“容量放大”而牺牲任务尾延迟。

### 未来趋势
状态与内存协同会从引擎内部缓存优化演进为推理数据面能力，KV、Prefix、Context 和 Expert 将被统一管理、分层放置、按需恢复并可观测治理。

## 3. 控制面调度管理：CPU 作为异构系统的大脑

### 问题
- **Request-level 调度看不到任务结构：**Agent 由多轮 LLM、工具、检索、失败重试和状态恢复组成，单次请求的 TTFT/TPOT 不能代表完整任务体验。
- **CPU 工具队列会反向影响 NPU：**沙箱、编译测试、浏览器、文档解析、向量库和文件系统队列会决定下一轮模型何时恢复。
- **工具等待改变 KV 策略：**短工具适合保留 KV，中等工具适合卸载并预取，长工具可能更适合释放或重算。
- **小并发场景容易队头阻塞：**Mini/塔式常是少量用户、高突发、前后台混部，长 prefill 会阻塞即将完成的短任务。

### 技术及趋势
- **Agent 负载感知调度：**将 session、workflow、stage、tool 和 expected resume time 下沉到 Runtime，让调度对象从单请求升级为任务关键路径。
- **工具感知 KV 保温：**按工具类型和预计时长选择 pin、offload+prefetch、discard 或 recompute，减少工具等待期间的无效占用和恢复重算。
- **CPU/NPU 联合准入：**同时观察 NPU 队列、CPU 工具池、DDR/SSD 水位、KV 空间和前台 SLO，优先保护可恢复、接近完成和交互敏感任务。
- **前后台隔离与恢复优先：**用 prefill 预算、resume-first 和 completion-aware 策略限制长任务阻塞，压低多轮 Agent 的 JCT 尾部。

### 未来趋势
控制面会从请求队列调度器演进为 Agent Runtime 策略层，以 session、program 和 workflow 为基本对象，围绕任务完成时间、状态价值和恢复优先级做全局调度。

## 4. Trace 与仿真闭环：性能预测、能效建模与硬件参数反推

### 问题
- **没有 Trace 就无法判断协同是否有效：**CPU offload、KV 分层、专家预取和 NPU-SSD 直通都可能被搬运、同步、恢复和管理开销抵消。
- **Tokens/s 不能解释任务慢在哪：**真实瓶颈可能在工具、检索、KV 恢复、SSD I/O、CPU 队列、前后台干扰或失败重试。
- **硬件诉求需要因果证据：**只看资源利用率不能反推规格，必须通过限算力、限容量、限带宽、限 CPU、限 DDR、限 SSD 和限功耗建立性能拐点。

### 技术及趋势
- **任务级 Trace 采集：**记录 Task、Turn、Request、Tool、KV、Memory、Resource 和 Power，把 prefix hit、KV restore、tool wait、CPU Core·s 和 NVMe bytes 对齐到同一条任务链路。
- **A+K 分阶段 Profiling：**按 Prefill、Decode、工具、检索、多模态和恢复阶段归因 NPU、Bailu、DDR、UB/PCIe、SSD 与 CPU 工具池的真实开销。
- **单资源扰动验证：**分别限制计算、近端容量、链路带宽、CPU、DRAM、SSD 和功耗，确认瓶颈因果关系，避免把相关性误当成硬件需求。
- **仿真反推规格：**把 KV/专家迁移、重算、预取、工具等待和功耗约束放入 what-if 仿真，反推 CPU 核数、DDR 带宽、SSD IOPS、UB 带宽和 Bailu 容量对 JCT 与 Tasks/J 的敏感度。

### 未来趋势
Trace 与仿真会从事后性能分析演进为软硬件协同设计输入，每个技术点和硬件规格都需要绑定 Trace、扰动实验和仿真结果。

## 5. 底部硬件诉求摘要

**硬件诉求摘要：**Mini / DV100 Lite 重点验证阶段级 A+K 分工、CPU 侧短算子、KV/Prefix 分层和工具隔离；塔式 / n+1 分级内存系统重点把 Bailu、DDR5、SSD、UB、NPU-SSD 直通和 CPU 沙箱池做成可编排层级；n+2 PIM+HMM 异构 SoC 重点用 HMM 承载大容量权重与状态，用 PIM 承接 KV/激活近存高带宽访问，并同步暴露观测、QoS、统一编址和仿真接口。
