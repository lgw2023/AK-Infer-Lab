## 0. 页面标题与主结论

### 标题
**A+K 协同异构推理：面向小算力一体机的主链路的计算加速与状态管理**

### 副标题

面向 AI Coding Agent、办公助手 Agent、多模态生成的一体机小算力增强路径

### 顶部一句话结论
**A+K 协同主旨是把 Ascend NPU 难以高效覆盖的小粒度、稀疏化和状态相关计算交给 Kunpeng CPU，并通过分层内存与目标硬件性能预测，形成“放得下、推得快、可预判”的本地协同推理系统。**

### 场景牵引短句

- **AI Coding Agent：**744B GLM、M 级上下文和长会话重入，核心矛盾是长前缀、KV 驻留、MoE 专家容量和恢复时延。
- **办公助手 Agent：**30B/70B/120B、本地文档和多轮上下文，核心矛盾是重复前缀复用、长上下文状态管理和低尾延迟交互。
- **多模态生成：**Helios-14B、Wan-27B、AR+DiT/DiT/VAE 流水，核心矛盾是主生成算力、滚动 KV/Noise Cache、激活状态和视频后处理带来的容量压力。
- **系统目标：**模型放得下、推理跑得快、状态恢复得回、规格可反推。

## 1. 数据面协同计算：CPU 参与模型推理阶段计算

### 问题

- **NPU 主路径存在细粒度低效区：**Prefill、Decode、Attention、MoE 和多模态生成主链路中，短序列矩阵、稀疏块、outlier、KV 选择和专家索引等计算不一定适合全部放到 NPU 执行。
- **CPU 参与计算必须受搬运成本约束：**如果把模型内子计算切得过细，UB/PCIe 同步、DDR 访问和数据格式转换会抵消 CPU 侧计算收益。
- **MoE 与长上下文放大协同诉求：**大 MoE 专家、长上下文 KV 和滚动生成状态使 NPU 近端容量持续承压，需要 CPU 在“选择、预取、补算、回退”环节参与推理链路。

### 关键技术点
- **关键技术：模型推理阶段的 A+K 分工。**按 Prefill、Decode、Attention、MoE 和 AR+DiT 阶段识别 CPU 可承接的子计算，让 NPU 保持高密度主路径连续执行，CPU 只处理短小、稀疏、状态相关和回退类计算。
- **关键技术： KV 相关计算的 CPU 侧加速。**将 KV 相似度、Prefix 匹配、部分 Attention 选择和长上下文热度判断下沉到 Kunpeng 侧执行，减少无效 KV 回取和重复 prefill。
- **关键技术： MoE 专家协同计算。**通过专家热度预测、冷/温专家索引、CPU 侧 fallback expert 和专家预取协同，降低 expert miss 对 TPOT 与尾延迟的影响。
- **关键技术：多模态生成状态辅助计算。**围绕 AR+DiT 的滚动 KV、Noise Cache 和分段生成状态，在 CPU 侧完成状态筛选、回温判断和轻量补算，减少 NPU 显存压力并稳定生成流水。

### 未来趋势
A+K 协同计算会从固定算子卸载走向推理阶段内的选择性混合执行：NPU 专注密集主算，CPU 只在可掩盖、可批量、收益为正的模型内子路径参与计算。

## 2. 模型状态 A+K 回流：CPU 侧状态计算与 NPU 连续推理恢复

### 问题

- **NPU 近端容量被权重、KV 与专家状态共同挤压：**百 B 模型、M 级上下文、MoE Expert 和多模态滚动状态会持续占用 Bailu/HBM，单纯分层存放无法保证主推理连续执行。
- **状态回流进入 A/K 关键路径：**KV、Prefix、Expert 和 Noise Cache 下沉到 DDR/SSD 后，恢复延迟、UB/PCIe 搬运、DDR/SSD 抖动和格式转换会直接放大 TTFT、TPOT 与 P95/P99。
- **状态价值需要 CPU 侧动态判断：**代码仓、系统提示、文档片段、长期上下文和 MoE 专家访问具有阶段性复用特征，不能只依赖 LRU 或 exact prefix。

  **MoE 与多模态生成放大恢复风险：**冷专家 miss、KV 回流不准、Noise Cache 回温失败会让 NPU 主生成链路出现等待和重算。

### 关键技术点

- **关键技术： KV/Prefix/Context 分级承载。**将可复用前缀、长上下文 KV 和生成中间状态按热度放入 Bailu、DDR 和 SSD，降低重复 prefill 和近端容量占用。
- **关键技术：权重与专家分层卸载。**将大模型权重和 MoE 专家按热度执行常驻、预取、换出和落盘，使热数据留近端、温数据可回温、冷数据可恢复。
- **关键技术：跨层直通回流。**通过 NPU-SSD 直通、CPU 聚合预取、异步 load/store 和读写分离，将冷 KV/冷专家的小粒度回取改造成可批量、可预测、可掩盖的数据回流。
- **关键技术：恢复优先的状态管理。**把状态命中率、恢复时延、专家 miss、重算成本和质量损失联合纳入策略，避免只追求容量放大而牺牲任务尾延迟。

### 未来趋势

模型状态管理会从通用缓存/卸载演进为 A+K 推理数据面能力：CPU 负责KV、Prefix、Context 和 Expert 等的状态价值判断与回流组织，NPU 保持主推理连续执行，DDR/SSD/HMM/PIM 成为可预测、可恢复、可度量的模型状态层。

## 3. 横向闭环：目标硬件性能预测与能效建模

### 问题
- **A+K 协同收益边界不清：**CPU 参与模型内计算、KV 回流、专家预取和 NPU-SSD 直通，都可能被链路带宽、同步、数据格式转换和恢复开销抵消。
- **目标硬件规格缺少推理链路证据：**CPU 核数、DDR 带宽、SSD IOPS、UB 带宽、Bailu 容量和功耗上限，需要从真实模型阶段和状态回流开销中反推，而不能只按峰值算力估计。
- **能效与尾延迟强耦合：**CPU/NPU 分工、状态驻留、DDR/SSD 访问和功耗限制会共同影响 TPOT、JCT、P95/P99 和 Tasks/J，需要在目标硬件上形成可解释的性能模型。

### 关键技术点

- **关键技术：构建模型阶段硬件画像。**在 Mini、DV100 Lite 和塔式形态上拆解 Prefill、Decode、KV 恢复、专家回取和多模态生成阶段，形成 NPU、Bailu、DDR、SSD、UB 与 CPU 占用的阶段化画像。
- **关键技术：构建 A+K 协同收益预测模型。**将 CPU 子计算、KV/专家迁移、重算、预取、直通和数据格式转换纳入统一代价模型，提前判断某项协同是否收益为正。
- **关键技术：构建面向整机规格的瓶颈验证方法。**以任务完成时延和单位任务能耗为主指标，逐项收紧近端内存、DDR/SSD 通路与 CPU 资源，判断 A+K 协同收益是否被数据搬运和状态恢复开销抵消，并形成 CPU、内存/存储与直通链路的规格建议。

### 未来趋势
A+K 横向闭环会从事后跑分转向面向目标硬件的性能预测：每一项 CPU/NPU 协同机制都需要能提前估算收益边界、能效变化和对硬件规格的敏感度。

## 4. 底部硬件诉求摘要

**硬件诉求摘要：**Mini / DV100 Lite 重点验证模型推理阶段 A+K 分工、CPU 侧 KV/短矩阵/冷专家计算和状态分层；塔式 / n+1 分级内存系统重点把 Bailu、DDR5、SSD、UB、NPU-SSD 直通做成可预测的数据回流层级；n+2 PIM+HMM 异构 SoC 重点用 HMM 承载大容量权重与状态，用 PIM 承接 KV/激活近存高带宽访问，并同步暴露性能计数器、QoS 和规格预测接口。
