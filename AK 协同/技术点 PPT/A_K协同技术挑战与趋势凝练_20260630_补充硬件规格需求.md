# **1. 执行摘要**

核心判断：A+K 协同不是“让 CPU 多算一点”，而是把 Kunpeng CPU、DDR/NVMe 与
Ascend NPU 围绕同一条 Agent critical path
组织成计算、状态、调度和评测闭环。NPU 仍承担主 LLM/VLM 推理、热点
attention、热点 MoE 专家与 DiT/diffusion 主干；CPU
则承担长尾计算、工具链、向量检索、短序列矩阵、KV/专家状态平面、控制面调度和观测预测。

| **方向** | **核心判断** | **代表证据** | **对当前项目的落点** |
|:---|:---|:---|:---|
| 1 数据面协同计算 | CPU 从 host 转为异构推理数据面协处理器，参与 KV 选择、短矩阵、稀疏块、冷专家、小模型 draft、embedding/rerank、工具执行、视频编解码。 | llm.npu、NEO、LIA、APEX、KTransformers、PowerInfer-2。 | 先做阶段级 CPU/NPU 执行矩阵和 SVE/OMP 加速库，避免把 dense Transformer 主路径错误下放 CPU。 |
| 2 状态与内存协同 | KV/Prefix/Context/Expert 从引擎内部缓存变成可观测、可迁移、可持久化的对象层，由 Bailu/DDR5/SSD/HMM 等分层承载。 | InfiniGen、Mooncake、LMCache、CacheSlide、MoE-Lightning、FineMoE、KVPR。 | 优先做热/温/冷对象层、工具感知 KV TTL、专家冷热预取、NPU-SSD 直通与恢复路径。 |
| 3 控制面调度管理 | 调度单位从 request/token 上升为 session/program/workflow；CPU 作为 Agent-aware / Memory-aware / Expert-aware 控制平面。 | Agentix、KVFlow、TokenCake、Continuum、Helium、MARS、Policy Runtime、AgentServe。 | 先做轻量 metadata/runtime、continuation priority、prefill budget、CPU/NPU 联合准入，再做复杂 DAG。 |
| 4 Trace 与仿真闭环 | 没有 trace、扰动实验和 cost model，就无法判断 A+K offload 是否收益为正，也无法反推下一代硬件规格。 | BurstGPT、ServeGen、Agentic AI Workload Characteristics、ProfInfer、Chakra、CCL-Bench、Frontier。 | 建立 Agent Appliance Trace Schema、资源扰动矩阵、A+K what-if simulator 和参数需求书。 |

近期优先级：P0 应落在 metadata、Trace Schema、KV/Prefix/Context
分层、工具感知 KV TTL/offload、CPU/NPU 联合准入、MoE
专家冷热预取和多模态 pipeline profiling。P1 扩展到统一对象层、选择性
E/P/D 或 P/D 分离、直接 SSD 路径和异构 runtime。P2
再考虑统一地址空间、near-data KV/expert 和自动混合执行计划。

# **2. 输入边界：目标场景与硬件约束**

目标应用场景固定为三类：AI Coding Agent、办公助手
Agent、多模态生成。硬件规格不是抽象算力比较，而由三类负载共同牵引：办公助手
30B/70B/120B LLM，AI Coding 744B GLM，多模态生成 14B Helios 与 27B Wan。

| **场景** | **任务包络** | **主要瓶颈** | **A+K 协同含义** |
|:---|:---|:---|:---|
| AI Coding Agent | 仓库理解、修改代码、编译测试、错误修复、工具循环；代表需求是 744B GLM、M 级长上下文。 | 长上下文 prefill / append-prefill、KV 驻留、工具等待、编译/测试 CPU·s、repo 文件 I/O、MoE 专家容量。 | A+K 需要同时优化 NPU 主推理、CPU 工具执行、KV/Prefix 复用、冷专家预取和 resume-first。 |
| 办公助手 Agent | 文档/表格/PPT/邮件/企业知识库；代表模型是 30B/70B/120B。 | 多轮工具调用、RAG、向量检索、短工具结构化输出、隐私本地化、少量并发低尾延迟。 | CPU 承担检索、rerank、schema/cache、文档解析和状态管理；NPU 承担高价值生成。 |
| 多模态生成 | 文生图/视频、图生视频、办公素材生成、长视频分段生成；代表模型为 Helios-14B、Wan-27B。 | Text/vision encode、LLM planning、DiT/diffusion、VAE、视频编解码、后处理、临时文件与 SSD 写放大。 | A+K 应做 E/P/D/codec 分阶段流水，CPU 做 media I/O 与后处理，NPU 做主生成与热点 attention。 |

| **硬件路径** | **对象** | **规格与约束** | **对 A+K 的设计要求** |
|:---|:---|:---|:---|
| DV100 Lite 原型 / Mini 形态 | DV100 Lite + Kunpeng CPU | 2027 形态材料中为 64GB Bailu 放热权重、64GB DDR 放冷权重，NPU 与 CPU/DRAM 之间 PCIe 64GB/s；适合做冷热权重调度、阶段级预取和多模态 media pipeline 验证。 | 不能把它作为 744B 或 400GB KV 的最终容量方案；应作为 A+K placement、CPU 编码/工具、权重预取和搬运成本的原型验证平台。 |
| 950PR Lite / 塔式近端分级内存系统 | 950PR Lite + Kunpeng CPU | L1 84GB Bailu 放热权重与热 KV；L2 384GB DDR5 放冷权重与温 KV；L3 512GB×4 SSD 放冷 KV；NPU-Bailu 1TB/s，NPU-CPU UB 200GB/s，CPU-DDR5 228GB/s，CPU-SSD 40GB/s。 | 适合验证 KV/Prefix/Expert 热温冷对象层、工具感知 KV TTL、专家预取、NPU-SSD 直通、A+K 成本模型和塔式多用户准入。 |
| 下一代 1920 / HMM+PIM 方向 | 1920 SoC / 1920Next | 原 1920 方案存在算力与容量余量问题；新方案引入 200GB/400GB HMM/HHM、3D DRAM PIM、>800GB/s Fabric，PIM 侧示例为 30GB KV/激活值与 8.2TB/s 访存能力，目标是更大容量和近存计算。 | 适合将对象层、专家热集、KV/激活值近存布局、统一编址和 10ms 低时延推理目标前置验证。 |
| 竞品与路线压力 | DGX Spark / DGX Station 等 | DGX Spark 对应 Mini 工作站，公开材料口径为 200B、GB10、1P FP4、128GB 内存、约 240W；DGX Station 对应塔式工作站，口径为 400B-1T、GB300、20P FP4、748GB 混合内存、约 1.6kW。 | A+K 不应只比 TOPS，应把“放得下、推得快、恢复得回、功耗可控、硬件需求可由 trace 反推”作为竞争力指标。 |

# **3. 技术方向一：数据面协同计算**

## **3.1 方向定位**

数据面协同计算的核心不是让 Kunpeng CPU 承担 dense Transformer
主推理，而是把 NPU/GPU 不擅长或不值得上 NPU/GPU
的长尾计算、稀疏计算、短小计算、工具计算和 I/O 计算下沉到
CPU。这个方向对应“CPU 作为异构推理数据面协处理器”。

| **核心挑战** | **说明** | **最相关场景** |
|:---|:---|:---|
| 挑战 1：NPU 主路径与长尾算子错配 | NPU 适合规则化大矩阵，但 tokenizer、shape/metadata、小算子、outlier、短序列矩阵、稀疏 index 和 JSON/grammar 等固定开销可能拖累端到端链路。 | 办公助手短工具输出、Coding Agent 频繁 tool call、多模态 preprocessing。 |
| 挑战 2：细粒度 offload 易被搬运成本反噬 | CPU/NPU 分工过细会放大 UB/PCIe、DDR、cache 同步和调度开销；必须以阶段/算子 profiler 和 cost model 决策。 | Decode attention、KV 部分恢复、MoE 冷专家、短序列矩阵。 |
| 挑战 3：工具与多模态外部计算被低估 | Coding 编译测试、浏览器、检索、数据库、视频编解码、后处理和封装常在任务关键路径上，但传统推理 benchmark 不覆盖。 | AI Coding Agent 与多模态生成端到端 JCT。 |
| 挑战 4：MoE 让 CPU 有机会进入正式计算平面 | MoE 冷专家/低频专家、小专家 fallback、expert prediction 与短矩阵可由 CPU 参与，但需要 expert-aware scheduler 与 CPU kernel。 | 744B GLM / DeepSeek 类大 MoE。 |

| **代表研究** | **会议/机构** | **核心机制与结果** | **A+K 迁移启发** |
|:---|:---|:---|:---|
| llm.npu / Fast On-device LLM Inference with NPUs | ASPLOS 2025；PKU/BUPT | Prompt 级分块、tensor outlier 放 CPU/GPU、block 级按硬件亲和度调度；报告 prefill 最高 22.4×、能耗 30.7× 降低。 | 证明 NPU 不必吃完整链路；A+K 可迁移为 NPU 跑规则化密集块，CPU 做 outlier、图管理、动态依赖与 fallback。 |
| NEO | MLSys 2025；PKU/UC Berkeley/Harvard 等 | 非对称 GPU-CPU pipeline，把部分 decode attention 与 KV state 放到 CPU；T4 上最高 7.5× 吞吐提升。 | Kunpeng 可吸收部分 decode attention/KV 工作，换取更大 NPU batch 和更低 HBM 压力。 |
| LIA / FlexInfer / APEX | ISCA 2025 / MLSys 2025 / arXiv 2025 | CPU 矩阵能力、阶段建模、profiling-informed CPU-GPU overlap；强调 placement 由硬件 profile 决定。 | A+K 必须先测 Kunpeng SVE/矩阵/向量能力，再决定短矩阵、rerank、index 和部分 attention 是否下放。 |
| KTransformers / CoX-MoE | SOSP 2025 / DAC 2026 | CPU 侧专家 kernel、异步 CPU-GPU 调度、Expert Deferral、coalesced expert execution。 | MoE 场景下 CPU 不是权重搬运器，而应成为专家执行器、索引器和预取控制器。 |
| EPD Disaggregation / DiffServe / CodecSight | 2025–2026 多模态系统方向 | Encode-Prefill-Decode 拆分、query-aware model scaling、codec metadata 协同。 | 多模态 A+K 要按 stage graph 分工：CPU 做 media pipeline，NPU 做主生成。 |

| **关键技术点** | **构建/探索能力** | **策略或效果** | **3–5 年趋势** |
|:---|:---|:---|:---|
| A+K 阶段级执行矩阵 | 构建 prefill / decode / tool / retrieval / multimodal 的阶段画像：NPU 负责密集主路径，CPU 负责 tokenizer、context assembly、embedding/rerank、KV similarity、tool、codec。 | 形成 placement 规则与动态 fallback；以 JCT、TTFT、TPOT、CPU Core·s、UB bytes/token 验证。 | 2026–2027 完成单机 profile + 规则矩阵；2027–2029 进入 cost-model 驱动；2029–2031 形成自动混合执行计划。 |
| 鲲鹏向量/短矩阵加速库 | 构建 SVE/OMP 向量检索、短序列矩阵、KV 语义聚合、离散 I/O 聚合、JSON/grammar/schema cache 加速库。 | 让 CPU 在“短小碎”与“近数据聚合”场景中真正降低关键路径。 | 先覆盖办公/RAG/Coding 工具链，再扩展到多模态 preprocessing 与冷专家小算子。 |
| MoE CPU 侧专家执行与预测 | 探索 CPU 侧小专家/fallback 专家、冷专家预计算、expert prediction、coalesced expert batching。 | 提升 expert hit rate，降低 miss penalty 与 SSD/DDR 拉取开销。 | 2026–2027 做热门 MoE 模型 trace；2027–2029 建设专家调度器；2029+ 与近存/统一内存结合。 |
| 工具与媒体数据面 | CPU 侧沙箱、编译测试、文档解析、视频编解码、VAE 后处理、封装与结果写盘独立资源池化。 | 降低工具队列对 NPU serving 的反向干扰，提升 Tasks/J 与前台交互稳定性。 | 工具资源进入 serving control loop；多模态生成引入媒体引擎和 SSD pipeline 指标。 |

# **4. 技术方向二：状态与内存协同**

## **4.1 方向定位**

状态与内存协同是 A+K 的主战场。它把
KV、Prefix、Context、Expert、Adapter、Index、Checkpoint
从“引擎内部临时缓存”提升为跨 HBM/Bailu、DDR5、SSD、HMM/PIM
的统一对象。该方向直接决定硬件规格：DDR 容量、DDR 带宽、SSD
IOPS、NPU-SSD 直通、UB/PCIe 带宽、NPU 可访问内存语义。

| **核心挑战** | **说明** | **最相关场景** |
|:---|:---|:---|
| 挑战 1：模型权重与 KV 同时挤压近端内存 | 100B–150B、200B+、744B MoE 与 M 级长上下文会让权重、KV、workspace、专家、索引共同竞争 Bailu/HBM。 | 办公助手百 B、Coding 744B、Deep Research 长上下文。 |
| 挑战 2：状态可放下不等于恢复得回 | KV/Prefix/Expert 放到 DDR/SSD 后，恢复延迟和跨层带宽会直接进入 TTFT/TPOT/JCT；错误预取会放大 P95/P99。 | 工具返回、resume prefill、长上下文 decode。 |
| 挑战 3：传统 prefix cache 命中边界不足 | Agent 中很多上下文不是完全前缀相同，而是相对位置稳定、绝对位置漂移；repo skeleton、工具描述、长期记忆块会移动。 | AI Coding Agent、办公知识库、长期记忆 Agent。 |
| 挑战 4：MoE 专家不是统一热度 | 不同 session、阶段和输入会产生专家热度差异；只做静态 expert cache 会导致 miss、I/O 放大和尾延迟。 | DeepSeek/GLM/Qwen-MoE 等稀疏模型。 |

| **代表研究** | **会议/机构** | **核心机制与结果** | **A+K 迁移启发** |
|:---|:---|:---|:---|
| InfiniGen | OSDI 2024；Seoul National University | ephemeral pruning + speculative prefetch，只拉回必要 KV；报告最高 3× speedup。 | KV 不应作为整块搬运；A+K 需要细粒度 KV object、预取 hint 与 sparse gather。 |
| Mooncake / LMCache / UCM / HiCache | FAST 2025 / 工程系统 | 把 KV 从引擎内缓存变成外部状态层，支持 CPU RAM、SSD、远端 store、PD disaggregation、持久前缀。 | 一体机需要 HBM→DRAM→Storage 的对象层，SSD/NVMe 是推理数据面的正式 tier。 |
| CacheSlide | FAST 2026；上海交大/浪潮/北大/华为云 | 跨位置 KV 复用，解决相对顺序稳定但绝对位置漂移的问题。 | Coding repo、工具定义、长期记忆块需要 cross-position context reuse，而不只是 exact prefix。 |
| KVPR | ACL Findings 2025；USC | “先传激活 + GPU 重算 + 并行传剩余 KV”遮蔽 PCIe 瓶颈；decode 延迟最高降低 35.8%。 | 在 UB/PCIe 受限的一体机上，传输-重算协同可能比全量 KV 回传更稳。 |
| MoE-Lightning / FineMoE / DAOP / MoE-APEX | ASPLOS 2025 / EuroSys 2026 / DATE 2025 / ASPLOS 2026 | 分页权重、专家语义/trajectory 预取、低精冷专家应急、CPU/GPU/I/O 流水。 | 构建热专家常驻、温专家预取、冷专家落盘、低精 fallback 的专家状态层。 |

| **关键技术点** | **构建/探索能力** | **策略或效果** | **3–5 年趋势** |
|:---|:---|:---|:---|
| KV/Prefix/Context/Expert 对象层 | 为每个对象维护 object_id、owner、version、ttl、hotness、tier、quality metadata、resume_hint、security scope。 | 支持跨层 locate/load/store/pin/move/compress/remove；避免策略散落在推理引擎内部。 | 2026–2027 做 KV/Prefix；2027–2029 扩展 Context/Expert；2029+ 对接统一地址空间/HMM/PIM。 |
| 热/温/冷分层策略 | L1 Bailu/HBM 放热权重和热 KV；L2 DDR5 放冷权重和温 KV；L3 SSD 放冷 KV/冷专家/检查点。 | 解决“放得下”与“恢复得回”的双目标；以 kv_restore_ms、NVMe bytes/token、P95 JCT 评估。 | 分层规则从 LRU 走向 next-use、TTL、semantic hotness、workflow criticality。 |
| NPU-SSD 直通与 I/O 聚合 | NPU 直接访问 NVMe，CPU 做元数据与聚合；读写分离、round-robin、SM/AIV 空闲感知。 | 降低 Prefix KVCache 128KB–1MB 等 I/O 粒度下的恢复时延，减少 CPU bounce buffer。 | 先做 Prefix/KV 冷层；后续用于 MoE 冷专家、checkpoint、视频临时对象。 |
| MoE 专家状态管理 | 热专家常驻、温专家预取、冷专家落盘；expert map、session/trajectory hotness、低精冷专家应急。 | 让 200B–700B+ 稀疏模型在一体机中“放得下、跑得快”。 | 2026–2027 验证单机专家预取；2027–2029 引入专家对象层；2029+ 扩展到小集群 expert pool。 |
| 恢复面与可靠性 | checkpoint、session resume、engine crash 后 KV/Prefix 恢复、校验与版本控制。 | 评价指标不能只看 cache hit，还要看恢复时延、恢复后任务成功率和安全边界。 | 长期 Agent 和企业本地部署会把恢复能力变成产品差异点。 |

# **5. 技术方向三：控制面调度管理**

## **5.1 方向定位**

控制面调度管理把 CPU 定义为异构系统的大脑。它不以 CPU
计算能力为核心，而以 CPU
可观测的信息广度、状态维护能力和全局控制粒度为核心。调度对象需要从
request/token 上升到 session、program、workflow，并把工具资源、KV
状态、专家热度、SLO、功耗与安全 scope 纳入同一控制回路。

| **核心挑战** | **说明** | **最相关场景** |
|:---|:---|:---|
| 挑战 1：request-level 调度丢失 Agent 语义 | Agent 的任务完成时间由多轮 LLM、工具、检索、失败重试与恢复共同决定；单个请求的 TTFT/TPOT 不等于任务体验。 | AI Coding、办公、多 Agent 编排。 |
| 挑战 2：工具资源与 NPU 资源强耦合 | 只看 NPU 利用率会忽略 CPU sandbox、编译测试、浏览器、检索、文件系统和网络队列；工具拥塞会制造更多暂停 session 与 KV 压力。 | Coding Agent / 办公 Agent。 |
| 挑战 3：KV 保温策略需要工具阶段信息 | 工具执行空窗有毫秒到分钟的长尾差异，短工具应保留 KV，长工具应卸载或接受重算。 | ReAct 类多轮任务。 |
| 挑战 4：静态部署与固定 batch 不适合小并发交互 | 小算力设备并发低、负载突发、前后台混部；需要动态 prefill token budget、continuation priority、模型/量化档位。 | Mini 工作站 / 部门级塔式。 |

| **代表研究** | **会议/机构** | **核心机制与结果** | **A+K 迁移启发** |
|:---|:---|:---|:---|
| Agentix | NSDI 2026；UC Berkeley/Google DeepMind/SJTU 等 | 把完整 agent program 建模为一等调度对象；报告同延迟约束下 program throughput 约 4–15×。 | 先实现 workflow_id、session_id、stage、continuation priority，避免长 prefill 阻塞即将完成的分支。 |

# **6. 技术方向四：横向闭环：性能预测、能效建模、Trace 与仿真**

## **6.1 方向定位**

横向闭环不是单独的业务执行路径，而是前三类协同能力的证据层和决策层。A+K
的核心风险是“看起来可卸载，实际被 UB/PCIe/DDR/SSD 搬运、同步、CPU
管理开销反噬”。因此必须以 profiler、trace、resource perturbation、cost
model 和 what-if simulator 决定哪些协同值得做。

| **核心挑战** | **说明** | **项目风险** |
|:---|:---|:---|
| 挑战 1：缺少真实 Agent trace 会误判硬件诉求 | Agent 真实瓶颈常来自多轮重入、长寿命 KV、工具调用、失败重试，而不是单次长 prompt。 | 会导致显存、CPU、SSD、互联需求书偏离真实场景。 |
| 挑战 2：只看 tokens/s 无法解释任务慢在哪 | 需要 JCT、任务成功率、goodput、KV GB·s、CPU Core·s、NVMe bytes、Tasks/J、P95/P99。 | 需要从模型级指标转向任务级和系统级指标。 |
| 挑战 3：资源瓶颈必须由扰动证明 | 观察到高利用率不是因果证据；需要单资源限算、限内存、限带宽、限 CPU、限 DRAM、限 IO、限网络、限功耗。 | 形成下一代硬件参数反推。 |
| 挑战 4：A+K 需要边界模型 | SuperInfer、APEX、CLO 等提醒：CPU offload、KV rotation、细粒度管理都有收益边界。 | 需要 link-aware / energy-aware / tail-aware cost model。 |

| **代表资源/论文** | **类型** | **核心机制** | **A+K 迁移启发** |
|:---|:---|:---|:---|
| BurstGPT / ServeGen / Mooncake traces | 真实请求 trace / 生产负载生成 | 提供真实到达、会话、token 长度、burstiness、client mix 与长上下文 replay。 | 为一体机压测提供 request-level reality 与生产负载合成底座。 |

# **8. 3–5 年趋势对应的硬件规格需求**

本节补充硬件需求口径：当前阶段的关键技术点应绑定 mini、塔式、DV100 Lite、950PR Lite 等已有硬件规格做原型验证；3–5 年趋势则应进一步转化为下一代分级内存系统和 PIM+HMM 异构 SoC 的规格需求。以下内容只用于 PoC 方案筛选和下一代参数反推，不构成采购或性能承诺；任何固定容量、带宽、CPU 核数、SSD 等级或功耗目标都需要由目标模型、量化格式、KV 结构、上下文长度、并发 trace、工具负载和实测服务曲线重新计算。

## **8.1 硬件 sizing 估算口径与边界**

对 A+K 协同而言，硬件需求不能只按模型参数量或 NPU TOPS 推断。近端容量需求至少应拆为权重、KV Cache、activation/workspace、专家热集、runtime buffer、工具/检索/媒体中间状态和安全余量；跨层搬运需求至少应拆为 Bailu/HBM 访问、NPU-CPU UB 或 PCIe、CPU-DDR、CPU-SSD、NPU-SSD 直通、Fabric/PIM 侧访问和 SSD 写放大。标准 attention 下，KV 可按层数、KV head 数、head dimension、KV dtype 和上下文/并发估算；GQA/MQA/MLA、MoE、KV 压缩和 cross-position reuse 会改变结果，因此不能只按 30B/120B/744B 参数量线性外推。

| **估算对象** | **当前可以稳定估算的部分** | **必须由模型/trace 提供的部分** | **硬件规格输出** |
|:---|:---|:---|:---|
| 权重容量 | 参数量、量化 bit 数、scale/zero-point 和 kernel format 开销可以形成初筛。 | MoE 激活专家数、热专家集合、专家命中率、权重预取窗口。 | Bailu/HMM 常驻热权重量、DDR/HMM 温层容量、SSD/SLC 冷层容量和预取带宽。 |
| KV / Prefix / Context | 单 token KV 的理论公式、目标上下文长度和会话并发可以形成下界。 | GQA/MQA/MLA 结构、压缩率、prefix reuse、工具等待分布、resume 频率。 | 近端 KV 容量、DDR offload pool、SSD 冷层容量、恢复时延和 NPU-SSD 直通需求。 |
| CPU 工具与媒体链路 | 编译测试、检索、文档解析、视频编码等可用 CPU Core·s 和队列时间评估。 | 工具类型分布、sandbox 并发、视频分辨率/时长/编码档位、失败重试。 | CPU 核数/主频、内存带宽、媒体/编码加速单元、工具资源隔离和功耗 headroom。 |
| 跨层搬运 | 现有口径可测：DV100 Lite PCIe 64GB/s；950PR Lite UB 200GB/s、CPU-DDR5 228GB/s、CPU-SSD 40GB/s。 | 每 token bytes moved、KV restore 粒度、专家 miss、SSD 读写放大、P95/P99 抖动。 | UB/PCIe/Fabric 带宽、NPU-SSD 直通、NVMe 队列深度/IOPS、SSD 耐久与 QoS。 |
| 功耗与热 | mini 与塔式功耗边界可作为形态约束，非固定性能承诺。 | 并发、上下文驻留、CPU 工具峰值、PIM/HMM 访问强度和温控曲线。 | power cap、动态频率、散热余量、Tasks/J 和功耗-SLO 曲线。 |

# **10. 建议优先启动的 P0/P1/P2 项目清单**

| P0 | 工具感知 KV TTL/offload | 工具等待期间 KV 的保留/卸载直接决定恢复时延。 | resume latency、reload miss、GPU KV 利用率、tool wait hidden ratio。 |
| P0 | MoE 专家冷热预取与 CPU fallback 原型 | 744B/大 MoE 是否可用取决于专家而非总 FLOPs。 | expert hit rate、miss penalty、NVMe bytes/token、quality drop、TPOT。 |

| P1 | NPU-SSD 直通与离散 I/O 聚合 | 冷 KV/冷专家/检查点恢复路径会进入前台关键路径。 | kv_restore_ms、SSD IOPS、读写放大、IO 对计算影响比例。 |
| P1 | 多模态 E/P/D/media pipeline profiling | 视频生成瓶颈会从主推理扩散到 encode/VAE/codec/postprocess。 | 首帧时延、稳帧率、生成 JCT、VAE/codec 占比、临时文件 bytes。 |
| P2 | A+K what-if simulator / 参数需求书 | 从“技术点”升级为“下一代节点设计输入”。 | 仿真误差、资源弹性、配置-收益 Pareto、参数下界与保守裕量；输出 n+1 分级内存和 n+2 PIM+HMM SoC 两套硬件规格需求。 |
