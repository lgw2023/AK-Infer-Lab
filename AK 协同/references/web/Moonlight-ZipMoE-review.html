    [论文评述] ZipMoE: Efficient On-Device MoE Serving via Lossless Compression and Cache-Affinity Scheduling

    ZipMoE 旨在解决 Mixture-of-Experts (MoE) 架构在资源受限的边缘设备上部署时遇到的内存占用过高问题，特别是在需要无损压缩以保留模型行为、不依赖有损量化的情况下。该论文提出了一种高效、语义无损的设备端 MoE 推理系统 ZIPMOE。

    核心问题与挑战： MoE 模型通过稀疏激活机制显著提升了模型容量，但同时也使系统瓶颈从计算转移到内存 I/O。现有的设备端 LLM 推理优化方法主要依赖模型量化和剪枝等有损压缩技术，但这可能改变模型行为，引入安全漏洞。另一种方法是专家 offloading，将非活跃参数卸载到低级存储，但这与边缘设备的硬件特性存在根本性不匹配。 具体来说，在基于 Unified Memory Architecture (UMA) 的移动 SoC 平台（如 NVIDIA Jetson）上，CPU 和 GPU 共享内存和总线资源。现有的 offloading 方案通常假设 CPU 主内存是独立的外部存储，这在 UMA 架构下并不成立。在边缘设备上，模型参数常需从 NVMe SSD 读取，其 I/O 带宽远低于服务器级 CPU 内存，导致严重的 I/O 停滞（I/O stalls），将推理速度瓶颈化，并使多核 CPU 和 GPU 资源严重利用不足。

    ZIPMOE 的核心方法： ZIPMOE 提出了缓存-调度协同设计，通过无损压缩和缓存感知调度来解决上述问题，将 I/O 瓶颈转换为以计算为中心的工作流，实现高效并行化。

    1. 无损压缩与参数重构：

      • 位域分解 (Bit-Field Decomposition)： ZIPMOE 利用 BF16 参数中固有的信息冗余。BF16 格式由一个符号位、8 个指数位和 7 个尾数位组成。研究发现符号位和尾数位接近随机分布，而指数位具有高度统计冗余（低熵）。
      • 压缩流程： 在离线初始化阶段，模型参数首先进行位域分解，将低熵的指数位（E-chunks）与高熵的符号-尾数位（SM-chunks）分离。E-chunks 被进一步分片并使用 LZ4HC 或 ZSTD 等通用无损压缩器压缩，SM-chunks 被打包成字节对齐表示。所有组件和元数据序列化后 offload 到 NVMe SSD。
      • 并行解压缩与零拷贝： 在运行时，ZIPMOE 在 CPU 上执行 E-chunks 的并行解压缩。实验表明，多线程解压缩的延迟低于 SSD I/O，且资源干扰可忽略不计。此外，SM-chunks 被直接读入 host-pinned 内存，实现零拷贝，避免冗余数据传输。
      • 内存合并的 Tensor 恢复 Kernel： 解压缩的 E-chunks 和加载的 SM-chunks 在 GPU 上通过内存合并 (memory-coalesced) 的 CUDA kernel 重新组装成 BF16 格式的 tensor。该 kernel 通过向量化加载指令获取连续数据段，在寄存器内完成位级重构，并使用向量化存储写回，最大化 DRAM 带宽利用率，有效覆盖 CPU 异步解压缩的延迟。

    2. 缓存感知调度与分级缓存管理：

      • 压缩状态抽象 (Compression States)： 为了实现细粒度内存管理，ZIPMOE 引入了压缩状态抽象。一个专家可以根据其 E-chunks 或 SM-chunks 是否被缓存，分别定义为 E-expert 或 SM-expert；若两者都被缓存，则为 compressed-expert。
      • DAG 任务表示： 每个专家重构任务被建模为一个有向无环图 (DAG)，节点表示 I/O、解压缩和 tensor 恢复等操作，边表示优先级约束。ZIPMOE 根据专家的运行时压缩状态动态构建 DAG 拓扑。
      • Cache-Affinity Scheduler： 调度器旨在最小化每个稀疏层的 makespan（最后一个专家执行的完成时间）。
        • 任务划分： 将专家任务分为两类：Type-I 任务需要从存储加载 SM-chunks（高 I/O 成本）；Type-II 任务的 SM-chunks 已缓存。
        • 块构建 (Block Construction)： 调度器将 Type-I 和 Type-II 任务分组，形成块 (blocks)，以重叠 SM-chunk 加载和解压缩。任务按对应专家执行时间 $p_{n(j)}$ 降序排序。调度器迭代选择 Type-I 任务作为块基准，然后从 Type-II 任务中增量插入到不引入额外解压缩线程空闲时间的位置。当系统变为 compute-bound（计算完成时间显著落后于 I/O）或无任务可插入时，块构建结束。块顺序执行，块内任务按优先级执行。
        • 性能保证： 调度器产生的调度 $ALG$ 在 makespan 上满足理论近似比 $ALG \le (3 - 1/L) \cdot OPT$，其中 $L$ 是解压缩线程数，$OPT$ 是最优值。
      • Compression-Aware Cache Management：
        • 分层缓存池： ZIPMOE 将可用内存划分为 F (full tensors)、C (compressed tensors)、S (SM-chunks) 和 E (E-chunks) 四种缓存池，提供不同的延迟-内存权衡。
        • 基于排名的工作负载建模： 通过历史专家激活计数，构建基于排名的专家边际包含概率列表 ($f_r$)，捕获工作负载固有的 skewness。
        • 池调度 (Pool Dispatching)： 专家根据其观察到的排名和预设阈值 ($\tau_i$) 分发到缓存池。当缓存池溢出时，按最低激活频率驱逐专家。
        • 分级缓存规划 (Hierarchical Cache Planning)： 通过最小化稀疏 MoE 层的预期 makespan 来确定最优缓存分区。使用动态规划 (DP) 算法计算在特定排名区间内观察到精确缓存命中数 ($h_p$) 的概率 $\Phi_p(h_p)$。这产生了一个最大熵分布，确保了方案的贝叶斯鲁棒性。然后，通过网格搜索枚举可行的缓存分区配置，计算每种配置的预期 makespan，并选择最优配置。

    实现与评估： ZIPMOE 基于 HuggingFace Transformers 框架实现，集成了 lz4 和 zstd 作为无损压缩后端。系统利用 C++/CUDA 实现执行引擎、调度器和内存管理。 在 NVIDIA Jetson AGX Orin 64GB/32GB 平台上，使用 DeepSeekV2-Lite、Qwen1.5-MoE (decoder-only) 和 SwitchTransformers-Large-128 (encoder-decoder) 模型进行评估。对比基线包括 MoE-Infinity、DeepSpeed 和 Accelerate。 结果显示：

    • 实时响应性： 在 offloading 必要场景下，ZIPMOE 实现了 62.65%-97.97% 的 TPOT (Time-Per-Output-Token) 减少和 53.25%-87.90% 的 TTFT (Time-To-First-Token) 减少。
    • 系统吞吐量： 在 batch size 为 1、4、16 的批处理推理中，ZIPMOE 始终优于所有基线，对于 decoder-only 模型有 1.79×-42.49× 的提升，对于 encoder-decoder 模型有 1.31×-5.82× 的提升。
    • 端到端延迟： 在不同输出长度下，ZIPMOE 的端到端推理速度快 3.03×-42.49×（decoder-only）和 1.11×-5.64×（encoder-decoder）。
    • 消融研究： ZIPMOE 的内置缓存算法（尤其是通过缓存规划优化的版本）在延迟-吞吐量权衡方面优于 FIFO、Marking 和 LRU 等基线缓存策略。

    结论： ZIPMOE 通过细粒度的稀疏层执行编排，结合无损压缩和缓存感知调度，显著提高了边缘设备上 MoE 推理的内存和 I/O 效率，同时保留了模型行为，为在资源受限的设备上部署大型 MoE 模型提供了高效的解决方案。
