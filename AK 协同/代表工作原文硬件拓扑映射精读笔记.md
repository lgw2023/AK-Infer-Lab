## 1. NEO

原文：`references/papers/NEO__arxiv-2411.01142.pdf`

### 1.1 核心架构一句话

NEO 不是普通的 KV cache swap，也不是把整条推理主链路搬到 CPU；它把一部分请求的 decode attention 计算和对应 KV cache 留在本地 CPU/CPU memory 侧，同时让 GPU 继续承担 prefill、线性层、GPU-request attention 和主计算，并用 asymmetric GPU-CPU pipeline 与 load-aware scheduling 平衡 CPU/GPU。

### 1.2 实际硬件链路

| 链路/组件 | NEO 原文中的角色 | 前端拓扑显示建议 |
|---|---|---|
| GPU / HBM | 保存模型权重、activation、GPU-request 的 KV cache；执行 prefill、线性层和 GPU-side attention | 作为热路径主算区高亮 |
| CPU main memory | 保存 CPU-request 的 KV cache，形成 CPU-cache | 作为温路径容量层高亮，不要画成冷存储 |
| CPU cores / SIMD | 执行 CPU-request 的 decode attention；实现 PACPU，使用 ISPC 生成 SIMD CPU kernel | 高亮 CPU compute，但标注“仅 decode attention 子路径” |
| PCIe / GPU-CPU communication | 不再每个 decode step 反复搬整段 KV；主要传必要张量，并在 prefill layer-wise 产生 KV 后尽早传输 | 高亮为温路径传输，不要写成主瓶颈已消失 |
| Scheduler / runqueue | CPU 上有 prefill waitqueue、GPU decoding runqueue、CPU decoding runqueue；每轮比较 GPU-only 与 two-batch pipeline | 高亮控制面，展示“动态选择 GPU-only 或 GPU+CPU” |
| NUMA | 原文核心设计没有把 NUMA 当作关键机制 | 前端可把 NUMA 放入 CPU memory 风险提示，不应作为 NEO 主组件 |

### 1.3 解决的问题

- GPU memory 限制 batch size 和并发，导致 GPU compute 资源空闲。
- 只把 KV cache 放到 CPU memory 并按需 swap 回 GPU，会被 GPU-CPU PCIe 带宽和同步开销卡住。
- 全量 CPU offload 会让 CPU 内存带宽和计算能力成为瓶颈。

### 1.4 关键机制

- Partial offloading：把已经 prefill 的请求划分为 GPU-request 和 CPU-request。GPU-request 的 KV cache 在 HBM；CPU-request 的 KV cache 在 CPU main memory。
- Asymmetric pipelining：每轮构造两个不对称 sub-batch。一个主要在 GPU 上跑 prefill/linear/GPU attention，另一个把 decode attention 放到 CPU；用 GPU 工作覆盖 CPU attention。
- Layer-wise KV transmission：新 prefill 请求的 KV value 按 layer 生成后即可通过 PCIe 传输，不等整个 iteration 完成。
- Load-aware scheduling：每轮估计 GPU-only schedule 与 asymmetric schedule 的吞吐，基于 offline profiling 的 `linear/GPU attention/CPU attention` 时间选择策略；同时避免 CPU decoding runqueue 过载。
- CPU attention kernel：PACPU 采用 vLLM 类 paged KV，使用 ISPC/SIMD 优化内存访问；原文明确 attention 是 memory-bandwidth-bound，因此 CPU 的价值来自相对可用的内存带宽和容量，而不是通用 CPU FLOPs。

### 1.5 边界与失败条件

- CPU-request 比例过高时，CPU memory bandwidth 和 CPU attention kernel 会成为上限。
- 如果 GPU 不能 offload 出足够 KV，two-batch asymmetric pipeline 的 batch size 反而可能小于 GPU-only，吞吐下降。
- NEO 的收益依赖 CPU/GPU overlap；如果 CPU attention 不能被 GPU prefill/linear/attention 覆盖，TPOT/P99 会恶化。
- NEO 不证明“CPU 可普遍加速推理”，只证明特定 decode attention 子路径可在容量受限、可重叠条件下利用本地 CPU。

### 1.6 前端校准点

- 点击 NEO 时，应高亮：`GPU/HBM -> PCIe/DMA -> CPU cores -> CPU memory -> scheduler`。
- NEO 详情栏应写“CPU-request 的 KV 和 decode attention 留在 CPU 侧”，不要写成“KV offload 到 CPU 后再频繁搬回 GPU”。
- NUMA 可以作为 CPU memory 边界项，而不是 NEO 的主贡献。
- 颜色建议：GPU/HBM 为热路径，PCIe/CPU memory/CPU attention 为温路径，scheduler 为控制面。

## 2. Tutti

原文：`references/papers/Tutti__arxiv-2605.03375.pdf`

### 2.1 核心架构一句话

Tutti 把 SSD-backed KV cache 从 CPU-centric 的三层 HBM-DRAM-SSD 数据路径，改成 GPU-centric 的 HBM-SSD KV object I/O：CPU 仍负责元数据和异步准备，但从 KV 传输关键数据路径和每次 I/O 控制路径中退出。

### 2.2 实际硬件链路

| 链路/组件 | Tutti 原文中的角色 | 前端拓扑显示建议 |
|---|---|---|
| GPU HBM | 存 vLLM 预分配 KV block pool；GPU 侧持有 I/O 所需映射元数据 | 高亮热路径与 I/O 发起端 |
| NVMe SSD | KV cache 的 SSD tier；多个 SSD 用于并行 object 读写 | 高亮冷路径容量层 |
| GPU direct/object I/O | GPU file pool、NVMe file pool、P2P memory mapping table 将 KV block 映射到 object/file | 作为 Tutti 的核心链路显示 |
| GPU io_uring | SQ/CQ ring buffer 在 GPU HBM 中，CPU 可映射准备，GPU issue I/O kernel 并轮询 completion | 作为“设备侧 I/O 控制面”显示 |
| CPU | 负责 CPU-side hash mapping、metadata、提前准备 IOCB、加载 I/O kernel；不在每次 I/O critical path | 用紫色控制面标注，不要作为 HBM-SSD 数据路径 |
| GPU SM partition | 通过硬件资源隔离让 I/O control kernel 和 compute kernel 避免互相饿死 | 在详情栏写明，不必在主图新增大节点 |
| Mooncake/RDMA | 单机外扩时 Tutti 可用 Mooncake 做集群控制面；当前 remote path 仍有 CPU-side fallback | 作为远端扩展边界，不应作为 Tutti 单机 fast path |

### 2.3 解决的问题

- Paged KV layout 让逻辑连续 KV 变成大量小随机 I/O，SSD 带宽无法有效发挥。
- 即使使用 GDS，CPU 仍需发起每个 I/O，CPU 处在 I/O control critical path。
- SSD->DRAM->HBM bounce copy、CPU-GPU 同步和小 I/O 让 GPU 出现高比例 bubble，有时 KV reuse 还不如 recomputation。
- naive layer-wise pipelining 会让 read/write 争用 SSD 资源，也会让 I/O kernel 争用 GPU SM。

### 2.4 关键机制

- GPU-centric KV object store：以 KV block 为 object，GPU file 组织为 `2 x layer` 的 key/value object，保持 inference engine 的 block 粒度。
- CPU-managed metadata off critical path：全局 sharing、hash mapping、file metadata 仍由 CPU 管，但不在单次 KV retrieve/store 的关键路径上。
- P2P memory mapping table：预计算 GPU KV pool 的映射，使用 SGL 描述大块连续内存，避免 PRP page-level 元数据膨胀。
- GPU io_uring：采用 CPU-prepared/GPU-executed 模型；SQ/CQ 在 HBM，GPU I/O kernel 生成并发 SSD commands。
- SM partitioning：为 I/O control kernel 保留资源，避免非抢占式 GPU 调度导致 compute kernel 被长 I/O kernel 卡住。
- Slack-aware I/O scheduler：离线 profile 每层 slack window；优先调度 critical read，write 可以延后；避免 read/write 并发导致 SSD bandwidth collapse。

### 2.5 边界与失败条件

- 需要 GPU-centric storage 能力、P2P 映射、可控 I/O kernel 与 SSD 队列；迁移到 Ascend 时不能直接假设存在 NPU-native storage submission。
- Tutti 单机 fast path 是 HBM-SSD，CPU 不在 critical data/control path；但 cluster remote fallback 目前仍有 CPU-side read + RDMA。
- 若没有足够可预测 slack window，或者 I/O kernel 与模型 kernel 资源隔离不足，SSD I/O 仍可能阻塞 decode/prefill。
- 它解决的是 SSD-backed prefix/KV restore 的 I/O 问题，不是通用的 CPU/DRAM warm tier 调度问题。

### 2.6 前端校准点

- 点击 Tutti 时，应高亮：`HBM/KV block -> GPU object I/O/gio_uring -> PCIe/P2P -> NVMe SSD`。
- CPU 节点只用控制面颜色显示“metadata / IOCB preparation / kernel launch once per layer”，不要用温路径把 CPU 画成数据搬运中心。
- 详情栏要写明“correct Tutti PDF is arXiv 2605.03375；旧 Tutti 缓存不可作为证据”。
- Tutti 对 A+K 的迁移提示应落到 P3：需要验证 NPU-native SSD/direct path 或 NPU-side async storage submission。

## 3. Mooncake

原文：`references/papers/Mooncake__arxiv-2407.00079.pdf`

### 3.1 核心架构一句话

Mooncake 是 KVCache-centric disaggregated serving architecture：分离 prefill 与 decoding 节点，并把 GPU 集群中未充分利用的 CPU、DRAM、SSD、RDMA 资源组织成分布式 KVCache pool，再由 Conductor 按 KV 分布和负载做全局调度。

### 3.2 实际硬件链路

| 链路/组件 | Mooncake 原文中的角色 | 前端拓扑显示建议 |
|---|---|---|
| Prefill GPU/VRAM | 执行 prefill，生成 incremental KV；可加载 reusable prefix KV | 高亮 prefill 侧热路径 |
| Decode GPU/VRAM | 执行 continuous batching decode；异步加载 full KV 避免 GPU idle | 高亮 decode 侧热路径 |
| CPU DRAM KV pool | 存 paged KVCache block；prefix/incremental/full KV 的中转和复用中心 | 作为温路径和远端数据面的交汇点 |
| SSD | 作为 disaggregated KVCache 的容量资源之一，但架构图中的核心工作流以 CPU memory KV pool 和 RDMA transfer 为主 | 在前端弱化为容量层，不要默认画成每次请求的必经路径 |
| RDMA / Messenger | 每个 inference instance 有 Messenger 进程，负责跨机器高速 KVCache transfer；使用 GPUDirect RDMA 语义 | 高亮远端数据面 |
| Conductor | 全局 scheduler，选择 prefill/decoding instance，平衡 cache reuse、queue/load、TTFT/TBT SLO | 高亮控制面 |
| Hash/block metadata | 每个 KV block 用 prefix-dependent hash 做 dedup 和匹配 | 作为 state object ledger/prefix metadata 显示 |

### 3.3 解决的问题

- 长上下文和多轮对话下，KVCache reuse、transfer、placement 决定吞吐与 SLO。
- Prefill 和 decode 计算特性不同，单一 GPU pool 调度难以同时优化 TTFT 和 TBT。
- KVCache 分布不均会造成热块拥塞、远端 transfer 等待和局部负载失衡。

### 3.4 关键机制

- KVCache-centric scheduling：Conductor 选择 prefill/decoding 节点时同时考虑 prefix cache hit length、queue time、transfer time 和 SLO。
- Prefill/decode disaggregation：请求先进入 prefill pool，随后 KV 通过 Messenger 到 decode instance。
- CPU memory KV pool：KVCache 以 paged blocks 存在 CPU memory；支持 LRU/LFU 或基于请求特征的 eviction。
- Layer-wise load/store and transfer：prefill 侧按 layer 读/写 KV 并与计算重叠；生成的 incremental KV 可 layer-wise streaming 到 decode CPU memory。
- Chunked pipeline prefill：长输入按 chunk/pipeline 处理，降低多节点 TP/RDMA all-reduce 的压力。
- Hot-spot migration/replication：Conductor 可复制或迁移热 KV blocks，避免后续 transfer 拥塞。

### 3.5 边界与失败条件

- P/D 分离只有在长上下文、高复用、可重叠 KV transfer、网络不过载时才明显赚钱。
- 远端 KVCache 等待可能违反 TTFT；Conductor 需要在 cache reuse 和排队/传输时间之间取舍。
- Decode 节点最终是否接受请求还要由本地 scheduler double-check TBT SLO；预先 prefill 的成本可能浪费。
- Mooncake 强依赖集群数据面和调度信息，对单机小算力部署应提取“KV object + host pool + connector”而不是照搬完整集群。

### 3.6 前端校准点

- 点击 Mooncake 时，应高亮：`request/Conductor -> prefill GPU -> CPU DRAM KV pool -> RDMA/Messenger -> decode CPU DRAM -> decode GPU`。
- SSD 应显示为可选 capacity tier，而不是 Mooncake 每个请求的核心 fast path。
- 详情栏要区分：Conductor 是控制面；Messenger/RDMA 是远端数据面；CPU DRAM KV pool 是温路径容量层。
- A+K 迁移应映射到 vLLM-Ascend KV Pool / MooncakeConnectorV1，而不是泛泛“用 Mooncake 加速”。

## 4. Bidaw

原文：`references/papers/Bidaw-FAST26.pdf`

### 4.1 核心架构一句话

Bidaw 面向 interactive multi-round LLM serving，把历史 KV 放在 host memory + SSD 两级存储里，并让 compute engine 与 two-tier storage 双向感知：compute 侧按 KV 位置/大小调度请求，storage 侧用模型上一轮回答长度预测下次访问距离来做 eviction。

### 4.2 实际硬件链路

| 链路/组件 | Bidaw 原文中的角色 | 前端拓扑显示建议 |
|---|---|---|
| GPU | 只计算 ready queue 中 KV 已经在 performance layer 的请求；需要把 KV 装入 GPU memory 后执行 | 高亮为最终消费端，不要画成直接访问 SSD |
| Host memory performance layer | two-tier storage 的快层；命中后可较快把 KV 送入 GPU | 温路径核心节点 |
| SSD capacity layer | two-tier storage 的容量层；miss 后需要先读入 performance layer | 冷路径核心节点 |
| Compute scheduler | 读取每个请求的 KV location 和 size，分 ready/preparing queue | 控制面高亮 |
| Ready queue | KV 在 performance layer 的请求，可进入 GPU inference | 控制面/队列显示 |
| Preparing queue | KV 在 capacity layer 的请求，先触发 SSD->host memory load，再 promote | 控制面/队列显示 |
| Eviction manager | 捕获模型回答长度，估计 weighted reuse distance，决定从 host memory evict 到 SSD | 控制面与状态对象热度显示 |

### 4.3 解决的问题

- 交互式多轮对话中，历史 KV 被频繁复用，但总量超过 GPU memory 和 host memory。
- SSD capacity layer 上的大 KV 读入很慢；如果 compute scheduler 不知道 KV 位置/大小，慢请求会阻塞快请求。
- 交互式会话 KV access temporal locality 差，纯 LRU/FIFO/queue-enhanced eviction 命中率低。

### 4.4 关键机制

- Dual queues：KV 在 host memory 的请求进入 ready queue；KV 在 SSD 的请求进入 preparing queue，加载到 host memory 后再 promote。
- KV-size-based request reordering：preparing queue 使用 disk-HRRN，优先小 KV 同时用等待时间避免大 KV 饥饿。
- Arrival-time based promotion：从 preparing queue promoted 的请求按原始到达时间插入 ready queue，控制尾延迟。
- Previous-answer-based eviction：上一轮模型回答越长，用户下一次请求通常越晚；用回答长度估计 weighted reuse distance 下界，决定哪些 KV 从 host memory evict 到 SSD。
- Storage-efficient tensor caching：按单位存储开销能省多少计算选择缓存对象；对 GQA 模型更适合缓存 KV。

### 4.5 边界与失败条件

- Bidaw 依赖 interactive multi-round workload；对单轮批处理或 KV 不复用场景收益有限。
- previous-answer length 是经验预测信号，不是确定性未来访问；用户行为偏离时 eviction 可能失效。
- GPU 侧优化不是核心贡献；它的主线是 host memory/SSD 存储层与 compute scheduler 的耦合。
- SSD 带宽很差或 host memory 过小会让 preparing queue 长期堆积，P99 仍可能恶化。

### 4.6 前端校准点

- 点击 Bidaw 时，应高亮：`request scheduler -> ready/preparing queue -> CPU DDR/host memory -> SSD -> GPU`。
- 颜色上，host memory 为温路径，SSD 为冷路径，ready/preparing queue 与 eviction manager 为控制面。
- 不要把 Bidaw 展示成 GPU direct SSD 或 P/D 远端系统；它是 interactive KV caching 的 two-tier local storage scheduler。

## 5. SolidAttention

原文：`references/papers/SolidAttention-FAST26.pdf`

### 5.1 核心架构一句话

SolidAttention 面向 memory-constrained PC，把完整 KV cache 放到 SSD，只把 dynamic sparse attention 选中的关键 KV blocks 装入 GPU memory，并通过 KV consolidation、speculative prefetch 和 SSD-aware scheduler 让 SSD I/O 粒度与 sparse attention 访问模式对齐。

### 5.2 实际硬件链路

| 链路/组件 | SolidAttention 原文中的角色 | 前端拓扑显示建议 |
|---|---|---|
| SSD storage | 存完整 KV cache；decode 时按 sparse attention 选择动态加载 | 冷路径核心 |
| DRAM buffer | host-mediated transfer 过程中的中间 buffer；预取/写回过程中出现 | 温路径辅助，不是核心创新 |
| GPU VRAM | 只保留当前 attention 需要的 KV blocks、新生成 KV 和执行 attention/FFN | 热路径消费端 |
| GPU compute | 执行 block-wise sparse attention、projection、FFN | 热路径 |
| KV consolidator | token-level interleaving K/V，扩大 transfer unit，减少 I/O 次数 | 状态布局层高亮 |
| Speculative prefetcher | 利用 block selection 的 temporal locality 预取下一层关键 blocks | 控制面/预取策略显示 |
| SSD-aware scheduler | 将 attention 分解为 microtasks，与 async SSD I/O 按依赖关系重叠 | 控制面与冷路径调度 |

### 5.3 解决的问题

- 长上下文使 KV cache 远超本地 PC 内存/VRAM 容量。
- Dynamic attention sparsity 虽减少需要计算的 KV，但访问模式细碎、不规则，与 SSD 大粒度高吞吐特性冲突。
- 小 batch 本地推理难以靠请求 batching 隐藏 SSD latency。
- 盲目扩大 sparse block 会损伤 recall/accuracy；过小 block 又造成 I/O 低效。

### 5.4 关键机制

- Block-wise dynamic attention sparsity：KV blocks 分为 init blocks、local blocks、selected blocks；selected blocks 由 query 与 block representative 的 top-k 相似度决定。
- KV consolidator：K/V token-level interleaving，把原本分离的 K 与 V 访问合并为更大的 bulk transfer；还通过预拼接 K/V projection weights 直接生成 interleaved KV，避免运行时重排。
- Speculative prefetcher：根据历史 block selection 的 temporal locality 提前 SSD->GPU/DRAM 预取关键 blocks；missed blocks 再补读，错误预取可覆盖。
- SSD-aware scheduler：把 attention 计算拆成 microtasks，按数据依赖与 SSD I/O 并行，并共享同步点降低同步频率。
- Single-layer dataflow：上一层完成后，先预取本层预测 blocks；选择后补读 miss；attention 时预取下一层；FFN 时写回新 KV 并加载下一层 blocks。

### 5.5 边界与失败条件

- 适用前提是 attention sparsity 足够有效；dense attention 或 selected blocks 过多时 SSD I/O 仍会压倒计算。
- 大 block 会降低 representative 的精度并影响 long-context retrieval；小 block 又难以发挥 SSD 带宽。
- 论文目标是 memory-constrained PCs，不是数据中心 P/D 分离或 RDMA KV pool。
- 当前描述中 transfer 是 host-mediated，不应画成 Tutti 式 GPU-native SSD direct path。

### 5.6 前端校准点

- 点击 SolidAttention 时，应高亮：`SSD -> DRAM buffer -> GPU VRAM -> sparse attention compute`，同时高亮 `KV block layout / prefetch / scheduler`。
- 详情栏要明确“算法层 sparse attention 与存储层 co-design”，不要写成普通 SSD KV offload。
- 颜色上 SSD 为冷路径，DRAM/VRAM load 为温路径，GPU attention 为热路径，prefetch/scheduler 为控制面。

## 6. FlexInfer

原文：`references/papers/FlexInfer.pdf`

### 6.1 核心架构一句话

FlexInfer 不是固定的 CPU/GPU offload 路径，而是一个阶段级 execution policy selector：针对 prefill 和 decode 分别在 CPU-only、GPU with tensor offloading、CPU-GPU static partitioning/SplitGen 之间选择低延迟策略。

### 6.2 实际硬件链路

| 链路/组件 | FlexInfer 原文中的角色 | 前端拓扑显示建议 |
|---|---|---|
| CPU memory | 存模型权重、KV cache、activation；CPU-only 和 GPU offload 都依赖其容量 | 作为 warm/capacity tier |
| CPU compute / matrix unit | CPU-only 路径使用 CPU 执行全部模型；AMX/TMUL 等矩阵单元影响策略选择 | 高亮 CPU compute，但标注依赖 CPU 实测吞吐 |
| GPU memory | SplitGen 中常驻最后 M 层及其 KV；GPU offload 中作为逐层计算端 | 高亮热路径或常驻层 |
| PCIe interconnect | GPU offload 逐层搬 weights/KV；SplitGen prefill 搬最后 M 层并传 CPU 输出到 GPU | 高亮温路径瓶颈 |
| Execution Planner | 用模型结构、输入/输出长度、batch、CPU/GPU throughput、memory bandwidth、interconnect bandwidth 估计 TTFT/TPOT | 放在控制面 |
| Inference Executor | 按选定策略执行，并管理 CPU/GPU 同步和数据传输 | 放在控制面与温路径之间 |

### 6.3 解决的问题

- 传统 offloading 把权重/KV 放 CPU memory，再经 PCIe 逐层搬到 GPU；在 A100/H100 这类强 GPU 上，暴露 PCIe load 时间仍可能支配端到端延迟。
- CPU 在 decode 阶段可能因数据都在本地内存而更有利，但在长输入、大 batch 的 prefill 阶段容易被 compute throughput 限制。
- 单一 execution policy 无法同时适配 prefill/decode、batch、sequence length 和硬件配置。

### 6.4 关键机制

- 三类策略：CPU-only、GPU with tensor offloading、SplitGen。
- SplitGen：前 `N-M` 层放 CPU 执行，最后 `M` 层放 GPU；`M` 由 GPU memory capacity 决定。decode 阶段最后 `M` 层和对应 KV 保持在 GPU memory，避免每轮重复传输。
- Execution Planner：分析每层 compute/data amount、CPU/GPU memory bandwidth、interconnect bandwidth，并加入 offline profiling 得到的 efficiency factor，分别估计 TTFT/TPOT。
- Inference Executor：执行 planner 选出的 prefill/decode 组合策略，重叠数据传输与计算，并负责 CPU-GPU synchronization。

### 6.5 边界与失败条件

- FlexInfer 的结论高度依赖 CPU/GPU throughput、memory bandwidth、PCIe 代际和模型 shape；不能把某个策略写死为最佳。
- CPU-only 更偏向 memory-bound decode 或小 batch/短输入；长输入 prefill 更可能被 CPU compute 限制。
- SplitGen 需要 GPU memory 能常驻最后 M 层；如果 M 太小或 CPU/GPU 同步过重，收益下降。
- 对 A+K 来说，Kunpeng SVE/SME 吞吐不能从 Intel AMX 结果直接外推。

### 6.6 前端校准点

- 点击 FlexInfer 时，应显示“策略选择器”而不是唯一硬件路径。
- 默认高亮：`Execution Planner -> CPU memory/CPU compute <-> PCIe -> GPU memory/GPU compute`。
- 详情栏应按 prefill/decode 展示可能不同的路径，并提示“策略取决于实测硬件 profile”。

## 7. KTransformers

原文：`references/papers/KTransformers.pdf`

### 7.1 核心架构一句话

KTransformers 面向大 MoE 本地/小资源部署，把 shared/hot experts 保留在 GPU，routed experts 放在 CPU 上直接计算，并通过 AMX/AVX-512 kernel、NUMA-aware tensor parallelism、CUDA Graph 式异步调度和 Expert Deferral 提升 CPU-GPU overlap。

### 7.2 实际硬件链路

| 链路/组件 | KTransformers 原文中的角色 | 前端拓扑显示建议 |
|---|---|---|
| GPU VRAM | 放 attention、shared experts、部分高优先级模块；执行 FlashInfer attention 和 GPU kernels | 热路径主算 |
| CPU DRAM | 常驻 routed expert weights，避免频繁 PCIe weight transfer | warm expert tier |
| CPU AMX / AVX-512 | routed expert compute；AMX 用于高 arithmetic intensity，AVX-512 用于低 ARI decode | CPU compute 节点高亮 |
| NUMA sockets | expert weights 切分到 socket-local memory，避免 cross-socket remote access | CPU memory 内部子层高亮 |
| CPU-GPU synchronization | gating 后 CPU 执行 routed experts，GPU 执行 shared experts；用异步 task queue 和 CUDA callback/graph 降低 host interruption | 控制面和温路径 |
| Expert Deferral | 推迟部分 routed expert 输出到后续层消费，以打破 MoE 与 attention 的硬依赖 | 算法/调度层显示 |

### 7.3 解决的问题

- 大 MoE 总参数量远超单 GPU VRAM；把 expert weights 按需搬到 GPU 会被 PCIe 卡住。
- CPU 直接跑 routed experts 后，瓶颈转为 CPU memory bandwidth、AMX/AVX kernel 效率、CPU-GPU 同步和 NUMA remote access。
- token-by-token decode 中 kernel launch 和同步开销会吞掉 GPU 时间，CPU/GPU overlap 不充分。

### 7.4 关键机制

- Expert placement：shared experts 或高热 experts 放 GPU；routed experts 放 CPU。
- Arithmetic-intensity-aware CPU kernel：AMX tiling-aware layout 和 cache-friendly AMX kernel 处理高 ARI；低 ARI decode 用更轻的 AVX-512 kernel。
- Fused MoE operator：合并 Gate/Up/Down 等 MoE 操作，减少同步和调度开销。
- Async CPU-GPU task scheduling：GPU gating 后，CPU control thread 把 routed-expert tasks 推入 lock-free queue，同时启动 GPU shared expert kernels；CPU workers 与 GPU 并行。
- CUDA Graph / host callback：把 submit/sync 封装到 GPU stream 回调中，减少 host interruption。
- NUMA-aware tensor parallelism：每个 expert weight 在 socket 间 column/row-wise 切分，各 socket 本地计算，再 reduce-scatter 合并。
- Expert Deferral：将部分 routed experts 的输出推迟到后续层消费，提升 CPU/GPU 并行度，但最后层不应用 deferral。

### 7.5 边界与失败条件

- 原文强依赖 Intel AMX/AVX-512 与 NVIDIA CUDA Graph/FlashInfer；迁移 Kunpeng+Ascend 需要重新证明 SVE/SME kernel、CANN stream/callback 能力。
- Expert Deferral 是模型结构级改动，必须保留精度评估，不应被前端简化成“纯系统调度”。
- NUMA-aware placement 是核心机制之一；这篇论文中 NUMA 应该高亮，不同于 NEO。
- 对非 MoE dense model 不应套用该路径。

### 7.6 前端校准点

- 点击 KTransformers 时，应高亮：`GPU VRAM/shared experts -> CPU DRAM/routed experts -> CPU AMX/AVX compute -> NUMA placement -> CPU-GPU scheduler`。
- 颜色上 GPU shared expert 为热路径，CPU expert compute/DRAM 为温路径，NUMA 与 task scheduler 为控制面/硬件边界。
- A+K 详情栏必须写“AMX 结论不能直接继承到 Kunpeng，需 SVE/SME microbenchmark”。

## 8. LMCache

原文：`references/papers/LMCache__arxiv-2510.09665.pdf`

### 8.1 核心架构一句话

LMCache 是位于 inference engine 与 heterogeneous storage/network devices 之间的 KV cache layer：它给 KV cache 提供 store/retrieve/lookup/management 接口，并优化 GPU、CPU memory、disk、remote memory/disk、object storage、RDMA/TCP 等后端之间的数据搬运。

### 8.2 实际硬件链路

| 链路/组件 | LMCache 原文中的角色 | 前端拓扑显示建议 |
|---|---|---|
| Inference engine / KV connector | 准备 tokenized prompt、GPU memory addresses、scheduler metadata；解耦 vLLM/SGLang API 变化 | 控制面 |
| GPU connector / streaming GPU buffer | 把 scattered paged KV 与连续 chunk/buffer 之间转换；load/store 时进出 GPU memory | 温路径数据搬运 |
| CPU DRAM | local backend / warm tier；也可承接动态 offloading | 温路径容量层 |
| Local disk / SSD | local lower-tier backend | 冷路径 |
| Remote DRAM / remote disk / object storage | distributed backend；用于跨 engine/GPU reuse 和 PD disaggregation | 远端数据面/冷路径 |
| RDMA/TCP/NVLink/P2P | 多后端 transfer channel | 远端或机内互联 |
| Cache controller / token pool | 保存 backend 中 token/KV 状态，供 router 做 KV-aware routing | 控制面与 state object ledger |

### 8.3 解决的问题

- GPU memory 已不足以保存不断增长的 KV cache，但跨 query reuse 和 PD disaggregation 都要求 KV 能离开 GPU 并被再加载。
- 现代 inference engine 的 paged KV 单位很小，直接 page-level transfer 无法吃满带宽。
- KV transfer 与推理计算并发时，数据搬运可能 stall compute；大量 CUDA memcpy/launch 也会带来 CPU overhead。
- 上层 scheduler/router 缺少统一的 locate/move/pin/compress/evict 接口，难以做 KV-aware routing。

### 8.4 关键机制

- Store/Retrieve/Lookup workflow：KV connector 准备 metadata；token processor 判断新 token 或 prefix match；storage manager 与 transfer channel 处理后端读写；cache controller/token pool 给上层查询 KV 是否存在。
- Configurable chunk size：把多个 paged KV 合并为更大 chunk，默认按 256 tokens 级别搬运，而不是 native page 粒度。
- Streaming GPU buffer：store 时从 scattered GPU paged memory 拷入连续 buffer 再 DMA 到 lower tier；load 时先取回 chunk 再用 CUDA kernel 拆回 paged layout。
- Parallel store/load：允许多个 source/destination 后端并发传输，如 GPU->CPU 的 hot context 与 CPU->disk 的 cold context 同时进行。
- Delayed decode KV storing：decode 中新生成 KV 不逐 token 立即写出，而是积累成 chunk 后批量存储。
- Layer-wise pipelining：用独立 CUDA streams 将当前层计算与下一层 KV load/store 重叠。
- Zero-copy / dynamic offloading：减少跨层级重复拷贝，并在 GPU free pages 上按指针推进做动态 offload。

### 8.5 边界与失败条件

- LMCache 是 cache layer/API，不是一种固定硬件拓扑；收益取决于后端延迟、chunk size、prefix hit rate、router 策略和 engine 集成。
- 它仍可能遇到 Tutti 指出的 SSD tiny I/O、CPU overhead 和 GDS control path 问题；Tutti 可视为针对 SSD-backed KV 的更底层 fast path 改造。
- 远端 backend 带来的网络延迟并不总能降低 prefill delay，原文也把企业部署中的经验教训作为重点。
- 对 Ascend/UCM 应映射为 external KV / connector / storage backend 接口，而不是直接套 CUDA stream 实现细节。

### 8.6 前端校准点

- 点击 LMCache/UCM 时，应高亮：`engine connector -> GPU buffer/HBM -> CPU DRAM -> SSD/remote backend -> cache controller/token pool`。
- 右侧详情栏应把它归为“状态对象 API + 多级数据搬运层”，不要和 Tutti 的 GPU-native SSD direct path 混同。
- 颜色上 connector/controller 为控制面，GPU/CPU 数据移动为温路径，SSD 为冷路径，RDMA/TCP/P2P 为远端或机内互联。

## 9. ServeGen

原文：`references/papers/ServeGen-NSDI26.pdf`

### 9.1 核心架构一句话

ServeGen 不改服务器硬件路径；它提供 production-scale LLM serving workload characterization，并用 per-client generation 生成更真实的 arrival、input/output length、multimodal/reasoning、conversation pattern，用来驱动 serving 系统评估和仿真。

### 9.2 实际硬件链路

| 链路/组件 | ServeGen 原文中的角色 | 前端拓扑显示建议 |
|---|---|---|
| Request arrival | 真实 workload 中短期到达常 bursty，且没有单一随机过程总是最佳 | L0 workload 输入 |
| Input/output length | 输入可用 Pareto+Log-normal 混合建模，输出可用 Exponential 建模；分布随时间变化 | L0 workload 字段 |
| Client decomposition | 少数 top clients 解释大部分 rate shift、burstiness 和 length shift | workload generator 控制面 |
| Multimodal/reasoning workload | 模态 load、reasoning length、多轮会话行为显著改变 TTFT/TBT 压力 | workload scenario 标签 |
| ServeGen framework | Client Generator、Timestamp Sampler、Request Data Sampler 聚合生成 workload | 仿真器输入，不画成硬件节点 |

### 9.3 解决的问题

- 以 Poisson/Gamma arrival + ShareGPT length 的 naive workload 可能比真实 workload 更容易 serve，导致 provisioning 和 PD-disaggregation 评估偏乐观。
- 真实 workload 的 rate、burstiness、input/output length 会随 top clients 和时间窗口变化。
- Multimodal 与 reasoning workload 的负载构成不再只由文本 token 决定。

### 9.4 关键机制

- Per-client generation：按 client 采样 trace 与 dataset，再聚合成整体 workload。
- Client Pool：可使用参数化真实 client behavior 或用户指定 client。
- Timestamp Sampler / Request Data Sampler：为每个 client 生成 request timestamps 和 request data。
- Conversation-aware mocking：保留 shared conversation histories，避免把多轮上下文打散。
- Use cases：用于 instance provisioning 和 PD-disaggregation 的工作负载敏感性评估。

### 9.5 边界与失败条件

- ServeGen 不提供 operator/hardware latency 模型；它必须接入 vLLM、SGLang 或仿真器才能转化为硬件结论。
- 它不能证明某条硬件 offload 路径有效，只能证明评估 workload 是否足够真实。
- 插件调用、外部 API、复杂 agent workflow 在原文限制中仍待进一步刻画。

### 9.6 前端校准点

- ServeGen 应放在“请求/会话”和“Telemetry/仿真器”周边，不应高亮 GPU/CPU/SSD 主板链路。
- 详情栏应展示可进入仿真器的字段：arrival process、client_id、prompt_tokens、output_tokens、modality、reasoning length、conversation/session id、SLO。

## 10. ProfInfer

原文：`references/papers/ProfInfer__arxiv-2601.20755.pdf`

### 10.1 核心架构一句话

ProfInfer 是面向 LLM inference engine 的非侵入式 eBPF fine-grained profiler：通过 user/kernel probes 采集 token、graph、operator、scheduler、PMC 等层面的 trace，把端到端延迟拆成可诊断的 timeline、DAG、operator stats 和 hardware counter。

### 10.2 实际硬件链路

| 链路/组件 | ProfInfer 原文中的角色 | 前端拓扑显示建议 |
|---|---|---|
| llama.cpp / GGML runtime | 被观测的 inference engine；支持 CPU、GPU/OpenCL、Rockchip NPU backend | 观测对象 |
| eBPF user space | 编译/加载/配置 BPF 程序，attach probes，poll ring/perf buffer | 观测控制面 |
| eBPF kernel space | probe handlers 读取 PID、thread id、CPU id、timestamps、tensor/operator metadata | 观测数据面 |
| Token tracing | probe `llama_decode` 获得 prefill TTFT 和 decode TPOT | L11 指标 |
| Graph tracing | probe graph compute async，观察 backend partition 与 graph execution time | L11 timeline |
| Operator tracing | probe CPU/GPU/NPU backend compute forward，采集 op type、tensor dims、backend time | operator timeline |
| Scheduler tracing | `sched_switch` / `sched_wakeup` 观察 inference threads 和干扰任务 | CPU scheduling 观测 |
| PMC counters | cache refill、memory write、major faults、cycles、backend stalls 等 | hardware counter |

### 10.3 解决的问题

- 现有 LLM inference profiler 常只能给端到端 token rate，缺乏 operator-level 和 hardware-counter 可见性。
- 移动/端侧/小算力部署中，经常不知道瓶颈是 compute、memory bandwidth、KV reuse、backend offload 还是 thread scheduling。
- 修改或重编译 inference engine 成本高，难以在生产/边缘环境做低侵入诊断。

### 10.4 关键机制

- Multi-granularity tracing：token-level、graph-level、operator-level、scheduler-level。
- Dynamic overhead control：根据 QoS 可以关掉部分 probe，降低 profiling overhead。
- PMC integration：在 operator 级读取硬件计数器，推导 DRAM 访问、cache miss、stall 等。
- MoE expert tracing：读取 activated expert IDs，用于分析 expert 不在 DRAM 需要从 storage fetch 时的影响。
- Views：ProfDAG 展示 operator topology 和 metrics；ProfTime 输出 Chrome Trace/Perfetto timeline；ProfStat 支持 operator/backend/expert 统计。

### 10.5 边界与失败条件

- 原文实现以 llama.cpp/GGML、Linux/eBPF、CPU/OpenCL/Rockchip NPU 为主；迁移到 Ascend/CANN 需要替换 probe 点和 counter 来源。
- eBPF 适合 CPU/runtime/thread 可见性；NPU 内部 stream、HBM、copy engine counter 仍需 CANN/Ascend profiler 对齐。
- ProfInfer 是观测系统，不直接提供调度策略或 offload 机制。

### 10.6 前端校准点

- 点击 ProfInfer 时，应高亮 `Telemetry/仿真器 -> Runtime -> CPU/GPU/NPU backend`，用灰色观测/仿真路径，不要高亮 SSD/RDMA 业务数据面。
- 详情栏应展示它能产出的字段：TTFT、TPOT、operator latency、backend type、thread id、CPU id、PMC、scheduler event、expert id、trace timestamp。

## 11. LLMServingSim2.0

原文：`references/papers/LLMServingSim-2.0__arxiv-2602.23036.pdf`

### 11.1 核心架构一句话

LLMServingSim2.0 是统一的系统级仿真器，把 workload、cluster config、operator hardware profile 放进同一个 runtime-driven loop，联合模拟 heterogeneous devices、multi-tier memory、PD disaggregation、prefix caching、MoE/expert offload、KV movement、network contention 和 power。

### 11.2 实际硬件链路

| 链路/组件 | LLMServingSim2.0 原文中的角色 | 前端拓扑显示建议 |
|---|---|---|
| Workload config | LLM 和 request pattern，如 arrival rate、per-request traces | L0 输入 |
| Cluster config | node count/type、CPU settings、memory capacity/bandwidth、device placement、routing/offload/cache policy | 仿真配置面 |
| Hardware profiles | operator-level latency/power profile；可来自真实硬件或外部硬件模拟器 | L11 profile 输入 |
| Model Serving Group | 每个模型的逻辑 serving unit；含 device pool、request queue、batch scheduler、operation mapper/scheduler | 仿真核心模块 |
| Memory model | device memory、host memory、storage、CXL-attached memory；模拟 KV/prefix placement、eviction、promotion、load/store cost | 多级内存模型 |
| System simulator | 执行 operator graph，计入 communication、synchronization、network contention、inter-device movement | 硬件链路模拟 |
| Power model | accelerators、CPUs、DRAM、interconnect、NIC、storage、motherboard/cooling 等七类能耗 | 能耗/规格反推 |

### 11.3 解决的问题

- 静态 operator/hardware 模型无法描述 serving-time request arrival、batch composition、KV locality、routing imbalance 和 policy feedback。
- 现有仿真器对异构硬件、PD disaggregation、prefix caching、MoE/expert offloading、multi-tier memory 和 power 的联合建模不足。
- 早期硬件/系统设计需要在真实大规模部署前评估调度、数据搬运和互联瓶颈。

### 11.4 关键机制

- Runtime-driven loop：Request Router -> MSG -> execution graph -> System Simulator，循环直到请求完成，并输出 TTFT/TPOT/queueing/end-to-end latency。
- Profile-based operator modeling：一次 profiling 采集 model-device pair 的 operator latency/power，可复用；也可接外部硬件 simulator profile。
- Batch scheduler + memory model：batch selection 时检查 device memory、KV footprint、prefix residency、eviction cost。
- Operation mapper/scheduler：按 parallelism、offload rules、heterogeneous device placement 生成带通信/同步/内存操作的 execution DAG。
- Multi-tier prefix/KV：显式建模 device memory、host memory、CXL memory、storage 中的 prefix cache；miss 时插入 layer-wise KV transfer。
- MoE modeling：支持 expert routing、expert placement/offload、expert loading、all-to-all communication。
- Power-aware modeling：DRAM/interconnect 与 data movement volume 关联，accelerator 用 idle/active/standby 状态模型。

### 11.5 边界与失败条件

- 仿真可信度依赖 operator profile、memory/interconnect profile 和 workload 输入；没有 A+K 本地 profile 就不能直接输出规格结论。
- 它是系统级模拟，不替代 cycle-level NPU/CPU microarchitecture simulator。
- 对 Ascend/Kunpeng 需要新增 CANN operator profile、NPU stream/copy profile、Kunpeng DDR/NUMA/SVE profile、HCCS/HCCL/UB/fabric memory profile。

### 11.6 前端校准点

- 点击 LLMServingSim2.0 时，应高亮 `request/workload -> telemetry/simulator -> accelerator/CPU/memory/network/storage/power model`，用灰色观测/仿真路径。
- 详情栏应把它作为“规格反推/瓶颈验证底座”，不要写成具体 offload 机制。
- 可直接映射到 A+K 仿真字段：workload、model、accelerator、CPU、interconnect、storage、state object、scheduler、telemetry。

## 12. llm.npu

原文：`references/papers/llm.npu__arxiv-2407.05858.pdf`

### 12.1 核心架构一句话

llm.npu 面向端侧 LLM prefill，把规则的大量 INT8 矩阵计算尽量放到移动 NPU，同时把 NPU 不擅长的 outlier/float/动态子图路径交给 CPU/GPU，并通过 chunk-sharing graph 和 out-of-order subgraph execution 避免动态 prompt 与 NPU 静态图编译成本吞掉收益。

### 12.2 实际硬件链路

| 链路/组件 | llm.npu 原文中的角色 | 前端拓扑显示建议 |
|---|---|---|
| Mobile NPU | 主跑 prefill 中适合 INT8 的固定 shape 大矩阵和可共享静态子图 | 热路径，但标注“端侧 NPU” |
| CPU/GPU | 执行 activation outlier shadow path、float operations、部分子图 fallback | 温路径/irregular path |
| Shared SoC memory | 移动 NPU 与 CPU/GPU 共享物理内存，避免独立 GPU 式显存拷贝 | 不直接外推到服务器 PCIe 架构 |
| NPU graph compiler/runtime | 静态 shape 图构建、优化、执行；动态 prompt 会导致 costly preparation | 控制面/编译路径 |
| Disk | shadow outlier 的部分权重按需从 disk 取回 | 冷路径边界项 |
| OOO scheduler | 根据硬件 affinity 和 accuracy sensitivity 排布 CPU/GPU/NPU subgraphs | 控制面 |

### 12.3 解决的问题

- 端侧任务的 prefill 常占主要端到端延迟，移动 CPU/GPU 并行能力有限。
- 移动 NPU 擅长 INT8 MatMul，但通常不支持 per-group quantization MatMul、动态 shape 和高效 FP 操作。
- 每种 prompt length 都重建/优化 NPU 图成本极高，padding 到最大长度又浪费计算。
- outlier activation 对精度重要，但直接在 NPU 上处理会导致量化/算子不匹配。

### 12.4 关键机制

- Chunk-sharing graph：把 variable-length prompt 拆成固定长度 chunks；共享与 prompt size 无关的静态 operators，如 Linear/LayerNorm/FFN；只为 Attention 等动态 operators 保留 chunk-specific 子图。
- Shadow outlier execution：提取少量重要 activation outlier channels，在 CPU/GPU 上并行执行补偿路径；NPU 继续跑原本的 dense INT8 path。
- Out-of-order subgraph execution：基于数据依赖和硬件亲和度，允许不同 chunks/subgraphs 不按原始顺序执行，以减少 CPU/GPU float path 造成的 NPU stall。
- Offline profiling：选择 chunk length、outlier threshold 和子图调度优先级。

### 12.5 边界与失败条件

- 原文是移动 SoC/NPU，不是服务器 Ascend NPU；共享内存、NPU SDK、图编译行为不能直接外推到 PCIe/UB/HCCS 服务器。
- 它主要优化 prefill，不是 decode KV 分层或 SSD/remote KV 路线。
- 迁移 A+K 时要重新验证 CANN 静态/动态图代价、Ascend INT8/FP 算子覆盖、Kunpeng 处理 outlier 的同步成本。
- 如果 outlier path 或 subgraph synchronization 无法被 NPU 主路径覆盖，收益会下降。

### 12.6 前端校准点

- 点击 llm.npu 时，应高亮：`NPU main prefill -> CPU/GPU outlier shadow path -> graph/runtime scheduler`。
- 不应高亮服务器 PCIe/SSD/RDMA 链路；如果放在服务器拓扑图里，应作为“端侧 NPU 异构调度对照”显示。
- 详情栏应写“方法论可迁移，硬件结论需重新 profile”。

## 13. FineMoE

原文：`references/papers/fMoE-FineMoE__arxiv-2502.05370.pdf`

### 13.1 核心架构一句话

FineMoE 把 MoE expert offloading 从粗粒度 request/layer 预测推进到 iteration-level expert map：结合历史 expert trajectory 与 prompt semantic embeddings，指导 GPU expert cache 的 prefetch、cache 和 offload。

### 13.2 实际硬件链路

| 链路/组件 | FineMoE 原文中的角色 | 前端拓扑显示建议 |
|---|---|---|
| GPU memory / expert cache | 存当前预测要命中的专家；hit 时直接服务 MoE layer | 热 expert tier |
| CPU memory | 存被 offload 的专家权重；miss 时从 CPU load 到 GPU | warm expert tier |
| PCIe / CPU-GPU transfer | expert prefetch 和 on-demand loading 的数据路径 | 温路径瓶颈 |
| Expert Map Store | 保存历史 request prompt 的 iteration-level expert activation patterns | 状态对象 metadata |
| Expert Map Searcher | 按 semantic similarity 和 trajectory similarity 找近似 expert map | 控制面 |
| Gate network | 运行时仍决定真实 expert；FineMoE 命中则复用已在 GPU 的 expert | 热路径入口 |

### 13.3 解决的问题

- MoE 推理中大量专家不活跃却占据 GPU memory；粗粒度 offload 要么 latency 高，要么 memory footprint 仍大。
- 现代 decoder-only MoE 有 load-balancing loss，coarse-grained expert usage 高熵、难预测。
- 预取预测错误会造成 expert miss 和 on-demand load 延迟。

### 13.4 关键机制

- Expert map：记录 iteration-level expert activation patterns，比 request-level/coarse heatmap 更细。
- Semantic hints：从 input prompt 的 semantic embedding 里提取上下文信号，辅助选择历史 expert map。
- Similarity search：结合 semantic similarity 与 trajectory similarity，为当前 request 找最合适 expert map。
- Guided prefetch/offload：用 expert map 动态计算 expert selection threshold，提前从 CPU memory 预取专家到 GPU；超过 expert cache capacity 时把低优先级专家 offload 回 CPU。
- Online update：推理结束后更新 Expert Map Store，持续积累新 trajectory。

### 13.5 边界与失败条件

- FineMoE 依赖 prompt semantic 与 expert trajectory 的可预测性；语义相近但 routing 差异大时会 miss。
- 它的硬件数据面主要是 CPU memory <-> GPU memory，不是 SSD/remote expert pool。
- 系统需要暴露 session/prompt semantic/expert activation trace；否则 expert map 难以在线维护。
- 对 A+K 迁移时，需要 Ascend runtime 暴露 MoE gate/expert IDs 和可控 expert prefetch API。

### 13.6 前端校准点

- 点击 FineMoE 时，应高亮：`prompt semantic/trajectory -> Expert Map -> CPU DRAM experts -> GPU expert cache -> MoE gate`。
- 颜色上 expert map/searcher 为控制面，CPU->GPU expert movement 为温路径，GPU expert cache 为热路径。
- 详情栏应强调“fine-grained expert hotness/next-use”，不要写成普通 MoE offload。

## 14. DALI

原文：`references/papers/DALI__arxiv-2602.03495.pdf`

### 14.1 核心架构一句话

DALI 面向本地 PC 上的 MoE 推理，把 activated experts 按当前 workload 动态分配给 CPU 或 GPU，并用 residual-based prefetch 和 workload-aware cache replacement 提高高负载专家在 GPU cache 中的命中率。

### 14.2 实际硬件链路

| 链路/组件 | DALI 原文中的角色 | 前端拓扑显示建议 |
|---|---|---|
| CPU DRAM | 部署时存所有 expert weights | warm/capacity expert tier |
| GPU memory / expert cache | 每层缓存固定数量专家；高负载专家命中后在 GPU 执行 | 热 expert tier |
| CPU cores | 执行低 workload experts | 温路径计算 |
| GPU cores | 执行高 workload experts | 热路径计算 |
| PCIe | GPU-assigned expert 若未缓存，需要从 DRAM 经 PCIe load 到 GPU | 温路径瓶颈 |
| Greedy Assignment | 按当前 MoE layer 的 expert activation workload 分配 CPU/GPU，目标是平衡 `Tcpu` 与 `Tgpu` | 控制面 |
| Residual-Based Prefetching | 利用相邻层 residual 信息预测下一 MoE layer 的高 workload experts | 控制面/预取 |
| Workload-Aware Cache Replacement | 利用相邻 token 高 workload expert 的 temporal locality 更新 expert cache | 控制面 |

### 14.3 解决的问题

- 静态 expert assignment 会造成 CPU/GPU load imbalance。
- 高 workload expert 若预测不准，PCIe transfer 会成为主要开销。
- 传统 LRU 或 score-based replacement 没有针对“高 workload experts”优化，GPU expert cache hit rate 低。

### 14.4 关键机制

- Greedy Assignment：把 expert assignment 建模为 0-1 optimization，最小化 `max(Tgpu, Tcpu)`；`Tgpu` 包含 PCIe transfer 与 GPU compute 的 pipeline 时间；再用贪心近似降低在线求解成本。
- Residual-Based Prefetching：利用相邻 MoE layers 的 residual information 预测下一层高负载 experts。
- Workload-Aware Cache Replacement：基于相邻 tokens 的 high-workload expert temporal locality 更新 cache。
- Iterative process：每层先按 workload 分配，GPU miss 时从 DRAM load，独立 work stream 预取下一层，再更新 cache。

### 14.5 边界与失败条件

- DALI 针对本地 PC CPU+GPU，不是数据中心 expert parallelism；硬件假设通常是 PCIe + 有限 GPU memory。
- 如果 expert workload temporal locality 弱，residual prefetch/cache replacement 会失效。
- 若 CPU expert kernel 太弱或 PCIe 太慢，greedy assignment 可能只能把更多专家放 GPU，受 GPU cache 容量限制。
- 对 A+K 迁移需要重测 Kunpeng expert compute、Ascend expert load、CANN stream overlap。

### 14.6 前端校准点

- 点击 DALI 时，应高亮：`CPU DRAM all experts -> Greedy Assignment -> CPU cores low-workload experts / GPU cache high-workload experts -> PCIe prefetch`。
- 与 KTransformers 区分：KTransformers 强调 AMX kernel/NUMA/Expert Deferral，DALI 强调 workload-aware assignment/prefetch/cache。

## 15. FluxMoE

原文：`references/papers/FluxMoE__arxiv-2604.02715v2.pdf`

证据修正：旧 `references/papers/FluxMoE__arxiv-2601.07343.pdf` 不可引用；arXiv `2601.07343` 实际标题为 “Mud-Standoff Effect Correction Based on Open-Short Calibration and Resistivity Consistency-Constrained Iterative Inversion for Oil-Based Mud Imagers”。本轮已用官方 arXiv API 核验到正确条目 `2604.02715v2` 并补抓 PDF。

### 15.1 核心架构一句话

FluxMoE 把 MoE expert weights 从“整场 serving session 常驻 GPU memory 的模型参数”改成“按 layer 执行临时 materialize 的 streamed resources”，通过 expert paging、PagedTensor、expert storage hierarchy 和 budget-aware residency planner，把 GPU memory 优先留给 KV cache 与 activation。

### 15.2 实际硬件链路

| 链路/组件 | FluxMoE 原文中的角色 | 前端拓扑显示建议 |
|---|---|---|
| GPU virtual address space | 每个 expert tensor 有稳定 tensor handle/virtual address，kernel 看起来像普通 resident tensor | 状态布局/虚拟化层 |
| GPU physical tensor buffers | 只为当前层和下一层 experts 预分配 sliding window buffers | 热 working set |
| Compressed GPU backend | 在 GPU memory 中以压缩形式保存部分 expert 参数，materialize 时 GPU 解压 | 热/温之间的高带宽 backend |
| Host DRAM backend | 保存 offloaded expert 参数，作为容量层；经 PCIe transfer 到 GPU | warm capacity tier |
| KV cache / activation buffers | FluxMoE 明确优先保留的 performance-critical runtime state | 热路径优先资源 |
| Forward proxy / PagedTensor synchronization | 在 layer 执行前 materialize 当前层 experts，同时 evict 旧层并 prefetch 下一层 | 控制面 |
| Budget-aware residency planner | 在固定 GPU memory budget 下，按 KV pressure 动态调整 expert residency | 控制面 |

### 15.3 解决的问题

- 大 MoE 的绝大多数 expert weights 在任一 decode step 都是 idle，却长期占用 GPU memory。
- KV cache capacity 直接决定 batch/context 和 throughput，和常驻 expert weights 争 GPU memory。
- 传统 vLLM 把 expert weights 当 persistent GPU-resident state，限制了 memory-intensive decode workload。

### 15.4 关键机制

- Expert paging：专家只在其 layer 被调度执行前 materialize 到 GPU，执行后立即 evict，保持两层 sliding residency window。
- PagedTensor：为 expert weights 提供稳定虚拟地址和可重用物理 tensor buffers，避免改现有 GPU kernels。
- Three-tier tensor abstraction：tensor handle、tensor buffer、tensor storage 分离逻辑身份、物理 GPU residency 和后端存储。
- Expert storage hierarchy：把 expert 参数分布到 compressed GPU memory 与 host DRAM；按后端带宽比例划分参数，以保持 materialization pipeline 满载。
- Budget-aware residency planner：在 KV cache pressure 低时保留更多专家，pressure 高时缩小 expert residency，把 memory 还给 KV/activation。

### 15.5 边界与失败条件

- FluxMoE 主要优化 memory-constrained decode，不以 TTFT/prefill 为主。
- 它依赖稳定虚拟地址、GPU memory mapping、CUDA event barriers、vLLM 集成等能力；Ascend 迁移要验证 CANN 是否允许类似 PagedTensor/虚拟化。
- Host DRAM backend 经 PCIe 供给专家，若 PCIe 带宽不足且 compressed GPU backend 占比过低，materialization 可能挡住 forward pass。
- FluxMoE 不解决 expert selection 预测问题；它默认 layer experts 按调度 materialize，重点是 residency 与 memory budget。

### 15.6 前端校准点

- 点击 FluxMoE 时，应高亮：`KV cache priority in HBM -> PagedTensor virtual handles -> compressed GPU expert backend -> host DRAM expert backend -> PCIe materialization`。
- 它应与 FineMoE/DALI 区分：FineMoE/DALI 关注“哪些 expert 预取/缓存”，FluxMoE 关注“expert 是否必须常驻 GPU 以及如何分页驻留”。
- 详情栏必须显示 FluxMoE 正确证据路径，并标注旧 arXiv ID 错误。

## 16. CacheSlide

原文：`references/papers/CacheSlide.pdf`

### 16.1 核心架构一句话

CacheSlide 面向 agent workflow 中“相对顺序不变、绝对位置漂移”的 prompt segment，提出 Relative-Position-Dependent Caching，通过 CCPE、Weighted Correction Attention 和 SLIDE manager 复用跨位置 KV，同时减少 SSD spill 和 load/write 串行造成的 prefill stall。

### 16.2 实际硬件链路

| 链路/组件 | CacheSlide 原文中的角色 | 前端拓扑显示建议 |
|---|---|---|
| KV cache / hash map | reused chunks 通过 content hash 查找 KV cache | 状态对象 metadata |
| Position encoding / CCPE | 给 reused chunks 分配稳定的低位置敏感 encoding range，降低 PMKD | 算法/状态布局层 |
| Weighted Correction Attention | 只重算少量 tokens 并和 cached KVs 加权融合，恢复 cross-attention | 热路径上的轻量校正 |
| SLIDE KV manager | 增加 KV load/update 并行，relocate/overwrite 选中 KVs，dirty-page eviction | 控制面 + SSD spill 管理 |
| SSD | memory pressure 下 spill clean pages，dirty/selected pages 特殊处理，减少小随机写 | 冷路径边界项 |
| vLLM KV management | 原文实现扩展 vLLM KV cache 管理 | 前端可归入 runtime connector |

### 16.3 解决的问题

- Agent prompts 中固定 segment 经常相对顺序不变但绝对位置变动，传统 PDC 只能复用固定 prefix，PIC 又有 positional misalignment。
- RoPE 等位置编码对 absolute shift 敏感，会造成 Positionally Misaligned KV Drift。
- PIC/PDC 系统层可能序列化 KV load/write，SSD spill 时产生小随机写，阻塞 prefill critical path。

### 16.4 关键机制

- RPDC：定义跨位置但相对顺序保持的 KV reuse 场景，区别于 PDC/PIC。
- CCPE：按 prompt template 把 prompt 划分为 reused/recompute chunks；用 CoPE 低位置敏感 encoding 和任务预训练得到稳定 reused-chunk encoding。
- Weighted Correction Attention：第一层全量/局部重算估计 token discrepancy，后续层只对 top-k discrepancy tokens 做 weighted fusion，并用相似度门控更新。
- SLIDE：Spill awareness、Load-write decoupling、Dirty-page Eviction；在内存压力下优先 spill clean pages，decode 中 overwrite selected-token pages 降低 footprint。

### 16.5 边界与失败条件

- CacheSlide 依赖 agent prompt template 和 reused/recompute chunks 可识别；完全自由变化的 prompt 难以收益。
- 位置编码类型重要；原文基于 CoPE/CCPE，迁移 RoPE 或 Ascend runtime 时要核验模型支持。
- 它解决 prefix/KV reuse 与位置漂移，不是通用 SSD direct I/O；SSD 只是 SLIDE spill-aware 管理的一部分。

### 16.6 前端校准点

- 点击 CacheSlide 时，应高亮：`agent prompt segments -> prefix/KV metadata -> position correction -> KV cache -> optional SSD spill`。
- 不应把它放在 PCIe/SSD 主数据面上；主贡献是状态布局和 cache reuse semantics。

## 17. BurstGPT

原文：`references/papers/BurstGPT__arxiv-2401.17644.pdf`

### 17.1 核心架构一句话

BurstGPT 是真实 LLM serving workload trace/benchmark：包含 Azure OpenAI GPT 服务长时间请求 traces，用于暴露 burstiness、conversation pattern、request/response length、failure rate 对 KV management、scheduler、PD ratio 和 provisioning 的影响。

### 17.2 实际硬件链路

| 链路/组件 | BurstGPT 原文中的角色 | 前端拓扑显示建议 |
|---|---|---|
| Request timestamp / concurrency | 描述 bursty arrivals、weekly/daily pattern、API/conversation 差异 | L0 workload 输入 |
| Request tokens / response tokens | 影响 prefill/decode 负载和 KV 增长 | L0/model 字段 |
| Conversation traces | 反映多轮间隔、KV cache offload/scheduling 需求 | session/reuse 字段 |
| Failure rate / latency jitter / throughput | 用于评估系统稳定性和可靠性 | L11 metrics |
| BurstGPT-Perf | 按 RPS scaling 或 modeled scaling 生成 benchmark workload | 仿真器/benchmark 输入 |
| PD ratio demo | 用 trace 指导 prefill/decode ratio 和动态调整 | scheduler policy 输入 |

### 17.3 解决的问题

- 非 LLM workload 或合成 workload 不能反映真实 LLM serving 的 burstiness、request length、response length 和 failure behavior。
- 相同平均 RPS 下，bursty trace 会显著提高 KV cache pressure、latency jitter 和 failure risk。
- 调度策略、workload provisioning 和 PD disaggregation ratio 在不同 workload 类型上不一定泛化。

### 17.4 关键机制

- Trace dataset：覆盖 Azure OpenAI GPT regional services 的 request start time、request/response token lengths、service/model type、failure requests 等。
- Scaling methods：RPS scaling 调整 trace 时间戳；modeled scaling 调整 Gamma burstiness 参数和 request length distribution。
- BurstGPT-Perf：用 concurrency generator、prompt sampler、CPU buffer、aiohttp 等形成轻量 benchmark suite。
- Evaluation metrics：request failure rate、token latency、latency jitter、throughput，区分 average 与 instantaneous。
- Demo uses：scheduler selection、workload provisioning、PD ratio simulation。

### 17.5 边界与失败条件

- BurstGPT 是 trace/benchmark，不是硬件机制；它不能单独给出 CPU/GPU/SSD 规格。
- 数据来自云端 GPT 服务，未必覆盖本地 coding agent、办公 agent、A+K 一体机多模态私有负载。
- 与 ServeGen 相比，BurstGPT 更偏真实 trace replay/benchmark；ServeGen 更偏 production-scale characterization 和 per-client generation。

### 17.6 前端校准点

- 点击 BurstGPT 时，应只高亮 `request/workload -> telemetry/simulator`，不要高亮主板上的 GPU/CPU/SSD 数据路径。
- 详情栏应显示它给仿真器提供的字段：timestamp、RPS/CV/burstiness、request tokens、response tokens、conversation id/interval、failure rate、latency jitter。

## 18. 给前端整体适配的 PDF 阶段结论

### 18.1 需要避免的误导

| 容易误导的展示 | 应改成 |
|---|---|
| NEO = CPU offload KV cache | NEO = 部分 decode attention + KV 留在 CPU 侧，并通过 asymmetric pipeline 与 GPU 重叠 |
| Tutti = SSD offload | Tutti = GPU-centric HBM-SSD KV object I/O，CPU 从 critical I/O path 中退出 |
| Mooncake = SSD KV pool | Mooncake = P/D 分离 + CPU/DRAM/SSD/RDMA 组成的 disaggregated KVCache，由 Conductor 调度 |
| Bidaw = SSD cache | Bidaw = host memory/SSD two-tier KV + compute-storage bidirectional awareness |
| SolidAttention = SSD 扩容 | SolidAttention = dynamic sparse attention 与 SSD I/O 粒度/预取/调度协同 |
| FlexInfer = 固定 CPU/GPU 切分 | FlexInfer = prefill/decode 分阶段选择 CPU-only、GPU offload 或 SplitGen |
| KTransformers = MoE 权重放 CPU | KTransformers = shared experts/GPU + routed experts/CPU + AMX/AVX/NUMA/CUDA Graph/Expert Deferral |
| LMCache = 一个 KV 后端 | LMCache = inference engine 与 CPU/SSD/remote/network backend 之间的 KV cache layer/API |
| ServeGen/BurstGPT = 硬件优化 | ServeGen/BurstGPT = workload/trace 输入，决定仿真和评测是否真实 |
| ProfInfer/LLMServingSim2.0 = 调度机制 | ProfInfer 是观测，LLMServingSim2.0 是仿真底座，不是具体 offload 策略 |
| llm.npu = 服务器 NPU 结论 | llm.npu = 移动 NPU prefill/offload 方法论，需要重新验证 Ascend/Kunpeng |
| FineMoE/DALI/FluxMoE = 同一种 expert offload | FineMoE 预测 next expert，DALI 动态分配 CPU/GPU，FluxMoE 分页化 expert residency |
| CacheSlide = 普通 prefix cache | CacheSlide = agent prompt 相对顺序保持、绝对位置漂移下的 cross-position KV reuse |

### 18.2 前端数据字段建议

后续更新 `WorkItem` 时，可为每篇增加或细化以下字段：

```text
topologySummary: 一句话核心架构
primaryPath: 热/温/冷/远端/控制面的主路径列表
hardwareClaims: 原文明确涉及的硬件组件
nonClaims: 原文没有证明或不应被前端暗示的内容
criticalAssumption: 依赖的 overlap / locality / hardware capability
frontendHighlight: 点击后应该高亮的节点和边
frontendWarning: 右侧详情栏必须提示的边界
```

### 18.3 前端校准优先级

1. NEO：移除“NUMA 是主贡献”的暗示；突出 GPU-cache/CPU-cache、CPU decode attention、asymmetric scheduling。
2. Tutti：把 CPU 从数据热/温路径里拿掉，只保留控制面/metadata；新增 GPU io_uring/object I/O 说明。
3. Mooncake：拆成 Conductor 控制面、Messenger/RDMA 数据面、CPU DRAM KV pool 温层、GPU prefill/decode 热路径。
4. Bidaw：新增 ready/preparing queue 与 previous-answer-based eviction，避免误画成 direct SSD-GPU。
5. SolidAttention：强调 sparse attention + SSD co-design，和 Tutti 的 direct/object I/O 路线区分开。
6. MoE 组：把 KTransformers、FineMoE、DALI、FluxMoE 拆成四种不同机制，不要合并成“expert 放 CPU/SSD”。
7. 仿真/观测组：ServeGen、BurstGPT、ProfInfer、LLMServingSim2.0 只应驱动 workload、trace、counter、sim schema，不应高亮主数据面。
8. 证据修正：FluxMoE 前端 evidence 应改为 `references/papers/FluxMoE__arxiv-2604.02715v2.pdf`，旧 `2601.07343` 路径必须标注不可引用。
