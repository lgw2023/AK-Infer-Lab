# 面向 Ascend NPU 与 Kunpeng CPU 协同的小算力一体机深度学术调研

## 研究范围与结论先行

这次调研把重点放在“**可迁移到 Ascend NPU + Kunpeng CPU 的机制**”，而不是只看是否直接使用昇腾或鲲鹏。按照你的要求，我优先筛选了近两年系统论文与官方工程资料中最适合落到**单机一体机、Mini 工作站、塔式工作站、单机 1–4 卡、2–8 节点小集群**的方案；同时把公开工业系统也纳入证据链，包括 vLLM、vLLM-Ascend、UCM、LMCache、SGLang HiCache、MindIE、BurstGPT、ProfInfer、Chakra 等。公开证据显示，**直接以“Kunpeng + Ascend 小算力一体机”为论文对象的顶会论文仍然稀缺**，但“CPU+NPU/GPU 协同、分层 KV、阶段化调度、MoE 专家卸载、Agent-aware serving、可观测性闭环”这几条路线已经足够清晰，可以把云侧与移动侧论文中的核心机制下沉到 A+K 单机体系。citeturn34view1turn34view0turn34view2turn34view3turn35view2

最重要的结论有五点。第一，**Kunpeng CPU 不能只当宿主**；近两年的最好结果几乎都来自“CPU 既做计算、又做状态层、又做调度层”，例如把 CPU 用于 attention 子计算、KV/Prefix 的 warm tier、专家索引、工具执行、I/O 聚合和 profiling。第二，**小算力一体机的核心矛盾不是单纯 tokens/s，而是状态对象化**：KV、Prefix、Context、Expert、Session/Trajectory 必须从“引擎内部临时缓存”变成“可持久、可预取、可恢复、可观测”的对象。第三，**P/D 分离不是银弹**；它在长上下文、多租户、严格 TTFT/TPOT 约束下通常有效，但在单机小系统上，逻辑分离往往比物理分离更现实，尤其要警惕 KV 搬运成本反噬。第四，**大 MoE 能否在一体机里“放得下、跑得快”取决于专家冷热分层**，而不是只靠 4bit 量化。第五，**真实 Agent 负载的优化目标应从 tokens/s 转向 TTFT、TPOT、JCT、resume latency、prefix/KV hit、CPU Core·s、NVMe bytes、Tasks/J、P95/P99 稳定性**。这些判断分别受到 llm.npu、PowerInfer-2、NEO、LIA、DistServe、InfiniGen、Mooncake、LMCache、CacheSlide、MoE-Lightning、FineMoE、PASTE、Helium、SAGA、BurstGPT、ProfInfer 与官方 Ascend 软件栈资料的共同支持。citeturn21view0turn27view0turn37academia9turn25view0turn26search0turn21view2turn21view3turn35view2turn21view4turn21view5turn21view6turn21view7turn29academia22turn32search0turn36view2turn30view0turn34view1turn34view2turn34view0

## 跨方向综合分析

**方向 A：CPU + NPU / CPU + GPU 异构协同推理。**  
小算力一体机的主要挑战不是“有没有 CPU 参与”，而是**CPU 参与到哪一层才值回搬运开销**。容量墙体现为 NPU 近端内存装不下大模型与长上下文；带宽墙体现为 CPU↔NPU / DRAM↔HBM / SSD↔DRAM 之间的来回取数；状态驻留体现为 KV、专家权重与工具中间结果若只能驻留在 NPU，就会反复失活；功耗热体现为 NPU 一旦承担非热点计算，单位任务能耗会恶化。当前路线可分为三类：一类像 NEO、LIA，把 CPU 纳入计算路径；一类像 llm.npu、PowerInfer-2，把 NPU 与 CPU 按 tensor/block/neuron cluster 细粒度异构切分；另一类像 Offload/MoE 系统，把 CPU 当成大容量 warm tier。未来 3–5 年，更合理的方向不是“CPU 替 NPU 干更多”，而是**CPU 接住更适合它的计算与状态面**：tool/sandbox、embedding/rerank、KV 相似度、小矩阵、冷专家索引、video codec、恢复控制、trace 采集。citeturn21view0turn27view0turn37academia9turn25view0

**方向 B：Prefill / Decode / Encode 分离与阶段化调度。**  
在云端，多 GPU 的物理 P/D 分离有效；在小算力一体机，更值得做的是**逻辑 P/D/E 分离**。主要挑战是 TTFT 和 TPOT 的最优配置不同，但单机里状态转移成本很高；如果直接照搬 DistServe，一旦 KV transfer 过重，收益会被吃掉。因此当前研究路线中，DistServe 代表“硬分离”，而 AlignedServe、prefix-aware batching、HiCache/UCM 的路子更适合小机：**不一定拆机器，但一定拆调度目标**。未来趋势是把 prefill 看作“计算密集 + 状态生成 + 预取窗口”，把 decode 看作“带宽密集 + 状态消费 + 恢复窗口”，再由统一 runtime 做 rate matching。对 Mini 工作站，我更建议“单机双队列 + 统一对象层”；对塔式工作站，可尝试“逻辑分离 + 独立 worker pool”；对 2–8 节点小集群，才建议真正的 centralized PD with persistent KV。citeturn26search0turn18search0turn34view0

**方向 C：KV / Prefix / Context 生命周期管理与分层卸载。**  
这是我认为与你主题最贴近、也最值得投入的主线。挑战包括：容量墙——长上下文和多轮记忆让 KV 线性增长；带宽墙——即使放到 DRAM/SSD 也要搬回；状态驻留——prefix cache 如果不能跨轮、跨会话、跨重启，就只是“临时省一点 prefill”；跨层搬运——如果缺少 object id / hotness / TTL，分层策略会沦为拍脑袋。当前路线有四类：InfiniGen 代表“只搬必要 KV”；Mooncake / UCM / LMCache 代表“把 KV 变成外部状态层”；CacheSlide 代表“跨位置复用”；ShadowKV 这类路线代表“通过低秩 / 稀疏近似降 footprint”。它们是互补关系，不是替代关系。未来 3–5 年，最有价值的趋势是**从 prefix cache 走向 context object store**：KV 不再只按 token 序列组织，还要附带 owner、version、quality metadata、semantic tags、resume hint。citeturn21view2turn21view3turn35view2turn21view4turn19search1turn34view0

**方向 D：MoE 专家权重管理、冷热专家预测与大模型卸载。**  
小算力一体机里，GLM5.2 / DeepSeek-V4 Flash 一类大 MoE 的核心不是“能不能量化”，而是**近端内存里究竟该常驻什么**。挑战集中在：容量墙——总参数过大；带宽墙——miss expert 的拉取开销；状态驻留——不同 session 热点专家差异大；功耗热——重复拉冷专家会拖垮系统效率。当前研究路线中，MoE-Lightning 代表“CPU-GPU-I/O 流水 + 分页权重”，FineMoE 代表“语义+trajectory 驱动的专家预取”，MoE-APEX 代表“冷专家低精应急加载”。这三条路线可以叠加。未来趋势会从“专家均匀调度”转向“**session-aware / layer-wise / expert-hotness-aware**”调度；更长远看，CPU 侧小专家 / fallback、热专家常驻、温专家预取、冷专家延迟加载，会比单纯 all-to-all expert parallelism 更适合一体机。citeturn21view5turn21view6turn10search2turn11search3

**方向 E：Agent workload 全生命周期感知调度。**  
在小算力一体机里，Agent 最大的问题不是峰值吞吐，而是**工具等待、状态恢复、失败重试与后台任务干扰前台交互**。PASTE、Helium、SAGA 给出的共同方向是：session / workflow / trajectory 应成为一等调度对象；tool-start / tool-end / expected resume time / priority / prefix id 应该下沉给 serving 层；KV pinning、resume-first、background throttling 应该由 runtime 控制。这里也有边界论文式提醒：SAGA 明确指出 workflow-aware scheduling 往往会牺牲一部分峰值吞吐，这是合理代价；对 Coding Agent 与办公 Agent，这种 tradeoff 大多是值得的。未来 3–5 年，**Agent-aware runtime** 会比单点推理优化更重要。citeturn21view7turn29academia22turn32search0turn21view9

**方向 F：多模态生成的异构流水线。**  
多模态生成对小算力一体机不是“再跑一个图像模型”那么简单。首帧、稳帧率、长视频一致性、codec、后处理和磁盘吞吐都会变成系统瓶颈。DiffServe 说明 query-aware routing 能减少无谓的重模型开销；MindIE 官方说明则表明 Ascend 已把 **MindIE SD / MindIE LLM / vLLM/SGLang / RAG/Agent enablement** 放进同一套应用使能层。对 A+K 来说，更合理的流水线是：**CPU 做 text/vision preprocessing、文件与视频 I/O、codec、postprocess、调度；NPU 做 LLM planning、主生成、热点 attention / DiT / diffusion**。未来要特别关注 rolling KV、noise cache、分段生成、首帧时延和能耗。citeturn21view8turn34view3

**方向 G 与方向 H：功耗、热、稳定性、可观测性、trace 与仿真。**  
小算力一体机最容易犯的错误，是拿云端的 tokens/s 当核心 KPI。BurstGPT 告诉我们真实 workload 会暴露完全不同的 burstiness 与 conversation pattern；ProfInfer 告诉我们没有细粒度观测就不知道瓶颈到底在 CPU、NPU、KV 还是 I/O；Chakra 告诉我们没有标准 trace 就做不了 what-if simulator 与硬件反推。未来 3–5 年，最关键的指标体系会从 tokens/s 转向 **TTFT、TPOT、ITL、JCT、prefix/KV hit、resume latency、CPU Core·s、NVMe bytes、Tasks/J、P95/P99、质量损失、温度与恢复时延**。vLLM 已公开支持比较完整的 server-level 和 request-level metrics，SGLang 也把 TTFT、TPOT、ITL、throughput 纳入 bench_serving；LMCache 进一步扩展到了 KV lifecycle 级可观测性。这些都非常适合成为 A+K 内部评测体系的基础。citeturn36view2turn30view0turn36view1turn34view4turn34view5turn34view6

## 面向 Ascend NPU 与 Kunpeng CPU 的小算力一体机技术布局

基于上面的论文与官方资料，我建议把 A+K 的技术布局设计成“**挑战 / 趋势 → 构建 / 探索能力 → 形成策略与指标**”的形式，而不是只做若干分散优化点。

**Agent 负载感知统一调度层。**  
挑战 / 趋势：Agent 负载从单请求变成 session / trajectory / workflow，前台交互与后台工具会相互抢占。  
构建 / 探索能力：在 runtime 中显式引入 `session_id`、`trajectory_id`、`workflow_stage`、`tool_start/tool_end`、`prefix_id`、`expected_resume_time`、`priority`、`resume_budget`。  
形成策略或达到效果：形成 **tool-aware KV 保温**、**resume-first admission**、**前后台任务隔离**、**CPU 工具执行与 NPU 推理协同 admission**。验证指标不只看 TTFT/TPOT，还看 JCT、任务成功率、tool wait hidden ratio、resume latency、CPU Core·s。短期适合 Mini 工作站先做单机版；中期在塔式工作站引入多用户与 LoRA/adapter 多租户；长期扩展到 2–8 节点小集群的 workflow-aware scheduler。citeturn21view7turn29academia22turn32search0turn21view9

**A+K 协同执行矩阵。**  
挑战 / 趋势：CPU 不能再只是 control plane，NPU 也不该处理所有低效碎任务。  
构建 / 探索能力：定义清晰的执行矩阵——**NPU** 负责主 LLM/VLM 推理、主生成、热点 attention、热点 MoE 专家、长序列高吞吐矩阵；**CPU** 负责 tokenizer、解析、sandbox/tool、repo 文件系统、编译与测试、检索、rerank、embedding、小模型、短序列矩阵、KV similarity、冷专家索引、视频编解码、后处理、checkpoint/recovery；**runtime** 负责阶段识别、数据搬运、负载回压、功耗控制、统一 counters。  
形成策略或达到效果：把 Kunpeng 变成真正的 **state plane + tool plane + auxiliary compute plane**。短期目标不是全部 CPU 卸载，而是挑选 decode 中收益最高的 CPU 子任务；中期建设统一 memory / storage metadata；长期形成 CPU/NPU/NVMe 共同组成的推理数据面。citeturn21view0turn27view0turn37academia9turn25view0

**KV / Prefix / Context / Expert 对象层。**  
挑战 / 趋势：一体机真正的容量墙与恢复墙都来自状态对象，而不是单个 kernel。  
构建 / 探索能力：把 KV、Prefix、Context、Expert 定义为统一对象，至少包含 `object_id`、`owner`、`version`、`ttl`、`hotness`、`persistence_level`、`quality_metadata`、`resume_hint`；采用 **HBM/Bailu → DDR/DRAM → SSD/NVMe** 的 hot / warm / cold 策略；支持预取、压缩、落盘、校验、语义聚合、离散 I/O 聚合。  
形成策略或达到效果：形成 **persistent prefix cache**、**cross-position context reuse**、**session-level KV pinning**、**cold object delayed materialization**。短期可先基于 vLLM-Ascend + UCM 或 MindIE + 外部对象层做 prefix/KV；中期把 CacheSlide 式跨位置复用与 LMCache 式引擎解耦能力纳入；长期形成上下文对象的统一数据面。citeturn34view0turn35view2turn21view4turn21view2turn21view3

**MoE 专家与超大模型能力。**  
挑战 / 趋势：小算力一体机既要“放得下” 200B–700B+ 稀疏模型，又要避免冷专家把时延拉爆。  
构建 / 探索能力：建立 **热专家常驻、温专家预取、冷专家延迟加载、CPU 侧小专家 / fallback / index、layer-wise expert prediction** 的五层机制；引入基于 session / trajectory 的 expert hotness 预测。  
形成策略或达到效果：形成适用于 coding / 办公 / 多模态不同流量特征的专家策略库。短期建议塔式工作站优先验证 **MoE-Lightning + FineMoE 思路**：分页权重 + 语义预取；中期叠加 MoE-APEX 式低精冷专家应急；长期再尝试 2–8 节点小集群上的 expert pool 协同。关键指标应增加 expert hit、expert miss penalty、NVMe bytes/token、quality drop。citeturn21view5turn21view6turn10search2turn11search3

**多模态流水线。**  
挑战 / 趋势：多模态生成的瓶颈扩散到 encode、planning、DiT/diffusion、VAE、codec、postprocess、多段缓存与 I/O。  
构建 / 探索能力：做 **text/vision encode → LLM planning → DiT/diffusion → VAE → codec → postprocess** 分阶段流水；NPU 主算、CPU 主 I/O 和 codec；加入 rolling KV、noise cache、segment/chunk generation；支持首帧优先与稳帧率模式。  
形成策略或达到效果：Mini 工作站适合图像与短视频；塔式工作站适合长视频分段生成；2–8 节点小集群再考虑粗粒度 stage parallel。指标要用首帧时延、稳帧率、端到端完成时延、质量损失、显存/内存占用、能耗，而不是只用单阶段吞吐。citeturn21view8turn34view3turn14search4

**Benchmark / Trace / 仿真闭环。**  
挑战 / 趋势：没有真实 traces，就无法把“技术点”变成“硬件需求”。  
构建 / 探索能力：建立三类 benchmark——**Coding Agent、办公 Agent、多模态生成**；采集任务层、模型层、工具层、记忆层、状态层、资源层事件；将 vLLM metrics、SGLang benchmark、LMCache observability、ProfInfer、Chakra 统一到 A+K 自己的 trace schema 上。  
形成策略或达到效果：输出 **硬件参数反推表**，明确某类业务需要多少近端内存、DDR 带宽、SSD IOPS、CPU 核数、NPU 带宽、对象层容量；并通过“单资源扰动”验证瓶颈因果性，例如只增 SSD IOPS、只提 CPU 核数、只扩对象层缓存，看 TTFT/JCT/Tasks/J 的变化。短期就可以用 BurstGPT + OpenHands/内部 Coding Agent traces 做第一版；中期完成 what-if simulator；长期形成下代 A+K 一体机的结构化设计输入。citeturn36view2turn30view0turn36view1turn34view4turn34view5turn34view6
