下面是把**方案 1**和**方案 2**合并后的收敛版。我按你补充的机器条件处理：**Atlas 800T A2，单 NPU 显存 64GB**。如果沿用两个方案里的“单机八卡”前提，那总近端显存约 512GB，但它不是一个天然统一的大显存池，仍要按 **8 张 64GB NPU + NPU 间通信 + CPU/DRAM/SSD 分层**来设计。

## 综合结论

这两个方案可以合成一个项目：

**Atlas 800T A2 上的 A+K 状态分层推理实验栈：以 Ascend NPU 保持热计算主路径，以 Kunpeng/CPU + DRAM/DUMA + NVMe SSD 承接 KV、Prefix、MoE Expert、权重切片、会话状态和 trace；先建立可观测和可复现实验底座，再做 KV/Prefix 分层、MoE 专家分层和阶段级 CPU/NPU 协同，最后用仿真器反推硬件规格与收益边界。**

方案 1 的强项是**体系结构清楚**：热层、温层、冷层、控制/观测层分得比较好；方案 2 的强项是**开工路径清楚**：6 周能产出什么、需要回答哪些硬问题写得更具体。两者应合并为“**6 周 MVP + 12 周深化 + 长期规格反推**”的节奏。

方案 1 明确建议不要先写全新推理框架，也不要先挑战 NPU-SSD 直通或完整 P/D 分离，而是用 MindIE/vLLM-Ascend 跑基线，用 CPU/DRAM/SSD 做状态底座，用 NPU 保持热计算路径；它还把系统分成热层、温层、冷层和控制/观测层。 方案 2 也给出同样判断：NPU 负责热计算主路径，CPU/DRAM/DUMA/SSD 负责状态底座、专家温层、KV/Prefix 温冷层、I/O 聚合、调度与观测，并把优先级定为 P0 可观测 + KV warm tier、P1 Prefix/UCM/Mooncake 与 MoE expert hotness、P2 SSD/CXL/remote cold tier、P3 NPU-native direct storage。

## 4. 技术方向重组为四条主线

### 主线一：可观测底座与硬件画像

这是 P0。不要先优化，先知道瓶颈在哪。

每个请求必须记录：

```text
request_id, session_id, model_name
prompt_tokens, output_tokens
prefill_start/end, decode_step_start/end
TTFT, TPOT / ITL, P50/P95/P99
NPU memory used, HBM occupancy, NPU utilization
CPU utilization, CPU scheduler delay
CPU DRAM used, DDR bandwidth
SSD read/write bytes, SSD queue depth, P95/P99 I/O latency
D2H/H2D bytes, D2H/H2D latency, copy stream overlap
KV hit/miss, prefix hit/miss
expert_ids_per_layer, expert_hit/miss
expert_load_bytes, expert_load_latency
stall_reason
```

方案 2 的 trace schema 已经很接近可执行版本，并强调这一阶段产物不是论文，而是硬件画像脚本、请求级 trace 和单资源扰动脚本。

### 主线二：KV / Prefix / Context 分层

这条先做 DRAM warm tier，再碰 SSD。

最小路径：

```text
active KV 留 NPU HBM
inactive KV async offload 到 CPU DRAM
prefix cache / reusable context 放 DRAM 或 UCM external KV
SSD 只放 cold KV / cold prefix / trace
```

实验矩阵：

| 实验                 | 变量                                    | 指标                                   |
| -------------------- | --------------------------------------- | -------------------------------------- |
| KV CPU Offload 开/关 | offload size、block size、pinned memory | TTFT、TPOT、P95、H2D stall、HBM 释放量 |
| Prefix cache 开/关   | prompt 重复率、prefix 长度              | TTFT、prefix hit、HBM 占用             |
| 长上下文恢复         | 4K/16K/32K/64K/128K                     | restore latency、recompute latency     |
| 多轮会话             | session 间隔 1s/10s/60s/10min           | resume latency、KV 保温收益            |

方案 2 强调先做 DRAM warm tier，不要先上 SSD；还给出了 KVObject schema，并指出 SSD 只适合 cold KV、cold expert 和 trace/profile。

### 主线三：MoE Expert 分层与预取

对 GLM5.2/DeepSeek/Qwen MoE，真正主战场是 expert weights，而不是单纯 KV。

三层专家池：

| 层级                               | 放什么                                                       | 策略                                                  |
| ---------------------------------- | ------------------------------------------------------------ | ----------------------------------------------------- |
| **NPU HBM hot expert cache**       | shared expert、non-expert 主干、hot routed experts、当前层即将使用的 experts | 常驻 + pin + 短窗口预取                               |
| **CPU DRAM/DUMA warm expert tier** | warm routed experts、session 相关专家、下一窗口候选专家      | mmap/pinned staging、大块预取、异步 H2D               |
| **SSD cold expert tier**           | cold experts、低频 experts、低精度 emergency experts、checkpoint | 大块顺序读；只允许预取式恢复，不允许随机逐 token 拉取 |

先做 expert trace，不要直接写复杂策略。每层记录：

```text
layer_id
token_id
topk_expert_ids
expert_score
expert_frequency
expert_reuse_distance
expert_tier: HBM / DRAM / SSD
expert_hit/miss
expert_miss_penalty
expert_load_bytes
expert_load_latency
prefetch_lead_time
wrong_prefetch_bytes
```

策略从简单到复杂：

| 版本 | 策略                                             | 目的                                 |
| ---- | ------------------------------------------------ | ------------------------------------ |
| V0   | 静态 hot experts 常驻 HBM                        | 验证容量分层是否跑通                 |
| V1   | 基于历史 top-k 的 sliding-window prefetch        | 降低 expert miss                     |
| V2   | session / prompt / layer-aware expert prediction | 面向 Coding Agent / 长会话压榨命中率 |
| V3   | cold expert 低精度 emergency fallback            | 减少 miss 搬运字节，但必须做质量回归 |

方案 2 明确把 MoE expert 管理列为核心方向，要求记录 expert size、top-k、routing distribution、miss penalty、prefetch accuracy；方案 1 也给出 HBM hot expert cache、DRAM warm expert tier、SSD cold expert tier 三层专家策略。

### 主线四：CPU/NPU 阶段级协同计算

这里要收敛口径：**CPU 不主算，CPU 只承接冷、短、稀疏、状态相关、可重叠的子路径。**

适合 CPU 的任务：

| 优先级 | CPU 侧任务                                                   | 说明                             |
| ------ | ------------------------------------------------------------ | -------------------------------- |
| P0     | tokenizer、sampling、request scheduling、KV metadata、prefix hash | 稳定收益，低风险                 |
| P0     | KV/prefix lookup、LRU/TTL、session-aware eviction            | 支撑状态对象层                   |
| P1     | expert hotness prediction、prefetch planner                  | 对 MoE 大模型收益最大            |
| P1     | I/O aggregation、SSD 大块读写编排                            | 避免小随机 I/O 进入热路径        |
| P2     | KV similarity、短矩阵、小 batch fallback                     | 只有 microbench 证明收益为正才做 |
| P2/P3  | fallback expert / partial attention                          | 必须证明可被 NPU 主路径掩盖      |

方案 2 明确说 Kunpeng 不要主算化，CPU 最适合 control plane、DRAM warm tier、可重叠计算、小矩阵/低频 fallback、I/O aggregation；高效 expert compute 要等逐 shape/SVE 实测后再判断。

## 10. 最终版项目方案表述

可以直接写成项目 charter 里的版本：

**项目名称：**
Atlas 800T A2 A+K 状态分层推理实验栈

**项目目标：**
基于 Atlas 800T A2 的 Ascend NPU 64GB 近端显存约束，构建 MindIE + vLLM-Ascend 双轨推理底座；通过 KV/Prefix DRAM warm tier、UCM/external KV、MoE expert hotness trace、专家冷热分层、SSD cold tier 和 trace-driven simulator，验证 Qwen/DeepSeek/GLM 类模型在 A+K 架构上的容量、性能和失效边界。

**技术主线：**
NPU 负责 dense hot path；CPU/DRAM/DUMA 负责 warm state、metadata、routing trace、prefetch planner、I/O aggregation；NVMe SSD 负责 cold KV、cold experts、checkpoint 和 trace；所有 KV、Prefix、Expert、WeightShard、Session 状态统一抽象为 state object，纳入可观测、可迁移、可恢复、可仿真的运行时。

**近期验收：**
6 周内跑通 Qwen/DeepSeek 小中模型基线，完成 KV CPU Offload、UCM/KVObject 原型、MoE expert trace 和 expert tier V0；输出这台 Atlas 800T A2 上 HBM、DRAM、CPU、HCCS/HCCL、SSD、CANN kernel 的瓶颈归因。

**中期验收：**
12 周内完成 KV/Prefix 分层、MoE expert 热缓存、DRAM/SSD expert 分层、prefetch policy 和 miss penalty 曲线；建立 A+K serving simulator，能回答 HBM 增减、DRAM 带宽变化、SSD P99 抖动、CPU 核数变化、TP/EP/DP/CP 配比变化对 TTFT/TPOT/P95/P99 的影响。

**明确不承诺：**
第一阶段不承诺 GLM5.2 满血低时延服务，不承诺 1M 上下文，不承诺 NPU-SSD 直通，不承诺 CPU 替代 NPU 主算，不把完整物理 P/D 分离作为默认收益路径。方案 1 和方案 2 都强调这三件事应避免：SSD 不做热层、Kunpeng 不做主算替代、物理 P/D 不先做。
