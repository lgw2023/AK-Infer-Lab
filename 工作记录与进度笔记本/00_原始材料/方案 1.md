可以开干，但路线要收敛：**不要一上来写一个全新推理框架，也不要先挑战 NPU-SSD 直通或完整 P/D 分离。你现在这台八卡 Ascend 服务器最适合作为“A+K 协同推理实验台”：用 vLLM-Ascend / MindIE 跑基线，用 Kunpeng+DRAM+SSD 做状态底座，用 Ascend NPU 保持热计算路径，然后逐步把 KV、Prefix、MoE Expert、权重切片、工具链状态纳入统一对象管理。**

项目内已有材料的核心口径也是这个：A+K 协同不是笼统“CPU 加速 NPU”，而是让 **NPU 专注热计算主路径，CPU/DRAM/DUMA/SSD 承接温冷状态、预取、恢复、索引、调度和可观测闭环**。

## 1. 总体技术路线：先做“状态分层”，再做“协同计算”

你这台八卡机可以按四层来搭系统：

| 层级            | 硬件                               | 应放什么                                                     | 软件动作                                                     |
| --------------- | ---------------------------------- | ------------------------------------------------------------ | ------------------------------------------------------------ |
| **热层**        | 8× Ascend NPU / HBM / HCCS         | 主干权重、当前 batch activation、active KV、hot experts、workspace | TP/PP/EP、PagedAttention、FlashDecoding、continuous batching |
| **温层**        | Kunpeng CPU + DRAM / DUMA          | inactive KV、prefix cache、warm experts、routing trace、page cache、staging buffer | CPU KV offload、UCM、Mooncake Store、expert prefetch         |
| **冷层**        | NVMe SSD                           | cold KV、cold experts、checkpoint、trace、历史 session       | SSD offload、chunk/object I/O、异步恢复、TTL/eviction        |
| **控制/观测层** | CPU runtime + profiler + simulator | 调度器、metadata、event trace、瓶颈归因、what-if 仿真        | CANN/MSProfiler/eBPF/fio/HCCL microbench/自研 simulator      |

项目文档里对八卡/一体机场景的硬件映射已经很清楚：Bailu/HBM 放热权重和热 KV，DDR 放冷权重与温 KV，SSD 放冷 KV，关键是“直通、预取、换出”的策略是否能避免进入逐 token 热路径。

## 2. 不要从 GLM5.2/DeepSeek 671B 直接开刀

建议分三档模型推进：

| 阶段             | 模型选择                                             | 目的                                                        |
| ---------------- | ---------------------------------------------------- | ----------------------------------------------------------- |
| **M0：基线模型** | Qwen/QwQ 32B、Qwen 72B、DeepSeek-R1-Distill-Qwen 32B | 跑通 vLLM-Ascend / MindIE、TP/PP、benchmark、profile        |
| **M1：中型 MoE** | Qwen-MoE、DeepSeek-V2/V3 小配置或可加载切片          | 采集 expert routing trace，验证专家热度、预取、miss penalty |
| **M2：巨型 MoE** | DeepSeek-V3/R1 671B、GLM5.2 744B、Qwen 大 MoE        | 做低比特 + 多卡并行 + 专家分层 + KV/Prefix 分层的极限实验   |

GLM5.2 这类 744B MoE 的第一矛盾不是 KV，而是 **MoE 专家权重容量**；项目材料里已经把“热专家近端保活、温专家提前回温、冷专家低成本恢复”定为第一优先级。

## 3. 直接站在这些工程框架上

### 主推底座：vLLM-Ascend + MindIE 双轨

**vLLM-Ascend** 更适合你做研究原型，因为它已经有 KV Cache CPU Offload：inactive KV 可从 NPU memory 异步 offload 到 CPU memory，使用 `OffloadingConnector`、Ascend-specific `NPUOffloadingSpec`、独立 NPU streams、CPU block pool 和 LRU 管理。([vLLM](https://docs.vllm.ai/projects/ascend/en/main/user_guide/feature_guide/kv_cache_cpu_offload.html))

**MindIE LLM** 更适合做稳定基线。官方 GitHub 描述它是昇腾大语言模型推理加速套件，包含 Continuous Batching、PagedAttention、FlashDecoding 等推理加速特性。([GitHub](https://github.com/Ascend/MindIE-LLM))

建议分工：

| 用途                                          | 选型                                     |
| --------------------------------------------- | ---------------------------------------- |
| 稳定服务基线                                  | MindIE                                   |
| 快速改 connector / KV / prefix / PD / profile | vLLM-Ascend                              |
| MoE expert offload 思想参考                   | KTransformers / FineMoE / DALI / FluxMoE |
| KV pool / Prefix cache / PD 参考              | UCM / Mooncake / LMCache                 |
| SSD-backed KV 远期参考                        | Tutti / SolidAttention / Bidaw           |

注意：**KTransformers 目前不能当作 Ascend 直接落地底座。** 它的支持矩阵明确把 Ascend NPU 旧页面列为历史文档、当前不支持，所以它更适合作为 MoE 专家分层与 CPU expert execution 的参考，而不是当前可直接跑的 Ascend runtime。([KTransformers](https://ktransformers.net/zh/docs/support-matrix))

## 4. 第一阶段：先建可观测底座，否则后面全是拍脑袋

你要先搭一个 `AK Trace / Profile / Simulator` 小系统。不要先优化，先能回答这些问题：

| 问题                     | 必须记录的字段                                               |
| ------------------------ | ------------------------------------------------------------ |
| NPU 卡在哪里等？         | kernel timeline、HBM occupancy、AIC/AIV utilization、copy stream |
| CPU 是否在拖后腿？       | CPU core util、NUMA、IRQ binding、scheduler delay、SVE/OMP kernel time |
| DRAM 是否够用？          | bandwidth、page fault、pinned memory、staging buffer 占用    |
| PCIe/UB/HCCS 是否瓶颈？  | H2D/D2H bytes、DMA latency、overlap ratio、HCCL all-reduce/all-to-all |
| SSD 是否可进入冷层？     | seq/random IOPS、P95/P99 latency、queue depth、read amplification |
| KV/Prefix 是否真的命中？ | HBM hit、CPU hit、external hit、restore latency、prefix reuse |
| MoE 专家是否有局部性？   | layer、expert id、top-k、hit/miss、next-use distance、prefetch accuracy |

项目 Deep Research 也把这一点放在 P0：workload/profiler/simulator host、KV Cache CPU Offload、Prefix/APC/UCM external KV 是 Ascend+Kunpeng 的近期工程支点。

工程上可以建一个仓库结构：

```text
ak-infer-lab/
  bench/              # NPU/CPU/DRAM/SSD/PCIe/HCCL microbench
  traces/             # request, kv, expert, kernel, io event schema
  vllm_ascend_patches/ # KV events, router trace, connector experiments
  policies/           # KV eviction, expert hotness, prefetch policies
  simulator/          # what-if: HBM/DRAM/SSD/PCIe/CPU/NPU 扫描
  dashboards/         # TTFT/TPOT/P95/P99/bytes/token/energy/task
```

## 5. 第二阶段：先做 KV / Prefix / Context 分层

这是最容易和现有框架接上的方向。

### 5.1 P0：打开 vLLM-Ascend CPU KV Offload

从小模型开始，配置 CPU block pool，验证：

```text
NPU HBM active KV
  -> inactive KV async D2H
  -> CPU DRAM block pool
  -> prefix hit 后 async H2D restore
```

vLLM-Ascend 文档明确支持 `kv_connector="OffloadingConnector"`、`kv_role="kv_both"`、`num_cpu_blocks`、`block_size` 等参数，并支持 KV cache events 用于监控和调试。([vLLM](https://docs.vllm.ai/projects/ascend/en/main/user_guide/feature_guide/kv_cache_cpu_offload.html))

### 5.2 P1：上 UCM / Mooncake Store

UCM 的方向更接近你要做的“状态对象层”：PipelineStore 把 Device↔Host 的 Cache Store 和 Host↔Storage backend 的 Posix Store 串起来，支持 prefix cache 复用、持久化和外部存储。vLLM-Ascend 的 UCM 文档还给了单机 8 卡 PD-Mixed 的参考配置。([vLLM](https://docs.vllm.ai/projects/ascend/en/main/user_guide/feature_guide/ucm_deployment.html?utm_source=chatgpt.com))

Mooncake 的官方定位是 KVCache-centric disaggregated architecture，利用 CPU、DRAM、SSD 资源构建 KVCache pool；在 vLLM 中，MooncakeStoreConnector 支持 CPU/disk offload、跨实例 prefix cache sharing、单机和多机部署。([KVCache AI](https://kvcache-ai.github.io/Mooncake/)) ([vLLM](https://docs.vllm.ai/en/latest/features/mooncake_store_connector_usage/))

### 5.3 P2：SSD 只做冷层，不进逐 token 热路径

Tutti 的反例非常关键：SSD-backed KV 的主要问题不是 SSD 标称带宽，而是 fragmented KV layout 导致 tiny random I/O、CPU-centric I/O control path、DRAM-HBM bounce copy 和 GPU/NPU stall。Tutti 的解法是 GPU-centric object store、GPU io_uring、slack-aware I/O；Ascend 上不能默认已有等价 NPU-native SSD path。([arXiv](https://arxiv.org/abs/2605.03375))

所以你当前的 SSD 路线应该是：

```text
DRAM warm KV / prefix 优先
SSD cold KV / cold prefix 只做可预取、可批量、可异步恢复
绝不让 SSD random read 成为每 token decode 的必经路径
```

## 6. 第三阶段：做 MoE Expert 分层，这是跑大模型的主战场

对 DeepSeek、GLM、Qwen 的大 MoE，核心不是“把权重放哪”，而是把每个 expert 做成对象：

```text
ExpertObject {
  model, layer_id, expert_id,
  size_bytes, precision,
  resident_tier: HBM | DRAM | SSD,
  hotness, last_used, next_use_prob,
  prefetch_deadline, miss_penalty,
  quality_risk
}
```

然后建立三层策略：

| 层                        | 放什么                                        | 策略                                                 |
| ------------------------- | --------------------------------------------- | ---------------------------------------------------- |
| **HBM hot expert cache**  | shared experts、主干权重、热点 routed experts | 常驻、pin、按 session/task 更新热度                  |
| **DRAM warm expert tier** | 大部分温专家、最近命中过的专家                | layer-wise prefetch、pinned staging、异步 H2D        |
| **SSD cold expert tier**  | 低频专家、checkpoint、备用量化版本            | 只允许 chunk 化、预取式恢复，不允许随机逐 token 拉取 |

代表工作怎么用：

| 工作              | 你该吸收什么                                                 |
| ----------------- | ------------------------------------------------------------ |
| **KTransformers** | shared/hot experts 留加速器，routed experts 可由 CPU/DRAM 承接；但 AMX 结论不能直接迁到 Kunpeng |
| **FineMoE**       | 用 prompt semantic hint、session、trajectory 预测 expert map |
| **DALI**          | workload-aware expert assignment、residual-based prefetch    |
| **FluxMoE**       | expert paging / transient expert object，让 HBM 优先给 KV、activation、hot expert |
| **MoE-APEX**      | cold expert 可用低精度应急，减少 miss 搬运字节               |

项目的 Deep Research 已经把 MoE expert 管理列为独立方向：核心瓶颈是专家参数大、token 激活稀疏、HBM 与 KV 争空间；建模指标包括 expert size、top-k、routing distribution、miss penalty、prefetch accuracy。

------

## 7. 第四阶段：再做 CPU/NPU 协同计算，不要反过来

Kunpeng CPU 的角色要分清楚。项目材料里已经把 CPU 分成五种角色：控制面、warm tier、可重叠计算、高效 expert compute、I/O aggregation；并强调 CPU expert compute 不能直接继承 Intel AMX 结果，需要 SVE/SME microbenchmark 证明。

你可以按收益概率排序：

| 优先级 | CPU 侧任务                                                   | 原因                                     |
| ------ | ------------------------------------------------------------ | ---------------------------------------- |
| P0     | tokenizer、sampling、request scheduling、KV metadata、prefix hash | 稳定收益，低风险                         |
| P0     | KV/prefix lookup、LRU/TTL、session-aware eviction            | 直接支撑长上下文和多轮会话               |
| P1     | 向量检索、BM25、rerank、小矩阵、index gather                 | CPU/SVE 更适合不规则和小批任务           |
| P1     | MoE expert routing trace、hotness prediction、prefetch planner | 对大 MoE 收益最大                        |
| P2     | partial attention / layer-ahead CPU precompute               | 参考 NEO/KVPR/ScoutAttention，但必须实测 |
| P2/P3  | CPU fallback expert / low-frequency expert compute           | 只有 SVE/OMP kernel 证明收益为正才做     |

Ascend 侧也有值得关注的新方向：2026 年 HyperParallel-MoE 论文显示 Ascend NPU 内部存在 AIC/AIV 等异构 on-chip compute 资源，可通过 tile-level taskflow、AIV-driven one-sided communication、event-driven static scheduling 来降低 MoE Dispatch-to-Combine 延迟；但这篇是训练场景，不应直接写成推理收益，只适合作为后续 AIC/AIV 内部协同调度参考。([arXiv](https://arxiv.org/abs/2605.23764))

## 8. 量化路线：先稳 INT8/FP8，再碰 W4A8/INT4

大模型能不能“尽可能跑下”，低比特是前提，但 Ascend 上不能照抄 CUDA 生态的量化结论。2026 年一个 Ascend NPU PTQ case study 报告了明显平台敏感性：4-bit weight-only 对较大模型可行，但 aggressive 4-bit weight-activation 在长上下文 reasoning 任务上有 layer-wise calibration instability 和 logic collapse 风险；标准 8-bit 更稳定，INT8 部署中动态量化开销也会限制端到端加速。([arXiv](https://arxiv.org/abs/2602.17693))

建议顺序：

```text
BF16/FP16 baseline
-> FP8 / W8A8 / INT8 稳定路径
-> W4A8 / INT4 weight-only
-> expert-level mixed precision
-> cold expert lower precision fallback
```

对 GLM5.2/DeepSeek 这类巨型 MoE，最终目标不是全模型统一低比特，而是：

```text
主干/高频专家：高稳定精度
温专家：低比特常驻 DRAM
冷专家：更激进低比特 + 需要时恢复
KV：先 FP8/BF16 可靠性验证，再做 KV 量化
```

## 9. 单机八卡上的建议实验顺序

### 第 0 步：硬件盘点

先拿到这些实测值，不要用宣传规格代替：

```text
每卡 HBM 容量/带宽
8 卡 HCCS/HCCL 拓扑与 all-reduce/all-to-all 带宽
CPU socket / NUMA / 内存通道 / DRAM 带宽
NPU <-> CPU H2D/D2H 带宽和 DMA setup latency
NVMe seq/random read/write、P95/P99、queue depth
CPU SVE/OMP 小矩阵、向量检索、prefix hash 吞吐
```

### 第 1 步：跑基线

用 MindIE 和 vLLM-Ascend 各跑一套：

```text
Qwen/QwQ 32B
Qwen 72B
DeepSeek-R1-Distill-Qwen 32B
一个中型 MoE
```

指标：

```text
TTFT, TPOT, tokens/s
P50/P95/P99
HBM occupancy
NPU util
CPU util
H2D/D2H bytes
HCCL/HCCS traffic
```

### 第 2 步：打开 KV CPU Offload

先验证：

```text
重复 prefix workload
长上下文 workload
多轮 session workload
```

不要先跑随机 prompt。随机 prompt 对 prefix/KV cache 不友好，会让你误判 offload 没价值。

### 第 3 步：接 UCM / Mooncake

先做单机：

```text
8 卡 PD-Mixed + UCM prefix cache
CPU DRAM external KV
本地 NVMe cold backend
```

再做逻辑 P/D：

```text
4 卡 prefill + 4 卡 decode
或 2P6D / 3P5D 扫描
```

物理 P/D 分离只有在 KV transfer 可被计算覆盖、prefix 命中足够高、长上下文占比足够大时才赚钱。项目报告也明确说，P/D 或 E/P/D 拆分不是默认正收益。

### 第 4 步：采集 MoE routing trace

在中型 MoE 上先记录：

```text
request_id
session_id
layer_id
topk_expert_ids
expert score
expert HBM hit/miss
expert load source: HBM/DRAM/SSD
load latency
prefetch lead time
```

然后再上 DeepSeek/GLM/Qwen 大 MoE 的切片或量化版本。

### 第 5 步：做 expert cache policy

先实现最简单三种策略：

```text
LRU expert cache
LFU / hotness expert cache
session-aware expert cache
```

然后再做 FineMoE/DALI 式预测：

```text
semantic hint + routing trajectory
residual-based prefetch
layer-wise sliding window prefetch
```

### 第 6 步：做 simulator 反推

每次实验必须能输出：

```text
如果 HBM +20%，TPOT 降多少？
如果 DRAM 带宽减半，expert miss penalty 增多少？
如果 SSD P99 翻倍，TTFT 尾部变多少？
如果 CPU core 从 64 降到 32，scheduler 和 prefetch 是否阻塞？
如果 4P4D 改成 2P6D，goodput 是否提高？
```

这就是把“开干”变成可积累技术资产的关键。

## 11. 当前最应避免的三件事

第一，**不要把 SSD 当热层。** SSD 是 cold tier，只有 chunk 化、对象化、异步预取、可被 slack 掩盖时才有价值；否则会被 tiny random I/O 和控制路径拖垮。([arXiv](https://arxiv.org/abs/2605.03375))

第二，**不要把 Kunpeng 写成主算替代。** CPU 优先做 metadata、KV/expert warm tier、I/O aggregation、routing trace、工具链、profile/simulator host；只有经过 SVE/OMP microbench 的低频子算子，才进入计算路径。

第三，**不要先做完整物理 P/D。** 单机八卡先做 PD-Mixed、逻辑 P/D、4P4D/2P6D 扫描；等 KV transfer、prefix hit、queueing 和 HCCS/HCCL 数据都清楚后，再判断是否值得物理分离。

## 12. 一句话落地方案

你的第一版工程原型可以定义为：

**基于 vLLM-Ascend 的八卡 A+K 异构推理实验栈：NPU 负责 TP/PP/EP 热计算，Kunpeng+DRAM 负责 KV/Prefix/Expert warm tier，NVMe 负责 cold state/checkpoint，UCM/Mooncake 负责外部 KV，对 MoE 专家建立 routing trace 与热度预测，对所有 HBM/DRAM/SSD/PCIe/HCCS/CPU/NPU 事件建立 trace 和 simulator，逐步从 Qwen/QwQ 32B/72B 推到 DeepSeek/GLM/Qwen 大 MoE。**

这条路线最现实，也最容易沉淀成你们自己的技术壁垒。
