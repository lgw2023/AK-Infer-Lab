common metric 读法：本节所有 vs 都是 prefix_cache_on vs prefix_cache_off。

Benchmark Duration = 最后一个请求结束时间 - 第一个请求开始时间；本轮表示 32 个请求整体完成窗口为 20.701s vs 37.693s。

Request Throughput = 成功请求数 / Benchmark Duration；两边都是 32 个成功请求，所以得到 1.546 vs 0.849 req/s。

Output Token Throughput = 总生成 token 数 / Benchmark Duration；两边都是 32 × 64 = 2048 个输出 token，所以得到 98.93 vs 54.33 token/s。

Total Token Throughput = (总输入 token + 总输出 token) / Benchmark Duration；本轮两边总输入 token 都是 220502，总输出 token 都是 2048，因此得到 10750.54 vs 5904.32 token/s。这个数受输入 token 主导，不应直接理解成 decode token/s。

边界：这些是 vLLM OpenAI streaming client 的 benchmark-window 口径，能说明 P1.29 这个固定 workload 下 on 组整体更快，但不能单独证明 HBM、KV cache 或 scheduler 的底层瓶颈原因。

P1.30 四个内存指标读法：这些读数来自 phase memory probe，用来回答“服务进程和整卡显存当时大概处在什么占用状态”，不是瓶颈归因。

P1.30 HBM occupancy = npu-smi 看到的整张 NPU HBM 已用量。55050.24 MB 表示采样时整卡已有约 55.05 GB 被占用，包含模型权重、运行时、KV cache、allocator 预留等整卡层面的占用；它不是某一个请求单独用了 55 GB，也不是 HBM 读写流量。

P1.30 HBM free min = 同一采样窗口里整卡 HBM 剩余量的最小值。10485.76 MB 与 55050.24 MB 基本相加为 64GB 级别卡容量；HBM usage max 84.0% 表示最高占用率约 84%，只能说明容量占用状态，不能说明带宽是否吃满。

Host RSS/PSS range = 主机侧 vLLM API server root pid 及子进程的内存足迹。RSS 是实际驻留在内存里的页总和，PSS 会把共享页按比例分摊，所以 PSS 通常低于 RSS；~5.09GB / ~4.38GB 是服务进程组级别，不是整机 DRAM 用量，也不是 NPU HBM。

KV usage proxy = vLLM server stats 暴露的 KV cache 使用率代理。9.4% vs 16.1% 分别是本地归档附件中的 prefix_cache_on 与 prefix_cache_off 最大值；off 较高可理解为 on 复用重复 prefix 的 KV blocks，而 off 仍需按请求保留 KV，因此峰值更高；它能作为 KV pool 占用线索，但不能替代 KV object bytes、page、hit/miss、eviction 或 restore latency 证据。
