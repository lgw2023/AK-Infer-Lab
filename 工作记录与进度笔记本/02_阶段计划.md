# 02 阶段计划

## P0 笔记本与范围固化

目标：把已有方案材料、当前工作目标和范围边界结构化。

交付物：
- 笔记本目录。
- 原始材料归档。
- 工作记录、阶段计划、进展、问题、下一步行动页。

验证：
- 当前范围明确排除真实 coding agent 运行。
- 原始方案文档可从笔记本中找到。

## P1 标准推理链路与资源账本

目标：定义一次请求从输入到输出的标准推理 stage，并为每个 stage 绑定硬件资源和指标。

关键内容：
- tokenize / detokenize。
- prefill。
- decode step。
- attention 和 KV cache。
- dense MLP 或 MoE expert。
- sampling。
- H2D/D2H 与 copy stream。
- CPU 调度、DRAM、SSD 和 profiler 侧数据。

交付物：
- `inference_stage_schema`。
- `hardware_resource_schema`。
- `bottleneck_reason` 枚举。

验证：
- 每个 stage 都能映射到至少一个采集指标。
- 每个瓶颈原因都能被 trace 或 microbench 数据支持。

## P2 静态提示词 workload

目标：构造用于推理测试的提示词数据，不运行真实 agent。

关键内容：
- 普通问答短 prompt。
- 中等代码解释 prompt。
- 长 system prompt + 用户任务。
- 多轮对话拼接。
- 工具输出文本粘贴为普通上下文。
- 重复 prefix 和 session 恢复场景。

交付物：
- prompt manifest。
- prompt 样例集。
- workload matrix：context_len、output_len、concurrency、prefix_reuse。

验证：
- 能覆盖 512、1K、4K、8K、16K、32K 上下文档位。
- 每条提示词都有 token 规模、场景标签和预期输出长度。

## P3 硬件 microbench

目标：先测硬件链路，建立解释模型推理 trace 的基础数据。

关键内容：
- H2D/D2H pageable、pinned、sync、async、overlap。
- NPU matmul shape。
- CPU cycles、instructions、cache misses、context switches。
- DRAM bandwidth。
- SSD fio latency 和 queue depth。

交付物：
- `h2d_d2h_latency.csv`。
- `h2d_d2h_bandwidth.csv`。
- `npu_matmul_by_shape.csv`。
- `cpu_perf.csv`。
- `dram_bandwidth.csv`。
- `ssd_fio.csv`。

验证：
- microbench 输出能进入统一结果表。
- 每类硬件都有可复现实验命令和环境信息。

## P4 小模型推理 trace smoke

目标：用小模型验证推理、metrics、profile、trace schema 全链路。

关键内容：
- Qwen/Qwen3 小模型或同等小模型。
- context_len：512、1K、4K。
- output_len：32、128。
- concurrency：1、2、4。

交付物：
- 小模型请求 trace。
- TTFT/TPOT/HBM/NPU/CPU/copy 汇总。
- 首版 trace 样例。

验证：
- 能生成文本。
- 能采集请求级、硬件级和 copy 指标。
- 能把每次请求写入 experiment card。

## P5 中型 MoE 正式 profile

目标：在可加载的中型 MoE 上拿到第一份有价值的专家路由和硬件 profile。

关键内容：
- Qwen3-30B-A3B 或同等可支持 MoE 模型。
- context_len：1K、4K、8K、16K、32K。
- output_len：128、512。
- concurrency：1、2、4。
- random prompt、repeated prefix、multi-turn prompt。

交付物：
- MoE profile report。
- expert routing trace。
- expert hotness histogram。
- TTFT/TPOT vs context/concurrency 曲线。

验证：
- 能判断 64GB HBM 剩余空间。
- 能判断 decode 是否 memory-bound。
- 能判断 expert top-k 是否有热度局部性。

## P6 瓶颈归因与 what-if 模拟

目标：用 trace 和 microbench 建立最小 simulator，回答硬件瓶颈和优化方向。

关键内容：
- HBM 容量变化。
- DRAM 带宽变化。
- H2D/D2H latency 变化。
- SSD P99 抖动。
- expert hot cache 容量变化。
- prefix/KV 命中率变化。

交付物：
- bottleneck attribution report。
- what-if table。
- 下一阶段工程选择建议。

验证：
- 对每个慢请求给出 top-3 stall reason。
- 能说明优先做 KV offload、expert hot cache、CPU fallback，还是只优化调度和观测。
