## 1. 前置证据

P0-P4 当前阶段已经按三类目标收尾：

- 服务器环境与硬件天花板基线：`obs_2026_0705_atlas800t_a2_006` 与 `hardware_ceiling_sweep_2026_0708_p0_007`。
- 小模型推理链路跑通：`Qwen3.5-4B + transformers + torch_npu` 与 vLLM / vLLM-Ascend 小模型路径。
- 小 prompt 推理链路观测：P1.23-P1.28 的 msprof、request-device 聚合和 readout 闭环。

关键边界：

- P0/P3 是 synthetic hardware ceiling，不是模型推理 benchmark。
- P1.28 是 raw counter readout，不是吞吐、scheduler 效率、prefix cache 命中率验收、瓶颈归因或优化建议。
- 本地机器是外部开发机；Ascend/NPU、权重路径、Docker 推理栈和服务器部署只能由后续服务器任务验证。

## 4. P6：单机八卡 baseline benchmark

目标：在 Atlas 800T A2 8x64GB 上建立 DeepSeek-V4-Flash official/degraded baseline。

首选对象：

```text
model_object: deepseek_v4_flash_w8a8_mtp_modelscope
runtime: vLLM-Ascend
quantization: ascend
tp: 8
ep: enabled
```

阶段：

- P6.0 权重路径与环境 readiness：只检查路径、metadata、CANN、torch_npu、vLLM-Ascend、8 卡健康状态。
- P6.1 quick smoke：4K + fixed 64 tokens，先不跑 msprof。
- P6.2 profiled smoke：同负载加 msprof 和 request-device aggregate。
- P6.3 controlled baseline：4/8/16 请求 mixed，固定输出长度。
- P6.4 单变量 A/B：prefix cache、chunked prefill、MTP、max_model_len、max_num_seqs。

判读边界：

- 如果降低上下文、关闭关键官方开关或修改并行策略才能跑通，标记为 `degraded_smoke`。
- benchmark 结论必须有固定输出长度、完整 server command、server stats、msprof、request-device aggregate 和已知混杂因素。

## 5. P7：单卡/双卡极致硬件场景

目标：验证 64GB/128GB 近端容量边界，不承诺 official DeepSeek-V4-Flash 生产部署。

对象分层：

- 小模型：验证 trace、prefix、KV offload、profiler。
- 中型 MoE：验证 expert trace、hotset、miss penalty。
- DeepSeek source checkpoint 或子图：验证 tokenizer/config、局部算子、权重分片、KV shape 和格式转换风险。
- 模拟 expert pool：验证 HBM/DRAM/SSD 分层策略。
- simulator-only full model：用全模型参数做 what-if。

验收：

- 能明确说明哪些路径可跑、哪些路径不可跑，以及失败发生在容量、格式、kernel、runtime、通信还是状态恢复。
- 不把 8x64GB 聚合 HBM 写成单卡 512GB。
- 单卡 64GB 和双卡 128GB 只输出边界、失效原因和 simulator 校准点。

## 6. P8：AK 协同技术注入与对照实验

优先级：

1. KV / Prefix 管理：prefix cache、KV Cache CPU Offload、UCM；Mooncake KV Pool 放在后半段。
2. MoE expert trace：先记录 router top-k、expert score、reuse distance、hit/miss、load latency。
3. Expert 热温冷分层：先 static hotset，再 warm tier prefetch，最后才考虑 cold tier。
4. CPU/NPU 阶段级协同：CPU 做 metadata、scheduler、prefix hash、prefetch planner、I/O aggregation 和离线 readout；不默认 CPU 主算。

判读边界：

- SSD 只做 cold tier，不进入逐 token decode 热路径。
- 没有 expert miss penalty，不声称专家分层收益。
- 没有 H2D/D2H overlap 证据，不声称 KV offload 收益。
- 没有固定输出长度和单变量控制，不做性能 A/B 结论。

## 7. P9：模型推理性能与硬件参数联合分析

目标：把 P0/P3 microbench、P6 baseline、P7 boundary、P8 state tiering trace 合并成下一代硬件优先级。

输出：

- `hardware_sensitivity_report.md`
- `hardware_ask_matrix.yaml`
- `bottleneck_attribution_report.md`

参数扫描：

| 参数 | 扫描范围 | 输出 |
| --- | --- | --- |
| HBM per NPU | 64/84/128/200/400GB | fit boundary、hotset coverage、KV headroom |
| HBM bandwidth | current x0.5/x1/x2 | TPOT sensitivity |
| DRAM bandwidth | current x0.5/x1/x2 | KV/expert warm tier sensitivity |
| NPU-CPU link | current/200/400/800GB/s | restore stall 与 overlap |
| SSD I/O | current/object/direct path | cold tier feasibility |
| CPU cores/vector | 48/96/192 + SVE/SME microbench | metadata、I/O、prefetch、fallback 边界 |
| HMM/PIM | modeled | next-gen benefit and risk |

验收：

- 每个硬件建议必须绑定触发实验、观测瓶颈、受影响指标、建模依据和软件前提。
- 只在 smoke、controlled benchmark、profile readout、request-device aggregate 和 simulator 证据闭合时，输出硬件优先级。
