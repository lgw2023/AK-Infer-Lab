## 7. P8：分层工程原型

### 7.1 总体口径

P8 不是“一步到位写完整框架”，而是四层组合：

1. **运行底座**：vLLM-Ascend 主路；MindIE availability-gated 对照路。
2. **可复用能力**：Prefix Cache、KV Cache CPU Offload、UCM/External KV Cache、EPLB/static expert map。
3. **自研控制面**：统一 `StateObject`、`StateEvent`、`PlacementDecision`，但不持有 tensor payload。
4. **实验与仿真**：KV real path、expert trace/static placement、Expert Tier V0、条件式 warm prefetch、P9 trace bundle。

### 7.2 StateObject 范围

统一对象：

```text
KV block
Prefix block
Expert weight
Weight shard
Session
```

Trace 是事件，不是对象；whole-device HBM、tensor footprint 或 server stats proxy 不能冒充 object bytes。

### 7.4 KV/Prefix 顺序

```text
P6 Prefix baseline
  -> KV Cache CPU Offload
  -> UCM/External KV Cache DRAM-first
  -> SSD/NFS/3FS cold persistence
  -> MindIE comparison when available
```

SSD 不进入逐 token decode 热路径。路径跑通但性能为负仍是有效结果。

### 7.5 MoE 顺序

```text
expert tensor/role bytes + TP4 owner mapping
  -> aggregated expert hotness
  -> EPLB recording/static map
  -> request/session/router trace only when needed
  -> Expert Tier V0 + TP4 capacity simulation
  -> gated payload mover / warm prefetch V1
  -> CPU-first loader + TP4 full-model capacity smoke
```

EPLB 只证明 load balance/placement，不等于 expert offload。没有 expert bytes、load latency、miss penalty 时不声称分层收益。TP8 no-MTP/MTP 的 `38.1255/39.2795 GB per worker` 与 P6.1C `61436–61447 MB` whole-device HBM 只证明直接 TP4 缺乏容量前提，不能直接推出固定 `150–170GB` offload 目标；P8.3/P8.4 必须先按 materialized tensor、TP4 rank ownership 和 runtime reserve 重算。四卡问题最终必须以 `4096+64,c1` path smoke 或可审计负结论关闭，不能只标 `not_started_by_gate`。
