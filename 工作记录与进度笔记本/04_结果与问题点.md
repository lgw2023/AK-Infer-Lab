# 04 结果与问题点

## 已确认结果

### R1 范围确认

本项目当前只关注模型推理本身，不负责真实 coding agent 负载。

实际口径：
- 用人工构造的提示词数据近似 coding-agent 类请求。
- 把工具输出、代码片段、历史对话、repo 摘要等内容作为普通 prompt 输入。
- 观测模型推理阶段的硬件负载，而不是 agent 系统本身。

### R2 第一阶段重点

第一阶段不是做复杂 A+K 改造，而是建立可复现实验底座：
- 容器和 CANN 环境。
- 单卡模型推理。
- profiler、trace、microbench。
- 请求级指标和硬件级指标统一记录。
- 状态对象级 trace、运行时队列/调度 trace、copy 与 compute overlap trace。
- 先能判断瓶颈，再决定做 KV offload、expert hot cache 或 CPU fallback。

### R3 原始材料已归档

原始方案材料已统一放入 `00_原始材料/`，后续新进展写入笔记本，不再把状态散落在长文档里。

### R4 P0.5 可观测能力体检框架已实现

已经把服务器体检设计落到代码框架：
- 字段目录覆盖 server、request runtime、scheduler、operator、NPU/HBM、CPU/DRAM、transfer overlap、SSD I/O、state object、KV/prefix、MoE expert、power/stability、microbench 等 profile。
- 每个字段有统一元信息：类型、单位、采样方式、工具来源、权限需求、join key、time base、验证方式、验收规则、fallback 和 availability。
- run 输出包含 manifest、field availability、probe results、join-key readiness、P0 acceptance fields 和人读摘要。
- 本地框架验证已通过；后续正式服务器证据见 R5。

### R5 服务器体检已进入第五次复测，NPU 与基础系统工具均已跑通

截至 `obs_2026_0705_atlas800t_a2_005`：
- 服务器 conda 环境、`torch_npu 2.9.0.post2`、8 张 NPU 可见性已经验证。
- NPU H2D、D2H、copy overlap、matmul shape、CPU kernel、DRAM smoke 已有真实 artifact。
- `npu-smi 26.0.rc1` 表格输出已解析出 `driver_version=26.0.rc1`、`npu_count=8`、`hbm_per_npu_gb=64.0`。
- `fio`、`numactl`、`perf` 已通过第五次 run 证实可用；第四次 run 中这三项的 `tool_missing` 已是安装前旧状态。

## 当前问题点

### Q1 统一事件模型仍需在服务器体检后继续收敛

体检框架已经先完成“字段能否采集”的盘点，但还没有进入复杂 runtime hook 和统一事件流落地。每个推理 stage 的字段、时间边界、关联资源和可采集证据仍需在服务器真实可测性明确后收敛。

影响：
- 没有 schema 就难以统一 profiler、请求日志和 microbench 输出。

下一步：
- 先跑服务器体检，按真实可测字段决定 P1 最小 schema，不追求一次覆盖所有 runtime。

### Q2 静态提示词 workload 尚未落地

还没有 prompt manifest 和样例提示词文件。

影响：
- 无法稳定复现 coding-agent 风格的长上下文推理压力。

下一步：
- 先做 10-20 条样例，覆盖短、中、长、重复 prefix、多轮恢复。

### Q3 vLLM-Ascend / MindIE 支持矩阵需实测

文档里提到的模型和特性需要以当前环境可跑为准。

影响：
- Qwen3-30B-A3B、量化格式、expert trace hook 是否可用不能只按文档假设。

下一步：
- 小模型 smoke 后再确认 MoE 模型入口。

### Q4 硬件数据采集权限可能不一致

NPU/HBM 数据、CANN timeline、CPU perf、eBPF、SSD fio 对宿主机权限和容器挂载要求不同。

影响：
- 同一套脚本可能在开发机和 Ascend 服务器上表现不同。

下一步：
- 把 env card 和 permission check 作为每次实验的第一步。

### Q5 原计划缺少三类关键观测的验收闭环

现有范围已经覆盖 NPU、HBM、CPU、DRAM、传输、SSD、KV/prefix 和 MoE expert 等硬件大类，但如果只记录大类利用率和平均延迟，仍然无法解释状态对象为什么迁移、调度器为什么等待、搬运是否真的挡住主链路。

影响：
- 无法区分“硬件本身慢”和“状态对象迁移 / 队列调度 / 同步等待导致硬件空转”。
- 无法判断 KV restore、expert prefetch、SSD cold restore、H2D/D2H copy 是被计算掩盖，还是暴露成 TTFT/TPOT 尖峰。

下一步：
- 在 P1/P4 中把状态对象级 trace、运行时队列/调度 trace、copy overlap trace 作为 P0 验收项。
- 每个慢 token 至少要能关联到 `stall_reason`、`policy_decision` 和对应事件证据。

### Q6 Atlas 服务器正式体检旧缺口已被服务器 run 替代

历史上只有本地框架验证；现在已经有来自 Atlas 服务器的正式 `observability_profiles/<run>/`，因此后续不再以“尚未执行正式体检”作为当前问题。

影响：
- 仍不能把本地 dry-run 的字段状态当成 Atlas 可测性结论。
- 需要以最新服务器 run 作为字段可测性判断；过期 run 中的 `tool_missing` 不能覆盖后续安装成功邮件。
- `blocked`、`partial`、`not_applicable` 的判断需要绑定具体 run id 和邮件时间线。

下一步：
- 按当前 `通信模块/docs/developer-to-server.md` 执行第五次复测 `obs_2026_0705_atlas800t_a2_005`。
- 重点审查安装后的 `perf_probe`、`fio_probe`、`numa_probe` 以及 `cpu_perf`、`ssd_fio`、`numa_topology` microbench。
- 再审查 `field_availability.yaml` 与 `join_key_readiness.yaml`，把剩余 blocked reason 汇总成下一轮修复清单。

### Q7 系统 microbench 指标解析已本地修复，待 006 服务器验证

第五次 run 已经固化 `fio`、`numactl`、`perf` 安装后的可用性；旧版 artifact 没有把部分工具输出转成归因可用的结构化指标。本地代码已经完成解析修复，下一步用第六次服务器 run 验证 Atlas 上的真实输出。

已修复：
- `ssd_fio.csv` 成功路径会写出 IOPS、bandwidth 和 latency 指标。
- `cpu_perf.csv` 会写出 `perf_task_clock_ms`，权限允许时还会写出 cycles、instructions 和 IPC；权限不足时降级为 partial 并保留原因。
- `numa_topology.csv` 会写出 node count、CPU list/count、memory/free MB 和 distance matrix。

下一步：
- 下发 `obs_2026_0705_atlas800t_a2_006` 复测，审查三类 CSV 的 status、metric_names 和数值。
- 如果 `cpu_perf` 因 cycles/instructions 权限仍为 partial，只把它记录为 caveat，不现场调整内核权限。
- 本轮不处理 `ascend910b-driver` 半配置问题，避免把 driver 维护混入观测复测。
