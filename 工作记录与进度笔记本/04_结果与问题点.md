# 04 结果与问题点

## 已确认结果

### R71 F1-R7 关闭 pressure 准入死锁，但全局 D2H 不能替代 target lineage

- F1-R7 有界包 transfer 为 `14 files / 24137 bytes`，逐文件 bytes/SHA-256 匹配；3 次请求均 HTTP 200 完整结束，唯一 fixed pressure 实际执行，运维恢复 clean。
- 134 次 runtime coordinator probe 全为 near-miss，logical hit 最大 0，target pool key 最大 0；因此未形成 R7 trigger、abort 或 restore。
- 4,467,519,488 bytes D2H 是 lazy `_prepare_lazy_store_specs` 扫描全局 GPU free queue 的累计量，不证明 target 16K prefix 进入 CPU；`cpu_target_eviction_event_count=0` 也因 observer 不知道 target key 而没有否定力。
- R7 `restore_groups_cpu_complete_count=6` 中五组 required=0，另一组虽 required=1/cpu=1，但 logical hit 仍为 0；零 key group 的 `cpu_complete=true` 是 vacuous 统计，不能作为 eligibility 证据。
- F1-R8 在 target finish 原始调用前捕获实际 group-wrapped keys，按目标 key 归因 lazy store schedule、八 worker completion 后 CPU insert 与 CPU/GPU eviction。full-attention 组必须 128 key 精确，空组为 N/A。
- 物理 full-attention `CPU=128/GPU=0` 与全部 applicable group CPU-only 是 abort 条件；restore 仍要求 abort 后新鲜物理窗口和 logical 16384-token hit。这样即使仍 RED，也能落到具体 mover/coordinator 状态，而非再次得到不可解释的 near-miss。

### R70 F1-R6 是 pre-pressure circular wait，不是 accepted capacity 失败

- F1-R6 有界包为 `13 files / 12315 bytes`，逐文件 bytes/SHA-256 匹配；D2H 8/8、cleanup、八卡健康、端口/vLLM 清理和 0–7 同卡恢复均成立，运维 grade=`operational_recovery_clean`。
- 实验只发送 warmup 与 target prime，`pressure_request_count_executed=0`、abort/idle/restore 均未进入；因此 RED 不能作为 128-block CPU-only 窗口的正反证据。
- target finish 回调发生在异步 D2H 刚提交时；之后控制器只重读静态 trace，却要求 runtime keyspace exact 后才启动下一条 pressure，形成“需要 scheduler activity 刷新、但禁止 scheduler activity 启动”的循环等待。
- F1-R7 pre-pressure 只要求 D2H complete 与 128 个逻辑候选，固定 pressure 必须执行；每个 request-local progress 都重新探测 runtime keyspace，近失、暂时不可观测或 CPU target loss 均不提前放弃。
- runtime lookup 的 `num_uncached_common_prefix_tokens` 可能产生观察副作用，F1-R7 在探测前后原值恢复；raw hash/request/token/content 不出 server-local trace，有界诊断只保留 counts、reason、error type、timestamp 与 group counts。
- exact trigger 后必须使用 abort wall-clock 之后的新鲜 retained-key snapshot 重验；没有新鲜事件时 fail closed，不复用中止前旧快照授权 restore。
- accepted `128 blocks/rank / 430604288 bytes/rank`、fixed 36800、单 lifecycle、零 retry/sweep 保持；不得因本轮仍未触发而轻易放弃 128-block 目标。

### R69 F1-R6 已把逻辑 128-block 覆盖与运行时物理 pool key 分离

- 观察器通过 runtime CPU coordinator 的只读 longest-hit 判定 `16384 tokens = 128 logical hash blocks`，不再假设每个 KV group 都有 128 个物理 key。
- trigger 同时要求 cardinality exact、runtime keyspace matchable、逻辑 16K 完整命中、全部返回 group key CPU resident、GPU key match=0；任一缺失都继续完整 pressure watcher或按精确 RED 终态 fail closed。
- 首次完整窗口的真实 pool keys 会留在进程内，用于后续 CPU eviction 和 post-abort revalidation；有界结果只输出 count，raw hash/block/request/token/content 均不保留。
- HTTP 200 完整响应数与实验 role 合同完成数已分离，pressure 自然完成但无 trigger 不再被误写为 transport 失败。
- keep-alive recovery 以 `#0#..#7#` 的 16 个真实 marker 和八卡覆盖验证；停卡仍是常规操作，所有退出路径恢复同一 0–7。
- 当前只授权一个 accepted-capacity、fixed 36800 lifecycle；仍不得用“本次未观察到”宣称完整逻辑 128-block CPU-only 窗口不可能，也不授权调 capacity/context、retry、sweep、性能、唯一根因、K2 或 P8.3-I1。

### R68 F1-R5 的 128 是 cardinality exact，不是 pool-key exact；accepted capacity 未被否定

- F1-R5 完整有界包为 `13 files / 18231 bytes`，逐文件 bytes/SHA-256 全部匹配；3 次请求均 HTTP 200 完整结束，D2H 8/8、cleanup clean。
- observer 选出的 128 个 request hashes 是未附 KV group ID 的逻辑 hash；CPU/GPU cache map 使用 group-wrapped keys。直接 lookup 得到 41 次 `CPU=0/GPU=0`，属于键空间不对齐，不是 residency 事实。
- 同一 trace 的真实 group-local keys 已显示 group0=`64 CPU/GPU0`、group1=`2 CPU/GPU0`；DeepSeek-V4 runtime 还含 compressed MLA 与 sparse/SWA 语义，不能把“每组物理 key 数”与“16K/128 logical hash blocks”混为一谈。
- 因而 F1-R5 应分类为 `target_key_space_alignment_blocked`；未授权 capacity failure、H2D absence unique cause 或性能结论。
- F1-R6 必须用 runtime coordinator 的只读 longest-hit 语义证明 16384-token logical coverage，再把返回的实际 group keys用于 GPU absence/eviction 跟踪；未达到 128 logical blocks 时继续完整 pressure watcher，不降低 accepted target。
- R5 的 keep-alive `restart_exit=0`、同卡集恢复、8 卡健康已成立，运维 red 是错误匹配 `npu_keep_alive.py` 的探测假阴性；F1-R6 改为核对实际 `#0#..#7#` markers。停 keep-alive 本身仍是常规资源操作。

### R67 F1-R4 不是 128-block 容量失败；F1-R5 闭合有效合同与服务器执行歧义

- F1-R4 有界包完整为 `12 files / 12098 bytes`，实验终态=`pressure_completed_without_trigger`，但 resource recovery 文件缺失；静态结果摘要还错误写成一次 abort，与实际 0 abort 不一致。
- 根因是跨 shell 参数覆盖：F1-R4 wrapper 设置 `P8_2_K1A_H2D_TARGET_BLOCK_COUNT=128` 后，通用 mode 又无条件覆盖为 `64`。所以 F1-R4 的 `CPU=64/GPU=0` 只证明错误有效合同下的窗口，不证明 accepted capacity 下 128-block window 不可形成。
- F1-R4 group0 的 65 个 non-null block 中只有 64 个可哈希，尾部 1 个 unhashable；restore eligibility 应以可查 lookup key 数为 required，同时把 theoretical/selected/non-null/hashable/unhashable 分开报告，不能把尾部结构差异误报为 residency 缺块。
- F1-R5 启动前必须自证 `effective target=128`、`restore=16384`、`block size=128` 与 geometry exact；observer 必须报告 target 捕获来源/数量和逐组有界几何，raw hash、request ID、token ID 与生成内容不得进入有界包。
- 完整 watcher 不因第一次 near miss 或早期 CPU eviction 停止；pressure 生命周期内任一时刻形成完整 `CPU=128/GPU=0` 且全组 eligible 即锁存，随后才允许 abort、idle、post-abort revalidation 和唯一 restore。若始终未形成，输出最接近快照及每组差额，不得扩写为唯一容量不兼容原因。
- 唯一 server driver 接管 audit、停卡、运行、所有退出路径同卡恢复、结构化 recovery 和 finalize；keep-alive 是常规资源操作，重点是任务后恢复 0–7，而不是把停卡本身当异常。

指标口径：
- `TTFT` 是 host client first non-empty text chunk。
- `TPOT` 是 `(response_end - first_token) / (generated_tokens - 1)`。
- `E2EL` 是 client wall time。
- `ITL` 是 host streaming inter-chunk median per request，不是 runtime native decode event。
- common metrics 的 throughput 使用 benchmark window denominator，不等于 per-request `OutputTokenThroughput` 平均值。
- `PrefillTokenThroughput` 是 `input_tokens / TTFT` 的 host-client approximation，不声称 MindIE native `prefill_time`。

server stats 事实：

- 这证明 on/off 开关路径和 server stats 信号有效；但 stats sample count 只有 2/3，不能当作完整资源时间序列。

denominator 事实：
- msprof shape denominator `overall_status=success`。
- `shape_row_count=160`、`op_type_row_count=33`。
- on/off 两侧各 `requests=32`、`shape_rows=80`。
