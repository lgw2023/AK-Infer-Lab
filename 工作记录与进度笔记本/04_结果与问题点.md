## 已确认结果

### R1 范围确认

本项目当前只关注模型推理本身，不负责真实 coding agent 负载。

实际口径：
- 用人工构造的提示词数据近似 coding-agent 类请求。
- 把工具输出、代码片段、历史对话、repo 摘要等内容作为普通 prompt 输入。
- 观测模型推理阶段的硬件负载，而不是 agent 系统本身。

### R2 第一阶段重点

第一阶段不是做复杂 A+K 改造，而是建立可复现实验底座：
- 容器和 CANN 环境。
- 单卡模型推理。
- profiler、trace、microbench。
- 请求级指标和硬件级指标统一记录。
- 状态对象级 trace、运行时队列/调度 trace、copy 与 compute overlap trace。
- 先能判断瓶颈，再决定做 KV offload、expert hot cache 或 CPU fallback。

## 当前问题点

### Q1 统一事件模型已有本地契约，仍需 runtime hook 验证

体检框架已经完成“字段能否采集”的盘点，P1 已把最小事件 schema 和 workload manifest 固化为本地机器可读契约。但还没有进入复杂 runtime hook 和统一事件流真实采集。

影响：
- 现在可以统一 profiler、请求日志和 microbench 输出的字段口径。
- 仍不能声称某个慢 token 已被真实 runtime trace 归因。

下一步：
- 先执行 `runtime_hook_discovery_2026_0705_p1_002`，确认服务器实际 vLLM/vLLM-Ascend hook 候选入口和最小 host marker 可行性。
- P1.3 通过后，再设计 wrapper 日志、profiler marker、state object hook 和 copy stream tagging 的真实服务器采集任务。

### Q2 静态提示词 workload 首批已落地，真实 token 校准待后续 smoke

首批 prompt manifest 和样例提示词文件已经落地。

影响：
- 现在可以稳定引用 P000-P012 作为后续小模型 smoke 和 runtime hook 设计输入。
- 当前 token 数是人工估算，不代表 tokenizer 实测值。

下一步：
- 小模型 smoke 阶段再做真实 tokenizer token 校准，并把校准值回写 manifest。

### Q3 vLLM-Ascend / MindIE 支持矩阵需实测

文档里提到的模型和特性需要以当前环境可跑为准。

影响：
- Qwen3-30B-A3B、量化格式、expert trace hook 是否可用不能只按文档假设。

下一步：
- 小模型 smoke 后再确认 MoE 模型入口。

### Q4 硬件数据采集权限可能不一致

NPU/HBM 数据、CANN timeline、CPU perf、eBPF、SSD fio 对宿主机权限和容器挂载要求不同。

影响：
- 同一套脚本可能在开发机和 Ascend 服务器上表现不同。

下一步：
- 把 env card 和 permission check 作为每次实验的第一步。

### Q5 原计划缺少三类关键观测的验收闭环

现有范围已经覆盖 NPU、HBM、CPU、DRAM、传输、SSD、KV/prefix 和 MoE expert 等硬件大类，但如果只记录大类利用率和平均延迟，仍然无法解释状态对象为什么迁移、调度器为什么等待、搬运是否真的挡住主链路。

影响：
- 无法区分“硬件本身慢”和“状态对象迁移 / 队列调度 / 同步等待导致硬件空转”。
- 无法判断 KV restore、expert prefetch、SSD cold restore、H2D/D2H copy 是被计算掩盖，还是暴露成 TTFT/TPOT 尖峰。

下一步：
- 在 P1/P4 中把状态对象级 trace、运行时队列/调度 trace、copy overlap trace 作为 P0 验收项。
- 每个慢 token 至少要能关联到 `stall_reason`、`policy_decision` 和对应事件证据。

### Q6 Atlas 服务器正式体检旧缺口已被服务器 run 替代

历史上只有本地框架验证；现在已经有来自 Atlas 服务器的正式 `observability_profiles/<run>/`，因此后续不再以“尚未执行正式体检”作为当前问题。

影响：
- 仍不能把本地 dry-run 的字段状态当成 Atlas 可测性结论。
- 需要以最新服务器 run 作为字段可测性判断；过期 run 中的 `tool_missing` 不能覆盖后续安装成功邮件。
- `blocked`、`partial`、`not_applicable` 的判断需要绑定具体 run id 和邮件时间线。

下一步：
- 后续以最新 `obs_2026_0705_atlas800t_a2_006` 作为 P0.5 服务器能力证据。
- 不再重复下发 006 原样复测；如果要新增服务器采集，需先明确新的 P1 schema、runtime hook 或 workload 目标。
- 继续用 run id、邮件时间和 commit 区分历史状态，避免把过期 artifact 当成当前阻塞。

### Q7 系统 microbench 指标解析已经通过 006 服务器验证

第五次 run 已经固化 `fio`、`numactl`、`perf` 安装后的可用性；第六次 run 已经验证本地解析修复在 Atlas 服务器上生效。

- `cpu_perf.csv`: `status=measurable`，包含 `perf_task_clock_ms`、cycles、instructions 和 IPC。
- `numa_topology.csv`: `status=measurable`，98 行指标，包含 8 个 NUMA node、CPU/内存和 distance matrix。

下一步：
- 将 microbench 能力视为 P0.5 已闭环，进入 P1 最小统一事件模型与 workload manifest。
- 仍保留 `bpftrace/eBPF` 为独立可选缺口，不把它作为进入 P1 的阻塞项。
- `ascend910b-driver` 半配置问题仍应另开维护窗口处理，不混入观测链路推进。
