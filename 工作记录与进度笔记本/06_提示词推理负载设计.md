# 06 提示词推理负载设计

## 设计目标

用静态提示词数据近似 coding-agent 风格的模型推理请求，但不运行真实 agent。这样可以把测试范围限定在模型推理侧，避免工具调用、文件系统、调度器和外部服务干扰硬件瓶颈判断。

## 负载类型

| 类型 | 说明 | 主要观测 |
| --- | --- | --- |
| W0 普通短问答 | 512 token 内的普通问题 | 基础 TTFT/TPOT |
| W1 中等代码解释 | 1K-4K token，包含代码片段和问题 | prefill、decode、CPU sampling |
| W2 长系统提示词 | 长 system/developer 指令 + 用户任务 | 长 prefill、HBM、KV 写入 |
| W3 工具输出文本 | 把搜索结果、日志、文件片段作为普通 prompt | 长上下文处理、attention/KV 压力 |
| W4 多轮对话拼接 | 多轮历史消息直接拼接成 prompt | KV/prefix、session 恢复模拟、reuse distance |
| W5 重复 prefix | 多条请求共享相同前缀，不同尾部任务 | prefix 命中收益、shared prefix tokens |
| W6 MoE 路由压力 | 多样化代码/分析/规划 prompt | expert 热度和 reuse distance |
| W7 burst 并发 | 多条请求按短间隔到达 | queue wait、batch 形成、continuous batching |

## 关键原则

1. 提示词是静态文本，不触发真实工具。
2. 工具返回、文件内容、命令输出都只作为文本粘贴进 prompt。
3. 每条 prompt 必须可复现，不依赖当前仓库实时状态。
4. 每条 prompt 都要记录 token 规模和场景标签。
5. prompt 输出长度要受控，避免不同请求之间不可比。
6. 需要显式记录 prefix 复用关系、到达模式和并发分组，否则无法解释队列等待和 KV/prefix 命中。
