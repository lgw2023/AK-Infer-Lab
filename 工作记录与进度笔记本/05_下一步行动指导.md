## 后续设计任务

### Step 1：定义统一事件模型与标准推理链路

输出：
- `unified_event_schema`。
- `inference_stage_schema`。
- `hardware_resource_schema`。
- `state_object_event_schema`。
- `runtime_queue_trace_schema`。
- `copy_overlap_trace_schema`。
- `bottleneck_reason`。

建议字段：
- `timestamp_ns`
- `request_id`
- `session_id`
- `trace_id`
- `phase`
- `layer_id`
- `op_name`
- `kernel_name`
- `device`
- `object_type`
- `object_id`
- `source_tier`
- `target_tier`
- `bytes_read`
- `bytes_write`
- `latency_us`
- `queue_wait_us`
- `hit_or_miss`
- `policy_decision`
- `tokens_in`
- `tokens_out`
- `resource_scope`
- `hardware_counters`
- `overlap_ratio`
- `stall_reason`
- `evidence_source`

完成标准：
- 能覆盖 prefill、decode、attention、KV、MLP/MoE、sampling、copy、CPU 调度、SSD I/O。
- 能单独表达 KV、Prefix、Expert、Weight、Activation、Workspace 的对象生命周期。
- 能记录请求队列、batch 形成、cache manager、prefetch/evict/restore 决策。
- 能区分 copy 与 compute 重叠的时间、暴露在主链路上的时间和同步等待。

### Step 2：建立静态提示词 workload manifest

输出：
- prompt 样例目录。
- manifest 表。

建议字段：
- `prompt_id`
- `scenario`
- `context_len_bucket`
- `estimated_prompt_tokens`
- `expected_output_tokens`
- `max_new_tokens`
- `arrival_pattern`
- `concurrency_group`
- `prefix_reuse_group`
- `shared_prefix_tokens`
- `reuse_distance`
- `turn_count`
- `contains_code`
- `contains_tool_output_text`
- `tool_text_tokens`
- `tool_output_type`
- `workspace_size_bucket`
- `notes`

完成标准：
- 至少 10 条样例。
- 覆盖 512、1K、4K、8K、16K、32K 档位。
- 明确哪些样例用于 repeated prefix，哪些样例用于 multi-turn 恢复。
- 明确哪些样例用于 burst、排队、continuous batching 和 prefix/KV 命中测试。

### Step 3：设计实验卡片

输出：
- 每次实验的 YAML 或 JSON 模板。

必须记录：
- 环境：OS、CANN、driver、torch-npu、vLLM-Ascend/MindIE、NPU 数量、HBM。
- 模型：名称、dtype、quant、context limit、MoE 参数。
- workload：prompt_id、concurrency、output_len。
- 指标：TTFT、TPOT、P95/P99、HBM peak、NPU util、CPU util、DRAM bandwidth、pinned memory、H2D/D2H、copy overlap、SSD I/O。
- trace：request timeline、operator timeline、state object events、runtime queue events、policy decisions。
- 结果：top-3 stall reason。

完成标准：
- 后续每次实验都能按同一个模板记录。
- 每个 top stall reason 都有对应事件或 microbench 证据。
