# 05 下一步行动指导

## 最近一次可直接执行的工作

### Step 1：定义标准推理链路

输出：
- `inference_stage_schema`。
- `hardware_resource_schema`。
- `bottleneck_reason`。

建议字段：
- `request_id`
- `session_id`
- `stage`
- `start_ts`
- `end_ts`
- `duration_ms`
- `tokens_in`
- `tokens_out`
- `resource_scope`
- `hardware_counters`
- `stall_reason`
- `evidence_source`

完成标准：
- 能覆盖 prefill、decode、attention、KV、MLP/MoE、sampling、copy、CPU 调度、SSD I/O。

### Step 2：建立静态提示词 workload manifest

输出：
- prompt 样例目录。
- manifest 表。

建议字段：
- `prompt_id`
- `scenario`
- `context_len_bucket`
- `expected_output_tokens`
- `prefix_reuse_group`
- `turn_count`
- `contains_code`
- `contains_tool_output_text`
- `notes`

完成标准：
- 至少 10 条样例。
- 覆盖 512、1K、4K、8K、16K、32K 档位。
- 明确哪些样例用于 repeated prefix，哪些样例用于 multi-turn 恢复。

### Step 3：设计实验卡片

输出：
- 每次实验的 YAML 或 JSON 模板。

必须记录：
- 环境：OS、CANN、driver、torch-npu、vLLM-Ascend/MindIE、NPU 数量、HBM。
- 模型：名称、dtype、quant、context limit、MoE 参数。
- workload：prompt_id、concurrency、output_len。
- 指标：TTFT、TPOT、P95/P99、HBM peak、NPU util、CPU util、H2D/D2H、SSD I/O。
- 结果：top-3 stall reason。

完成标准：
- 后续每次实验都能按同一个模板记录。

## 本周建议顺序

1. 先完成 schema。
2. 再完成 prompt manifest。
3. 再整理 env card 模板。
4. 然后才开始写或迁移采集脚本。
