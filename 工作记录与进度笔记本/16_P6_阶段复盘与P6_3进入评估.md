## 6. P6.3 计划复审结论

P5→P6→P7→P8→P9 的总路线不变，但原先把 Prefix Cache、MTP、Chunked Prefill、
`max_num_seqs`、`max_model_len` 平铺为五项同级 A/B 不够严谨。建议改为以下顺序。

### P6.3A：matched MTP on/off（第一优先级）

现有 no-MTP control 是修复前 degraded cell，不能与 P6.1 MTP matrix 直接形成严格 A/B。
P6.3A 已在同一 runtime、body、warmup、cell 和非目标开关下只改变 MTP。执行合同扩展为
8 个 cell，覆盖 `4096/65536/131072`、`64/256` output 与 c1/c4/c8；每 mode 每 cell 3 个
measured batch，合计 48 batch/108 request。开发机已接受为 `green_p6_3a_mtp_matched_ab`；
用户性能以 unprofiled run 为准，若需要 operator 归因则另起 profiled 跟进。

### P6.3B：Prefix Cache on/off（第二优先级）

P6.1 的 canonical bodies 有意降低公共前缀，不能拿来验证 Prefix Cache 收益。P6.3B 使用
purpose-built repeated-prefix bodies，固定共享前缀长度/比例、请求顺序、复用距离、输出和并发，
只改变 Prefix Cache 开关。合同覆盖 `4096/32768/65536/131072 × 50%/90%` 八个 group；
每 group 每 mode 1 prime + 3 measured follower。该结果同时是 P8.2 K0 的直接输入，但不复制
Qwen 的 15-cell 结论或把 target ratio 冒充 observed hit ratio。

### P6.3C：Chunked Prefill on/off（条件项）

只在关闭 chunked prefill 不要求同时改变 `max_num_batched_tokens`、graph、context 或其他启动参数时
才构成有效单变量。优先使用 `131072+64+c1`；若 runtime 组合约束迫使第二变量变化，则记录为
unsupported feature combination 并停止，不伪装成 A/B。

### 后移的两项

- `max_num_seqs` 是 scheduler/capacity sweep，不是简单开关 A/B。它应作为可选 P6.3D，只有在明确要回答并发/排队问题时执行，并重新定义 graph/HBM/queue 验收。
- `max_model_len` 更接近容量与 reservation 边界。official 135168 已验证后，不应作为 P6.3 必做性能 A/B；建议移入 P7 capacity boundary，除非后续提出“固定 workload 下仅改变 reservation 上限”的明确问题。

## 9. HBM 采集与干扰设计复审

### 9.1 结论

2026-07-15 对外部技术对话《HBM 采集与干扰设计》完成项目内复审。该对话对后续合同设计有
较高价值，但属于待验证的技术输入，不是本项目服务器事实。它不撤销 P6.1C-R1；已完成的
P6.3A 全程禁用 profiler/HBM sampler，因而没有把该采集变量引入 matched MTP on/off 对照。

P6.1C-R1 的 `green_mtp_official_context_ladder` 保留，但其中“容量”只表示冻结配置下
`131072+64+c1` 请求成功且未见请求、健康、queue、MTP counter 或 cleanup 失败；它不表示
KV pool 剩余 token/block 数、逻辑 KV headroom 或物理 HBM 安全余量已经被完整量化。

### 9.2 本项目已有证据

1. 原 P6.1C 确实因 `npu-smi info -t usages` 缺少 `-i` 而出现 36/36 parse failure，且
   “6 秒窗口内 20 样本 + 5% duty”不可满足；对话指出的合同矛盾与项目历史一致。
2. P6.1C-R1 的单次八卡 `npu-smi info` sweep p50/p95 为
   `1.056859569/1.071280632s`。即使选择 5 秒周期，p95 subprocess wall ratio 仍约为
   `21.4%`；该 wall ratio 不能直接等同于推理干扰，但足以说明它不适合作为短窗口高频主信号。
3. 五档 official ladder 的 per-card peak HBM 仅落在 `61436-61447 MB`；4K 档最紧张卡
   为 `61442 MB`，128K 档为 `61447 MB`，只增加 `5 MB`。这与“物理 KV pool 已在服务启动
   阶段建立、请求长度主要改变逻辑 block 使用”的解释一致，也直接证明当前 HBM trace 对
   context ladder 的信息增益很低；但该结果不能单独证明 allocation/residency/driver-reserve 的
   精确实现语义。
4. 当前干扰门只有一组 control 与 selected-validation：control=`22101.745289 ms`，
   sampled=`19542.7234 ms`，因此记录为 0 增幅。它只证明本次没有观察到回归，不能证明
   sampler 零干扰，也不足以支持统计型 10% 上界。

### 9.3 后续采用规则

1. **P6.3A 不改。** 保持 HBM sampler/profiler 禁用，避免在已冻结的性能 A/B 中引入新变量。
2. **P7 capacity boundary 改用分轴证据。** 启动时保存 KV cache token capacity 与理论最大并发；
   运行时优先检查 `vllm:kv_cache_usage_perc`、`vllm:num_preemptions`、
   `vllm:num_requests_waiting_by_reason{reason="capacity"}`、
   `vllm:request_prefill_kv_computed_tokens`、request success、health、queue 和 cleanup。
   这些指标在当前 vLLM-Ascend 0.22.1rc1 上仍需 server preflight；缺失时标记 logical-capacity
   evidence incomplete，不得伪造或由 HBM 代替。
3. **HBM 降为物理旁证。** 默认只取 server-ready、请求前、TTFT/prefill 附近、请求结束、cleanup
   后 3-5 个 event-aligned bounded snapshots；不再机械要求每个请求 20 个样本。只有新问题明确
   需要捕获短时物理峰值时，才单独做 instrumentation qualification。
4. **状态轴分离。** 后续合同分别记录 functional、logical KV capacity、physical HBM evidence、
   instrumentation health。HBM sampler 失败不应自动把 functional ladder 判红；它只阻断
   physical HBM headroom/均衡/cleanup-leak 相关 green。若 KV 指标也缺失，则 capacity 标记
   incomplete，而不是把请求成功扩写为完整容量 green。
5. **干扰验证独立。** 若未来要把周期 sampler 放进 official run，至少使用多组
   `control_before -> sampled -> control_after`；比较 TTFT/prefill，不用单个 control-sampled 对
   声称零干扰，也不把 qualification 变成正式性能 benchmark。

### 9.4 DCMI 与 Exporter 的采用边界

- Ascend NPU Exporter 官方源码确实暴露 HBM used/total，并按 `updateTime` 更新 cache；若服务器
  已有可用部署，读取 cache 是值得优先探测的低扰动候选，但必须同时记录 producer timestamp、
  freshness、八卡覆盖和实际 update period。当前项目没有证据证明服务器已经部署，不能直接写入
  official run。
- `dcmi_get_device_hbm_info` 是公开 DCMI API 名称，但本轮找到的接口页不是当前
  Atlas 800T A2 + 910B1 + 26.0.rc1 的精确支持矩阵。是否有可链接库、容器/宿主机权限、八卡
  sweep p50/p95/p99、计数器刷新周期及是否影响 prefill，均需单独服务器 preflight 后才能把
  “常驻 DCMI sampler”升为项目事实。
- 单次 `npu-smi info` 继续按 bounded sequential sweep 处理：记录 sweep start/end/span，不宣称
  八卡原子同刻快照，也不并发启动八个 `npu-smi` 进程。

### 9.5 外部一手资料边界

- 华为 Atlas A2 25.2.0 `npu-smi` 命令参考只把 `HBM-Usage(MB)` 定义为片上内存占用，并说明
  无业务时系统也会占用一部分内存；它没有承诺 allocation/residency/driver-reserve 分类、刷新周期
  或八卡原子性：<https://info.support.huawei.com/enterprise/zh/doc/EDOC1100493502/a23468a4>。
- vLLM 0.22.1 production metrics 明确定义上述 KV usage、preemption、capacity waiting 和 prefill
  KV token 指标；是否由当前 Ascend runtime 实际暴露仍以服务器 `/metrics` preflight 为准：
  <https://docs.vllm.ai/en/v0.22.1/usage/metrics/>。
- Ascend NPU Exporter 的 HBM metric 与定时 cache 更新可由官方源码确认，但不证明本服务器已安装：
  <https://gitee.com/ascend/ascend-npu-exporter/blob/f40fabec8c7e79d7cd327695c362d31fee03af45/collector/npu_collector.go>。
