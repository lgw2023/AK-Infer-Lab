## 6. P6.3 计划复审结论

P5→P6→P7→P8→P9 的总路线不变，但原先把 Prefix Cache、MTP、Chunked Prefill、
`max_num_seqs`、`max_model_len` 平铺为五项同级 A/B 不够严谨。建议改为以下顺序。

### P6.3A：matched MTP on/off（第一优先级）

现有 no-MTP control 是修复前 degraded cell，不能与 P6.1 MTP matrix 直接形成严格 A/B。
第一项在同一 runtime、body、warmup、cell 和非目标开关下只改变 MTP。已授权合同扩展为
8 个 cell，覆盖 `4096/65536/131072`、`64/256` output 与 c1/c4/c8；每 mode 每 cell 3 个
measured batch，合计 48 batch/108 request。用户性能以 unprofiled run 为准，profiler 只在差异成立后另起跟进。

### P6.3B：Prefix Cache on/off（第二优先级）

P6.1 的 canonical bodies 有意降低公共前缀，不能拿来验证 Prefix Cache 收益。P6.3B 应使用
purpose-built repeated-prefix bodies，固定共享前缀长度/比例、请求顺序、复用距离、输出和并发，
只改变 Prefix Cache 开关。该结果同时是 P8.2 K0 的直接输入；先做少量 tracer cells，不复制
Qwen 的全 15-cell 矩阵。

### P6.3C：Chunked Prefill on/off（条件项）

只在关闭 chunked prefill 不要求同时改变 `max_num_batched_tokens`、graph、context 或其他启动参数时
才构成有效单变量。优先使用 `131072+64+c1`；若 runtime 组合约束迫使第二变量变化，则记录为
unsupported feature combination 并停止，不伪装成 A/B。

### 后移的两项

- `max_num_seqs` 是 scheduler/capacity sweep，不是简单开关 A/B。它应作为可选 P6.3D，只有在明确要回答并发/排队问题时执行，并重新定义 graph/HBM/queue 验收。
- `max_model_len` 更接近容量与 reservation 边界。official 135168 已验证后，不应作为 P6.3 必做性能 A/B；建议移入 P7 capacity boundary，除非后续提出“固定 workload 下仅改变 reservation 上限”的明确问题。

## 7. 对 P7-P9 的影响

- P7 总目标不变；`max_model_len` boundary 和 `max_num_seqs` capacity/scheduler sweep 可在 P6.3
  核心 A/B 之后并入 P7 校准，而不是阻塞 P6 reference。
- P8.1 observe-only adapter 的早期 no-MTP 前置已被 official MTP、unprofiled 和 profiled green
  超越。进入 P8 前应新建或提升 baseline contract，保留旧 `frozen_degraded` 合同为历史 provenance，
  不直接覆盖。
- P8.2 K0 必须消费 P6.3B 的 purpose-built Prefix Cache A/B；KV offload/UCM 仍在 K0 之后。
- P9 输入和证据等级不变。P6.2 只证明 profiler/readout 可用；没有 P6.3、P7 和 P8 校准时，
  仍不能输出 bottleneck attribution 或 hardware priority。
- MindIE 继续是 availability-gated 条件项，不因 P6 green 自动安装或对照。
