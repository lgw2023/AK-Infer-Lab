# 07 可观测能力体检执行说明

## 目标

运行一次服务器可观测能力体检，生成 `observability_profiles/<run>/`。这不是性能测试，不判断模型性能优劣，也不承诺 KV offload、expert hot cache、SSD cold tier 或 CPU fallback 收益。

## 当前实现状态

体检框架已经实现，代码入口在 `tools/observability_profile/`，测试在 `tests/observability_profile/`。

当前能力：
- 静态字段目录生成。
- run manifest 生成。
- 非破坏性 probe runner。
- field availability 填充。
- join-key readiness 和 time alignment 判断。
- P0 acceptance fields 生成。
- 人读 summary 渲染。
- `collect` CLI 生成固定 run 目录。
- 可选 microbench artifact 生成；默认关闭，必须显式传 `--include-microbench`。

本地个人开发机 dry-run 只能证明框架可运行，不能证明 Atlas 服务器字段可测。服务器能力结论必须来自 Atlas 服务器正式 run。

## 输出

一次体检应生成：

- `manifest.yaml`
- `server_observability_profile.md`
- `field_availability.yaml`
- `join_key_readiness.yaml`
- `p0_acceptance_fields.yaml`
- `probe_results/`
- 可选：`microbench/*.csv`

默认输出位置：

```text
工作记录与进度笔记本/observability_profiles/<run-id>/
```

如果同名 run 目录已经存在，CLI 会拒绝覆盖；请换新的 `--run-id`。

## 只读体检命令

```bash
python -m tools.observability_profile.cli collect \
  --run-id obs_2026_0703_atlas800t_a2_001 \
  --server-id atlas800t-a2-node-001 \
  --operator codex
```

旧的无子命令入口仍保留用于兼容已有命令，但新任务统一使用 `collect`。

## 含 microbench 的服务器命令

```bash
python -m tools.observability_profile.cli collect \
  --run-id obs_2026_0703_atlas800t_a2_002 \
  --server-id atlas800t-a2-node-001 \
  --operator ascend-server \
  --include-microbench \
  --scratch-dir /data/ak-trace/observability_scratch \
  --copy-sizes 4K,16K,64K,1M,16M,256M,1G \
  --fio-qdepth 1,4,16,32 \
  --microbench-duration 10
```

microbench 保护条件：

- 没有 `--include-microbench` 时，不运行 microbench。
- SSD/fio 写入类 benchmark 必须提供 `--scratch-dir`。
- 没有 scratch 目录时，不运行 fio，不写系统盘，只在 `microbench/ssd_fio.csv` 里记录 `scratch_missing`。
- `fio`、`perf`、`numactl`、`torch-npu` 缺失或权限不足时，不中断整次体检，只记录对应 blocked reason。

## 判读顺序

1. 先看 `manifest.yaml`，确认硬件拓扑、软件栈和 probe 脚本版本。
2. 再看 `field_availability.yaml`，确认每个字段是 `measurable`、`partial`、`blocked`、`unknown` 还是 `not_applicable`。
3. 再看 `join_key_readiness.yaml`，确认多源 trace 是否能通过 join key 和 time alignment 做归因。
4. 再看 `p0_acceptance_fields.yaml`，确认 P3/P4/P5 可引用的 P0 硬验收字段。
5. 最后看 `server_observability_profile.md`，只作为人读摘要。

## P0 边界

P0 只打通体检框架、工具权限、字段状态和 artifact 链路。不实现复杂 runtime hook，不做 SSD cold tier 性能优化，不做 MoE expert hot cache，也不做 trace-driven simulator。

## 正式 run 后需要回填

正式服务器体检完成后，至少回填三处：

1. `03_阶段性进展.md`：记录 run id、服务器、生成文件和验证结果。
2. `04_结果与问题点.md`：记录 measurable / partial / blocked / unknown / not_applicable 的总体分布，以及关键 blocked reason。
3. `05_下一步行动指导.md`：把下一步动作从“执行体检”改成具体缺口修复或 P1 runtime hook 任务。
