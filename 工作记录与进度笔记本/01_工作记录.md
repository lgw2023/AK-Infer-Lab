# 01 工作记录

## 总目标

构建一套面向模型推理的硬件负载观测与瓶颈归因工作流：基于标准推理框架和静态提示词 workload，分析模型推理在 NPU、HBM、CPU、DRAM、H2D/D2H、PCIe/UB、SSD、KV/prefix 和 MoE expert 等环节上的压力，判断瓶颈来自哪里。

## 范围边界

### 范围内

- 标准模型推理链路：prefill、decode、attention、MLP/MoE、KV cache、sampling。
- 静态提示词 workload：用长上下文、多轮对话、代码修改请求、工具输出文本等提示词近似 coding-agent 类请求。
- 硬件链路观测：AI Core/AIV、HBM 容量和带宽、CPU 调度和指令、DRAM 带宽、H2D/D2H、PCIe/UB、SSD I/O。
- MoE 行为观测：router top-k、expert 热度、reuse distance、expert tier hit/miss。
- trace-driven simulator：用真实 trace 和 microbench 数据做瓶颈归因与 what-if 分析。

### 范围外

- 真实 coding agent 运行。
- 工具调用系统、终端执行、浏览器自动化、文件读写代理。
- 真实仓库扫描、代码补丁生成质量评估。
- agent 调度器、多 agent 协作、任务规划质量评估。
- 一开始就挑战 GLM5.2 满血低时延、NPU-SSD 直通或 CPU 替代 NPU 主算。

## 当前工作条目

| 编号 | 方向 | 价值 | 首批输出 | 验收口径 |
| --- | --- | --- | --- | --- |
| T1 | 标准推理链路资源账本 | 把推理步骤映射到硬件资源，避免只看 tokens/s | 推理 stage schema、resource schema | 每个 stage 能对应至少一种可采集指标 |
| T2 | 静态提示词 workload | 用可控输入近似 agent 类长上下文请求 | prompt manifest、样例提示词集 | 能覆盖短/中/长上下文、重复 prefix、多轮恢复 |
| T3 | 硬件 microbench | 建立 NPU/CPU/DRAM/SSD/传输基线 | copy、matmul、CPU、DRAM、SSD CSV | 能解释模型 trace 里的基础带宽和延迟上限 |
| T4 | 小模型 trace smoke | 验证采集链路完整 | 小模型请求 trace、profile 样例 | 能同时看到请求级、NPU/HBM、CPU、copy 指标 |
| T5 | MoE expert trace | 判断专家热度和分层价值 | expert routing trace、hotset 统计 | 能计算 top-N expert 常驻命中率 |
| T6 | 瓶颈归因模拟器 | 把观测数据转成决策依据 | bottleneck report、what-if 表 | 能回答瓶颈是 HBM、AI Core、CPU、DRAM、传输还是 SSD |

## 当前优先级

1. 先完成 T1 和 T2，统一“测什么”和“用什么输入测”。
2. 再做 T3 和 T4，验证硬件与推理 trace 可采集。
3. 最后进入 T5 和 T6，用 MoE trace 和模拟器判断后续是否值得做 KV offload、expert hot cache 或 CPU fallback。
