## 1. 任务目的回顾

本阶段目标不是证明模型性能，也不是判断 SSD cold tier、KV offload、MoE expert hot cache 或 CPU fallback 的收益。

真正目标是建立后续 P3/P4/P5 可以引用的可观测底座：

- 明确 Atlas/CANN/NPU/CPU/DRAM/SSD/profiler 等字段在真实服务器上哪些可测。
- 对不可测字段给出结构化 `blocked_reason`，区分缺工具、缺权限、缺代码解析、缺 runtime hook。
- 形成稳定 run 目录，后续 trace、profiler、microbench 都能用同一套字段和 join key 做归因。
- 所有服务器动作必须写入 `通信模块/docs/developer-to-server.md`，服务器只 `git pull`，不改代码、不 push，结果通过邮件发回。

## 4. 当前代码与任务文档的偏差

### 4.1 CLI 偏差

任务文档要求：

```bash
python -m tools.observability_profile.cli collect ...
```

当前代码只有：

```bash
python -m tools.observability_profile.cli --run-id ... --server-id ...
```

影响：

- 服务器无法执行 microbench 参数。
- run 目录实际带日期前缀，和任务文档里的固定目录不一致。

### 4.2 manifest 偏差

当前 `manifest.py` 已运行 `npu-smi info`、`lspci`、`numactl --hardware`、`lsblk -J` 作为 hash 输入，但返回字段仍把 `driver_version`、`npu_count`、`hbm_per_npu_gb` 写死为 `unknown`。

影响：

- `npu-smi` 明明可用，却没有提升 NPU/HBM 字段的可测状态。
- CANN 路径包含 `cann-9.0.0`，但版本没有 fallback 解析。
- `torch_npu` import 超时被当成 version 文本进入 P0 acceptance。

### 4.3 probe 偏差

当前 standard probes 只映射工具可用性字段：

- `npu_smi_available`
- `cann_profiler_available`
- `perf_available`
- `ebpf_available`
- `fio_available`
- `numa_available`
- `container_cgroup_readable`

影响：

- `npu_hbm_profile`、`cpu_dram_profile`、`ssd_io_profile`、`microbench_profile` 全部停留 unknown。
- 无法产生 `microbench/*.csv`。
- 无法验证 scratch-dir 保护策略。

  ```yaml
  bench_name: <name>
  status: measurable|partial|blocked|unknown|not_applicable
  ```

  - 生成 blocked 结果。

  - 不偷偷写系统盘或项目目录。
