## 2026-07-12（P5 no-MTP YELLOW，P8 observe-only adapter gate 已打开）

tokenizer MRO 修正后的八卡任务已全流程通过：固定 vLLM `0.22.1` loader 路径、4096 个合法 token ID、`19487-byte` payload、NPU 0-7 资源门、8 worker 权重加载、`FULL_DECODE_ONLY` server ready 和一个 `4096+64` HTTP 200 请求均有界记录，实际生成 64 tokens。该结果评级为 `yellow_no_mtp_graph_request_success`，只验证 exact no-MTP cell，不验证 MTP、128K 或性能。

## 2026-07-12（原生 tokenizer 内容门通过，转入 MRO 断言修正后的单请求重试）

第一次原生 tokenizer 重试结果为 `red_client_tokenizer_class_assertion`。服务器直接诊断证明 `DeepseekV4Tokenizer.from_pretrained(local_files_only=True)` 返回 `CachedDSV4TokenizersBackend`，其 MRO 含 `DSV4TokenizersBackend`；已生成恰好 4096 个 token ID，范围 `16..90868`，全部小于 tokenizer size `129283`。失败仅因交接脚本断言顶层类名必须 `startswith("DSV4")`，而 vLLM `get_cached_tokenizer()` 会按设计加上 `Cached` 前缀。

本轮没有启动 vLLM server、没有执行形式化八卡资源门，也没有写出 payload 或发送 HTTP 请求，成功请求仍为 0。下一任务 `p5_deepseek_v4_flash_w8a8_8card_no_mtp_tokenizer_mro_retry_v0221rc1_2026_0712` 只把 runtime identity 门改为“MRO 中存在以 `DSV4` 开头的 backend”，其余 loader 来源、4096 长度、token 范围、payload 哈希、no-MTP TP8/EP graph 与单个 `4096+64` 全部保持不变。P8 baseline/adapter 继续关闭。
