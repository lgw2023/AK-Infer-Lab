## 总判断

你这台 **Ascend D910 / 单机八卡**，最适合从 **“A+K 状态分层运行时”** 开始做，而不是一上来就魔改 attention kernel 或做完整 P/D 分离。

当前最现实的路径是：

**NPU 负责热计算主路径；CPU/DRAM/DUMA/SSD 负责状态底座、专家温层、KV/Prefix 温冷层、I/O 聚合、调度与观测。**

本项目材料里的结论也基本一致：短期优先级应是 **P0 可观测 + KV warm tier，P1 Prefix/UCM/Mooncake 与 MoE expert hotness，P2 SSD/CXL/remote cold tier，P3 才考虑 NPU-native direct storage / NPU-SSD 直通**。

## 0. 先确认一个硬边界：你的 “D910” 到底是哪代 Ascend

这一步很关键。公开 vLLM-Ascend FAQ 的旧分支明确说，当时只支持 Atlas A2 / A3 系列，`Ascend 910 / 910 Pro B`未规划支持；但当前主线文档已经在 A2/A3、Ascend950 等方向扩展，并且模型支持矩阵覆盖 Qwen、DeepSeek Distill、Qwen3-based、Qwen2.5 等系列。([vLLM](https://docs.vllm.ai/projects/ascend/en/v0.9.1/faqs.html)) ([vLLM](https://docs.vllm.ai/projects/ascend/en/main/user_guide/support_matrix/supported_models.html))

所以开工前先做硬件/软件 inventory：

| 项目 | 必须确认                                                     |
| ---- | ------------------------------------------------------------ |
| NPU  | 是否是 910、910B、910C、A2、A3、Ascend950 派生；每卡 HBM 容量；8 卡互联拓扑 |
| CPU  | Kunpeng 型号、核数、NUMA 拓扑、SVE/向量能力                  |
| 内存 | DDR 容量、通道数、NUMA 亲和、DUMA 是否可被 runtime 明确控制  |
| 存储 | NVMe 数量、PCIe lane、顺序读写、4K/64K/1M random I/O、文件系统 |
| 软件 | 驱动、固件、CANN、torch-npu、MindIE、vLLM-Ascend 版本是否互相匹配 |
| 互联 | NPU-NPU、CPU-NPU、CPU-SSD、NPU/CPU copy engine 的真实带宽与同步开销 |

没有这张表，不要开始谈“能跑多大模型”。Ascend 生态的版本组合会直接决定你能不能用 vLLM-Ascend、MindIE、UCM、Mooncake、KV CPU Offload、PD 分离和量化权重。

## 1. 模型目标要分三级，不要一上来冲 GLM5.2 / DeepSeek 满血

建议按三档开工。

| 阶段               | 目标模型                                                     | 目的                                         | 成功标准                                                   |
| ------------------ | ------------------------------------------------------------ | -------------------------------------------- | ---------------------------------------------------------- |
| **A 档：基线模型** | Qwen2.5/Qwen3 7B、14B、32B；DeepSeek-R1-Distill-Qwen 7B/14B/32B | 打通服务、性能观测、KV 事件、D2H/H2D trace   | 能稳定服务；拿到 TTFT、TPOT、HBM、DDR、CPU、SSD 数据       |
| **B 档：中大模型** | Qwen 72B、Qwen3-MoE、Qwen3-VL-MoE、Qwen3.5-A 系列            | 验证 TP/EP、MoE 专家热度、KV/Prefix 分层     | 能跑多并发/长上下文；能看到 expert hit/miss 与 KV hit/miss |
| **C 档：极限模型** | DeepSeek-V3/R1/V3.2、GLM5.2 744B、超大 Qwen MoE              | 做容量压榨和状态分层实验，不直接承诺生产性能 | 能低比特加载、少量 token 生成、记录瓶颈和失效边界          |

MindIE 的 DeepSeek-V3.2 指南显示，DeepSeek-V3.2 是 MoE + DSA 稀疏注意力模型，MindIE-LLM 在 NPU 上用 aclgraph 图模式做高性能推理；但它的硬件支持表写的是 **Atlas 800I A2 四机 32 卡**、**Atlas 800I A3 双机 16 卡**，不是单机八卡。文档还显示 DeepSeek-V3.2 支持 W8A8 量化、Data Parallel、Context Parallel、Tensor Parallel、Expert Parallel、Chunked Prefill、Prefix Cache、PD 分离等特性。([GitHub](https://github.com/Ascend/MindIE-LLM/blob/master/docs/zh/user_guide/models/deepseek/deepseek_v3.2.md))

这意味着：**单机八卡可以做 DeepSeek/GLM/Qwen 的分层卸载与极限探索，但不要把 DeepSeek-V3.2/GLM5.2 满血低时延服务作为第一阶段目标。**

## 2. 框架选择：两条主线并行

### 主线一：MindIE 做稳定高性能基线

MindIE 更适合作为 **Ascend 官方高性能基线**。尤其是 DeepSeek-V3.2 这种模型，MindIE 文档已经给出 W8A8、MTP、Prefix Cache、PD 分离、EP/TP/DP 等配置路径。([GitHub](https://github.com/Ascend/MindIE-LLM/blob/master/docs/zh/user_guide/models/deepseek/deepseek_v3.2.md))

你应该用 MindIE 做这些事：

| 用途       | 具体动作                                                     |
| ---------- | ------------------------------------------------------------ |
| 基线性能   | Qwen/DeepSeek 小中模型跑 TTFT、TPOT、吞吐、显存占用          |
| 量化基线   | 先测 W8A8，再看 W4A8/W4A16 是否值得                          |
| 多卡并行   | 测 TP=8、EP、DP、CP 对吞吐和尾延迟的影响                     |
| 大模型参考 | 看 DeepSeek-V3.2 这类官方配置如何处理 EP、PD、MTP、Prefix Cache |
| 对照组     | 后续所有 A+K 改造都要和 MindIE 基线比                        |

### 主线二：vLLM-Ascend 做可改造研究底座

vLLM-Ascend 更适合做 **状态对象、KV/Prefix 分层、外部缓存、实验性调度器**。它已经有 KV Cache CPU Offload：可把 inactive KV blocks 从 NPU memory 异步 offload 到 CPU memory，使用 CPU block pool、D2H/H2D separate NPU streams 和 LRU 管理。([vLLM](https://docs.vllm.ai/projects/ascend/en/main/user_guide/feature_guide/kv_cache_cpu_offload.html))

vLLM-Ascend 的价值在于：

| 能力               | 你能做什么                                                   |
| ------------------ | ------------------------------------------------------------ |
| KV CPU Offload     | 把 NPU HBM 里的 inactive KV 下沉到 CPU DRAM                  |
| UCM / external KV  | 把 Prefix/KV 从引擎内部缓存变成外部状态层                    |
| KV Pool / Mooncake | 探索跨实例、P/D、外部 KV pool                                |
| metrics / hooks    | 插入自己的 KV、expert、copy、stall 事件                      |
| 与 LMCache 接近    | 借鉴 LMCache 的 KV object API、connector、batch movement、pipeline 思路 |

LMCache 论文也强调，它把 vLLM/SGLang 生成的 KV 从 GPU 内存抽取出来，跨引擎、跨查询共享；支持 cache offloading 和 P/D disaggregation，并提供面向 GPU、CPU、storage、network 的一等控制 API。([arXiv](https://arxiv.org/abs/2510.09665))

这一阶段的产物不是论文，而是一套 **硬件画像脚本 + 请求级 trace + 单资源扰动脚本**。

## 4. 第二阶段：做 KV / Prefix / Context 分层

这是最容易落地、也最容易证明收益的方向。

### 4.1 先做 DRAM warm tier，不要先上 SSD

vLLM-Ascend KV Cache CPU Offload 已经支持 inactive KV 从 NPU memory 到 CPU memory 的异步 offload，D2H/H2D 在独立 NPU stream 上执行，CPU 侧用 LRU block pool 管理。([vLLM](https://docs.vllm.ai/projects/ascend/en/main/user_guide/feature_guide/kv_cache_cpu_offload.html))

你先做这个实验矩阵：

| 实验                 | 变量                                    | 指标                               |
| -------------------- | --------------------------------------- | ---------------------------------- |
| KV CPU offload 开/关 | offload size、block size、pinned memory | TTFT、TPOT、P95、H2D stall         |
| Prefix cache 开/关   | prompt 重复率、prefix 长度              | TTFT、prefix hit、HBM 释放量       |
| 长上下文恢复         | 4K/16K/32K/64K/128K                     | restore latency、recompute latency |
| 多轮会话             | session 间隔 1s/10s/60s/10min           | resume latency、KV 保温收益        |

### 4.2 再做 UCM / LMCache 式对象层

不要把 KV 视为“显存块”，要抽象成对象：

```text
KVObject {
  object_id
  session_id
  layer_range
  token_range
  model_version
  dtype
  bytes
  tier: NPU / DRAM / DUMA / SSD
  hotness
  ttl
  last_access_ts
  restore_cost
  recompute_cost
  quality_risk
}
```

Mooncake 的核心启发是：KVCache 可以成为 serving 架构中心，P/D 分离时还可以利用 CPU、DRAM、SSD 资源做 disaggregated KVCache；其实验报告在长上下文场景下有效，模拟场景最高吞吐提升 525%，真实 Kimi 负载多处理 75% 请求。([arXiv](https://arxiv.org/abs/2407.00079))

### 4.3 SSD 只做 cold tier

Tutti 这类研究给出的反面教训很重要：**直接把 KV 放 SSD 不一定赚钱**。普通 SSD-backed KV 容易被 tiny random I/O、CPU control path、DRAM-HBM bounce copy 和同步开销反噬。项目材料也明确建议：小算力一体机优先做 HBM 内 prefix/APC、CPU DRAM warm tier、UCM/Mooncake external KV，最后才上 SSD cold tier。

SSD 方向只建议做这三类：

| SSD 对象        | 合理用途                                   |
| --------------- | ------------------------------------------ |
| cold KV         | 多轮历史会话、长时间不用但可能恢复的上下文 |
| cold expert     | 低频专家、低精专家、检查点                 |
| trace / profile | 请求轨迹、expert 路由、KV 生命周期日志     |

## 5. 第三阶段：做 MoE 专家分层，这是跑大模型的主战场

对 DeepSeek / GLM5.2 / Qwen MoE，真正的容量矛盾往往不是 KV，而是 **MoE expert weights**。

本项目材料把 MoE expert 管理列为核心方向：瓶颈是专家参数大、token 激活稀疏、HBM 与 KV 争空间；代表工作包括 KTransformers、FineMoE、DALI、FluxMoE、MoE-APEX；建模指标包括 expert size、top-k、routing distribution、miss penalty、prefetch accuracy。

建议的三层专家池：

| 层级                | 放什么                                                       | 策略                            |
| ------------------- | ------------------------------------------------------------ | ------------------------------- |
| **NPU HBM**         | shared expert、non-expert 主干、hot routed experts、当前层即将使用的 experts | 常驻 + 短窗口预取               |
| **CPU DRAM / DUMA** | warm routed experts、session 相关专家、下一窗口候选专家      | mmap / pinned / 大块预取        |
| **SSD**             | cold experts、低频 experts、低精度 emergency experts、检查点 | 大块顺序读，不进每 token 热路径 |

先做 **expert hotness trace**，不要直接做复杂调度。每层记录：

```text
layer_id
token_id
topk_expert_ids
expert_frequency
expert_reuse_distance
expert_tier
expert_hit
expert_miss_penalty
prefetch_lead_time
wrong_prefetch_bytes
```

随后实现三个策略版本：

| 版本 | 策略                                             | 目标                                 |
| ---- | ------------------------------------------------ | ------------------------------------ |
| V0   | 静态 hot experts 常驻 HBM                        | 验证容量分层能否跑通                 |
| V1   | 基于历史 top-k 的 sliding-window prefetch        | 降低 expert miss                     |
| V2   | session / prompt / layer-aware expert prediction | 面向 Coding Agent / 长会话压榨命中率 |

注意：KTransformers 不是“把专家权重放 CPU”这么简单。它是 shared/hot experts 在 GPU/NPU，routed experts 在 CPU，并结合 CPU 特化 kernel、NUMA、异步调度和 Expert Deferral；迁移到 Kunpeng 必须重新做 SVE/SME microbenchmark。

------

## 6. 第四阶段：CPU/NPU 协同计算只做“冷、短、稀疏、可重叠”路径

不要把 Kunpeng 写成主算。CPU 最适合五类角色：control plane、DRAM warm tier、可重叠计算、小矩阵/低频 fallback、I/O aggregation。本项目硬件地图也明确指出，CPU 不能混成一个“CPU 加速”，Kunpeng P0 应该先做 metadata manager、KV/index manager、CPU DRAM KV warm tier；可重叠计算和高效 expert compute 要等逐 shape/SVE 实测后再判断。

优先尝试这些 CPU 子任务：

| CPU 子任务                         | 为什么适合 CPU              | 风险                               |
| ---------------------------------- | --------------------------- | ---------------------------------- |
| tokenizer / chat template / parser | 控制流多，CPU 天然适合      | 不是主瓶颈，但影响 TTFT            |
| Prefix lookup / cache metadata     | 指针、哈希、树结构、对象表  | 元数据过碎会拖慢                   |
| KV similarity / hotness 估计       | 可批处理、可异步            | 不能进逐 token 热路径              |
| expert hotness / prefetch planner  | 依赖历史、session、路由轨迹 | 误预取会挤占 HBM/DDR               |
| low-frequency fallback expert      | 只在 miss 或冷专家场景触发  | Kunpeng SVE 性能未证明前不能强承诺 |
| SSD / filesystem I/O aggregation   | CPU 适合做大块 I/O 编排     | 小随机读会反噬                     |
| embedding / rerank / vector search | 可与 LLM 主路径并行         | 需要和 NPU 负载错峰                |

NEO 的启发是：CPU 只承接一部分 decode attention 与对应 KV，NPU/GPU 仍保持主路径；CPU 参与必须受容量、搬运和重叠条件约束。

## 7. 量化路线：先 W8A8，再小心 W4A16/W4A8

对 Ascend 来说，低比特不是“量化了就快”。一篇 Ascend 910 W4A16 矩阵乘论文指出，W4A16 对大模型很关键，但在 Ascend 910 上高效部署受限于 mixed-precision 支持与解耦计算架构；其 W4A16 kernel 通过 vector cores 做 INT4→FP16 dequant、cube cores 做 GEMM，速度提升约 1.01×–1.74×，且瓶颈主要不是 dequant 本身，而是额外权重全局内存传输。([arXiv](https://arxiv.org/abs/2601.16536))

所以建议：

| 阶段 | 量化策略                                                     |
| ---- | ------------------------------------------------------------ |
| P0   | BF16 / FP16 跑通小模型；W8A8 跑 Qwen/DeepSeek 中模型         |
| P1   | W8A8 + KV cache 量化，验证长上下文与并发                     |
| P2   | W4A16/W4A8 只针对 decode GEMM、expert weights、cold experts 做局部实验 |
| P3   | 2bit/极低比特只做研究，不作为工程承诺                        |

DeepSeek-V3.2 MindIE 指南也选择了 W8A8 作为主要量化路径，并给出 QuaRot、Flex Smooth Quant、Attention/MoE 不同量化策略。([GitHub](https://github.com/Ascend/MindIE-LLM/blob/master/docs/zh/user_guide/models/deepseek/deepseek_v3.2.md))

## 10. 不建议你现在主打的三件事

第一，**不要先做完整物理 P/D 分离**。P/D 分离只有在长上下文、高并发、共享 prefix、KV transfer 可重叠、调度合理时才赚钱；单机八卡更适合先做 prefill/decode 逻辑队列、独立 stream、prefix-aware scheduling。项目报告也明确说，P/D 或 E/P/D 拆分不是默认正收益。

第二，**不要先做 NPU-SSD 直通承诺**。SSD-backed KV 的主要风险不是账面带宽，而是 tiny random I/O、CPU control path、DRAM-HBM copy 和同步 stall。没有 NPU-native storage submission / direct path 前，SSD 只做 cold tier。

第三，**不要把 Kunpeng CPU 主算化**。Kunpeng 的第一角色应是状态底座、调度底座、I/O 聚合、KV/expert warm tier、trace/profiler host。只有 microbenchmark 证明某些 expert shape、短矩阵、KV similarity 在 SVE/SME 上收益为正，才进入 CPU compute path。

## 最小可行工程目标

第一版不要追求“跑下 GLM5.2 满血”。第一版目标应该是：

**在 D910 单机八卡上，用 MindIE + vLLM-Ascend 建立 Qwen/DeepSeek/GLM 类模型的可观测推理底座；通过 KV/Prefix DRAM warm tier、MoE expert hotness trace、专家分层预取和离散事件仿真，证明哪些 CPU/DRAM/DUMA/SSD/PCIe 协同机制在你的硬件上真实收益为正。**

这就是站在前人肩膀上的开局方式：先复用 MindIE/vLLM-Ascend/UCM/KV Offload/Mooncake/LMCache 的成熟接口，再把创新集中在 **A+K 统一状态对象层 + MoE 专家分层 + trace-driven 规格反推**。
